# Архитектура

## С высоты птичьего полёта

Этот репозиторий содержит Shubin Framework — набор конвенций для использования markdown-репозитория как общего командного контекста для людей и AI-агентов. Сам репо — это документация: нет компилируемого артефакта, нет рантайма, нет тестов. «Код» — это markdown, «билд» — рендер на GitHub.

Фреймворк нацелен на команды от двух до восьми человек, которые используют AI-агентов вроде Claude Code как полноценных контрибьюторов и нуждаются в одном месте, где живут решения, фичи, знание и внешний research. Он адресует две боли, приходящие парой: AI-агенты стартуют каждую сессию с нуля и забывают, что было решено вчера; а люди через три месяца не помнят, почему API возвращает camelCase. Утверждение фреймворка — что это одна и та же проблема (контекст живёт в головах, а не в читаемой форме) и что markdown-репозиторий, читаемый людьми и AI-агентами одинаково, — самый дешёвый фикс.

Этот документ — для контрибьюторов и AI-агентов, работающих *над* самим фреймворком, а не для команд, его адаптирующих. Он картирует, где что лежит в репозитории, и какие инварианты мы держим. Туториал по использованию фреймворка — `framework.md`; «лифт-питч» — `README.md`; вариант для соло или пары — `lite.md`.

Сквозные принципы — два, и они проходят через каждый выбор и в репо, и во фреймворке, который он документирует. *Append-only для истории, mutable для state*: решения, фидбек, инциденты — только дописываются; roadmap'ы, feature-страницы, статусы ADR — меняются. *Один источник правды на факт*: у каждого факта есть ровно один канонический дом, и layout существует, чтобы эти дома были очевидны. Большинство инвариантов ниже — прямые следствия этих двух принципов; относитесь к ним как к дешёвому способу держать принципы истинными.

Репозиторий находится в середине миграции v1 → v2. Где текущий layout отличается от целевого, этот документ описывает текущее состояние и помечает то, что ожидается. Миграция отслеживается вне этого репо и не применяется к самому репо: для работы над фреймворком нет ADR или feature-артефактов.

## Структура репозитория

```
README.md              англоязычная точка входа и elevator pitch
README.ru.md           русскоязычная точка входа (зеркало README)
ARCHITECTURE.md        этот файл — карта репо для контрибьюторов
AGENTS.md              инструкции для AI-агентов, работающих над репо
CONTRIBUTING.md        как предлагать изменения
CHANGELOG.md           история релизов
LICENSE                MIT

framework.md           полная спецификация для tech lead'ов
lite.md                вариант для соло и команд 1–2 человека
knowledge.md           спецификация модуля LLM-wiki по Карпати
projects.md            спецификация интеграции GitHub Projects v2
non-tech.md            git и репозиторий без терминала
non-engineering.md     минимум по Finance / Legal / Security / Data / Support

conventions/           cross-cutting конвенции для контрибьюторов
  GIT_CONVENTION.md    правила коммитов и веток

templates/             файлы, которые adopting team копирует к себе
  .github/             GitHub-специфичное (Issue Templates)
  knowledge/           AGENTS.md для опционального knowledge-модуля

i18n/                  переводы lower-case spec-документов
  ru/                  русские переводы
```

Lower-case файлы в корне — это сам фреймворк, документы, которые читает adopting team. UPPER_CASE — мета-документы об этом репозитории. `CONTRIBUTING.md`, `CHANGELOG.md` и `AGENTS.md` появились в ходе миграции v2; директория `templates/` приземлилась вместе с `AGENTS.md`. Остальные канонические UPPER_CASE-файлы, ожидаемые в v2 — `STACK.md`, `GLOSSARY.md` — пока отсутствуют и появятся на более позднем этапе; эта секция обновится, когда они появятся.

### `README.md`, `README.ru.md`

Точки входа. `README.md` — каноническая английская версия; `README.ru.md` — русское зеркало. Не-англоязычный посетитель должен находить русский README, не спускаясь в подкаталог; именно поэтому эти два файла нарушают в остальном строгое правило «переводы живут в `i18n/`».

### `ARCHITECTURE.md`

Этот файл. Существует, чтобы у контрибьютора или AI-агента, впервые попавшего в репо, был один короткий документ, объясняющий layout и инварианты. Это единственный мета-документ о layout; если рацио не здесь — оно либо во `framework.md`, либо в commit message.

### `CONTRIBUTING.md`

Точка входа для контрибьютора: как предложить изменение (issue first для всего, кроме опечаток), какие виды контрибуций приветствуются (контрпримеры, реальные кейсы, отчёты о неработающих практиках), что требует обсуждения до PR (новые top-level директории, новые канонические файлы, переименования глав) и редакционный стиль. Прочитайте до нетривиального PR.

### `CHANGELOG.md`

История релизов в формате [Keep a Changelog](https://keepachangelog.com/). Блок `[Unreleased]` накапливает текущую v2-работу; запись v2.0.0 заполняется на релизе. До тех пор авторитетный лог — это per-stage `v2:` коммиты на ветке `v2`.

### `AGENTS.md`

Правила для AI-агентов, работающих *над этим репозиторием*, а не для AI-агентов в репозиториях adopting team. Фреймворк предписывает adopting team положить свой `AGENTS.md` в корень их markdown-репы с правилами, заточенными под *их* продукт. Этот файл — версия для редактирования самого фреймворка: говорит Claude Code, Codex, Cursor и аналогичным инструментам, какие документы читать сначала, какие инварианты соблюдать, какой формат ADR и стиль прозы поддерживать. Если вы AI-агент, читающий этот файл по порядку, — следующий читайте `AGENTS.md`.

### `LICENSE`

MIT. Фреймворк отдаётся даром; ожидается, что adopting team копирует или форкает структуру свободно. Лицензия не менялась с v1 и не ожидает изменений в v2.

### `framework.md`

Полная спецификация фреймворка для tech lead'а, настраивающего репозиторий с нуля. Покрывает два принципа, layout директорий, который создаёт adopting team, жизненный цикл фич и ADR, роль journal'а, scaling-гайдлайн и опциональный модуль Карпати. Около семисот пятидесяти строк в v2; running onboarding-пример вынесен в `guides/feature-cycle-example.md`.

### `lite.md`

Вариант для соло и команд 1–2 человек. Для MVP или pet-проекта, где полный фреймворк был бы overhead, `lite.md` сохраняет те же принципы, но убирает ceremony. Команда из одного человека с `lite.md` должна суметь перейти на `framework.md` позже без переписывания заметок.

### `knowledge.md`

Спецификация модуля LLM-wiki по Карпати. Три слоя — иммутабельные raw-источники, LLM-compiled wiki с cross-references и `AGENTS.md`, говорящий LLM, как поддерживать wiki. Это *опциональный* модуль: команда может адаптировать остальную часть фреймворка без него. Разделение на отдельный документ отражает эту опциональность.

### `projects.md`

Спецификация интеграции с GitHub Projects v2 — когда Project нужен, рекомендуемые custom fields, views, итерации, cross-repo auto-add, hierarchy view для cross-cutting фич, Issue dependencies, roadmap layout, project templates, каденс ритуалов, известные ограничения и AI-агент-операции. Companion-документ к `framework.md`; необязателен, если соло или 1–2 человека.

### `non-tech.md`

Для PM, дизайнеров и CEO, которым нужно читать и редактировать репозиторий без терминала. Покрывает GitHub web UI flow, branch protection и review-этикет для нетехнических участников. Прочитайте, если настраиваете репозиторий с нетехническими контрибьюторами или если вы не-инженер, пытающийся в нём участвовать.

### `non-engineering.md`

Минимум, нужный фаундеру по Finance, Legal, Security, Data и Support. У каждого домена короткая секция с вопросами, которые фаундеру зададут, и где в репо живут ответы. Stack-агностичен и сознательно неглубок — это стартовая точка, не мануал.

### `conventions/`

Cross-cutting конвенции для контрибьюторов этого репо. Сегодня это только `conventions/GIT_CONVENTION.md`, который предписывает формат commit-сообщений и naming веток — включая v2-специфичное правило, что все миграционные коммиты идут на ветке `v2` с префиксом `v2:` и атомарным scope. Когда добавятся другие cross-cutting правила (стиль прозы, review-этикет, релиз-процесс), они тоже сюда — а не как новые top-level файлы; корень зарезервирован для канона и spec'ов.

### `templates/`

Файлы, которые adopting team копирует к себе. Фреймворк — это документация, не CLI: когда он предписывает конкретный файл (GitHub Issue Template, `AGENTS.md` для конкретной поддиректории), каноническая версия живёт здесь, а команда копирует её в нужное место в своём репо. Текущее содержимое: `templates/.github/ISSUE_TEMPLATE/` (девять шаблонов: Feature, Feature-light, Bug, Sub-issue, Cross-cutting, Vault-task, ADR-initiative, Retro-item плюс `config.yml` и copy-guide) и `templates/knowledge/AGENTS.md` (auto-ingest spec для опционального knowledge-модуля). Шаблоны сознательно не auto-installed; см. `templates/README.md`.

### `i18n/`

Переводы lower-case spec-документов. `i18n/ru/` несёт русские переводы и активная директория переводов; редактору, меняющему root-spec, ожидается либо обновить `i18n/ru/` в том же PR, либо открыть follow-up issue, отметив дрейф. Канонический английский текст — всегда root-копия. Нет `i18n/en/` — корень *и есть* английский источник, а раннее зеркало под `i18n/en/` было удалено, как только перестало добавлять что-либо, кроме дрейфа.

## Иерархия документов

Файлы этого репо делятся на четыре слоя, различимых по месту и регистру.

### UPPER_CASE в корне = канон

Документы, имена которых контрибьютор или AI-агент должен выучить первыми: `README.md`, `ARCHITECTURE.md`, `AGENTS.md`, `CONTRIBUTING.md`, `CHANGELOG.md`, `LICENSE`. Будущие канонические файлы, ожидаемые: `STACK.md` и `GLOSSARY.md`. Они меняются редко и описывают, чем этот репозиторий *является*, а не что фреймворк предписывает. Shouty-case намеренный: контрибьютор, сканирующий корень, должен мочь заметить мета-документы, не вчитываясь в имена файлов.

### lower-case в корне = framework specs

Документы, из которых фреймворк состоит, каждый под свою аудиторию: `framework.md`, `lite.md`, `knowledge.md`, `projects.md`, `non-tech.md`, `non-engineering.md`. Это *продукт*; они эволюционируют вместе с фреймворком. Каждый рассчитан на чтение под конкретную задачу, не cover-to-cover. Tech lead тянется за `framework.md`; соло-фаундер за `lite.md`; PM за `non-tech.md`.

### `conventions/` = cross-cutting правила

Конвенции, применяющиеся ко всему репо, а не к конкретному документу. Это мета-правила о *том, как этот репозиторий поддерживается*, не часть user-facing спецификации фреймворка.

### `i18n/<lang>/` = переводы

Зеркала lower-case spec'ов на другом языке. Корневой `README.<lang>.md` — единственный перевод, живущий на верхнем уровне, потому что README — точка входа, и не-англоязычный посетитель должен попасть на него, не спускаясь в подкаталог.

### Что фреймворк сам предписывает (не присутствует здесь)

Фреймворк — описанный во `framework.md` и `lite.md` — предписывает дополнительные директории, которые adopting team создают в *своих* репо: `decisions/` для ADR, `patterns/` для инженерных паттернов, `journal/{feedback,incidents,retros}/` для append-only хроники, `knowledge/{raw,wiki}/` с `knowledge/AGENTS.md` для модуля Карпати, и `guides/` для развёрнутых примеров. Ничего из этого не существует в этом репо, и не будет: согласно policy, фреймворк не применяется к собственной мета-работе. Где фреймворк предписывает copy-target файл (спека `knowledge/AGENTS.md`, GitHub Issue Templates), каноническая версия живёт в `templates/`, а не по предписанному пути.

Заметно отсутствует из этого списка — и отсутствует в репо adopting team тоже — папка `features/`. В v2 фичи живут в GitHub Issues, не в markdown. Issue — это спецификация, tracker и conversation; структура для P0/P1 фичей обеспечивается Issue Template, а не markdown-скелетом.

Причина правила «не применяй к себе» — фреймворк это движущаяся цель в течение v2, и использование движущейся цели для отслеживания собственного движения создаёт круговые ссылки и бесполезный churn. Когда v2.0.0 зарелизится, будущая версия может это пересмотреть — но планка для пересмотра в том, что фреймворк должен быть достаточно стабилен, чтобы dogfood'ить его на себе.

## Инварианты

Многие из них формулируются как *отсутствие* чего-то, а не наличие — наблюдение matklad'а, что инварианты часто лучше выражать негативно. Список ниже — то, что контрибьютор должен держать истинным при редактировании репо.

- Нет директории `vault/`. v2 сознательно сплющена. Ранние черновики оборачивали каждую working directory в `vault/`; миграция v2 поднимает их в корень, потому что GitHub и файловая система уже дают обёртку, а избыточный префикс на каждом пути добавляет шум без информации.
- Нет `COMMON.md`. Инструкции для AI-агентов живут в `AGENTS.md` в корне и — для adopting team, использующих модуль Карпати — в `knowledge/AGENTS.md`. `AGENTS.md` — кросс-инструментальный стандарт, принятый Claude Code, Cowork и другими; мы ему следуем, а не изобретаем своё имя файла. Каноническая `knowledge/AGENTS.md` для копирования adopting team живёт в этом репо в `templates/knowledge/AGENTS.md`.
- Нет вложенных подпапок под `decisions/`. ADR плоские и последовательно нумерованы (`0001-...md`, `0002-...md`, ...). Нумерация не сбрасывается, даже после крупных переписываний фреймворка или репо adopting team. Плоская нумерация — это индекс; категоризация — через теги внутри документа, а не через директории.
- **Нет директории `features/`** — ни в этом репо, ни в репо adopting team. Фичи — это GitHub Issues, не markdown-документы. Markdown feature-page, зеркалящий состояние Issue, разойдётся с ним за дни; страница без state'а задублирует тело Issue. Дизайн v2 закрывает дилемму, удалив markdown-слой целиком. Markdown держит знание (ADR, паттерны, retro, wiki); GitHub держит workflow (фичи, milestones, статус, owner, priority).
- Нет emoji в канонических (UPPER_CASE) документах. Lower-case spec-документы могут содержать emoji только там, где их содержит цитируемый внешний источник.
- Каждый lower-case spec самодостаточен для своей аудитории. Соло-фаундер, читающий `lite.md`, не должен нуждаться в `framework.md` для применения; tech lead, читающий `framework.md`, не должен нуждаться в `intro.md`. Cross-references между spec'ами разрешены, но читатель, не открывший ни одной, должен всё равно получить рабочую картину.
- Английский в корнях; русский только в `i18n/ru/`. Канонический текст — английский; переводы следуют. Изменение в английском root — источник правды; русский перевод может отставать, и это ожидаемо.
- `knowledge/raw/` иммутабельно. Новая версия внешнего источника — это новый dated-файл (например, `paper-2026-04-17.pdf`), никогда не in-place правка предыдущего. Правка in-place молча инвалидирует любую wiki-страницу, цитировавшую раннюю версию.
- `knowledge/wiki/` поддерживается AI-агентом, следующим `knowledge/AGENTS.md`. Routine-апдейты приходят из auto-ingest `knowledge/raw/`, закрытых Issues, смерженных PR, новых ADR и записей `journal/`. Люди редактируют wiki только для исправления ошибок; если ручная правка повторяется, исправляйте `knowledge/AGENTS.md`, а не патчите симптомы.
- Фреймворк не применяется к собственному репозиторию. Нет `decisions/0001-v2-restructure.md`, нет ADR или feature-артефактов для самой v2-работы. Мета-работа по фреймворку трекается в commit-сообщениях и описаниях PR. Это сознательный выбор в течение миграции v2; может быть пересмотрено в будущей версии.
- В течение миграции v2 коммиты не идут напрямую в `main`. Вся работа приземляется на ветке `v2` с префиксом `v2:` и мержится одним куском, когда v2.0.0 готова. Этот инвариант исчезнет, когда v2 зарелизится.
- Ни один перевод не живёт вне `i18n/<lang>/`, кроме language-specific README в корне. Новый язык перевода добавляет директорию `i18n/<lang>/` и root-уровневый `README.<lang>.md`; никогда не добавляет sibling-tree в корне.
- Файлы под `templates/` — это *copy-targets для adopting team*, не активная конфигурация для этого репо. Изменение шаблона не влияет ни на какое поведение в этом репозитории; влияет только на то, что новые adopting team получат при копировании. Сознательно нет скрипта, материализующего шаблоны по предписанным путям.

Сам фреймворк предписывает, куда adopting team кладёт свои артефакты (`decisions/`, `patterns/`, `journal/`, `knowledge/`, `guides/`) и маршрутизирует фичи в GitHub Issues, а не в директорию; см. `framework.md` для этой карты. Ни одна из этих директорий не существует в этом репо, и контрибьютор, работающий *над* фреймворком, не должен их здесь создавать. Где фреймворк предписывает copy-target файл, каноническая версия живёт в `templates/`.

Читатель, новый во фреймворке, должен начать с `README.md`, затем либо `framework.md` (для полной mental model), либо `lite.md` (для меньшей версии). Читатель, новый в *этом репо* — например, AI-агент, помогающий с редактированием, — должен прочитать этот файл плюс `conventions/GIT_CONVENTION.md` до любых коммитов.

Читатель, ищущий *обоснование* layout-выбора, должен искать его здесь под "Инварианты"; если его там нет — обоснование либо в commit-сообщении, либо во `framework.md`. Сознательно нет `RATIONALE.md` или `DESIGN.md` — этот файл единственный мета-документ о layout, и он остаётся достаточно коротким, чтобы прочесть в один присест.

## Non-goals

- **Никакого CLI или тулинга.** Фреймворк — это markdown плюс конвенции плюс GitHub. Любой будущий тулинг — линтинг feature-страниц, валидация ADR-нумерации, scaffolding директорий из шаблона — принадлежит отдельному проекту, не этому репо. Ценность фреймворка — конвенции; тулинг, автоматизирующий их, можно строить сверху, но возможность жить без него должна оставаться.
- **Никакого team-vault hosting.** Этот репо — не SaaS и не template registry. Команды форкают или копируют структуру в свой приватный репозиторий. Намерения добавлять «Use this template» workflow, тянущий из этого репо в рантайме, нет.
- **Stack-агностичен.** Фреймворк не диктует языки, build-системы или test-фреймворки. Ожидает, что каждая adopting team запишет свой стек в собственный `STACK.md`, но не интересуется, что там. Rails-команда и Rust-команда должны мочь использовать один фреймворк без изменений.
- **Не граф знаний и не wiki-движок.** Cross-references — это plain markdown links. Нет frontmatter сверх того, что рендерит GitHub, нет custom build-step, нет plugin model. Предполагаемый зритель — рендеренный markdown GitHub'а плюс редактор контрибьютора по выбору.
- **Не opinionated по агентам сверх `AGENTS.md`.** Фреймворк именует AGENTS.md местом, где живут инструкции для AI-агентов, и в остальном остаётся в стороне. Нет рекомендации, какого агента использовать, какую модель предпочитать или насколько агрессивно делегировать. Модуль Карпати — единственное исключение, и оно opt-in через `knowledge.md`.
