# Shubin Framework: lite version

> **TODO (v2):** этот перевод отстал от английского `lite.md`. В v2 файл `CLAUDE.md` переименован в `AGENTS.md` (cross-tool стандарт), терминология `vault` заменена на «репозиторий». Содержательно остальное прежнее. До пересинхронизации источник истины — корневой `lite.md`.

**Полное название:** Репозиторий для маленьких проектов — лайт-версия
**Часть:** Shubin Framework (starter document)
**Версия:** 2.0 (устаревшая, см. TODO выше)

**Для кого:** 1–2 человека, работающих над MVP или pet-проектом с Claude Code. Скорость важнее процесса. Нанимать пока никого не планируете.

**Что это:** минимальная практика работы с markdown-папкой как внешней памятью для Claude. Без ADR, без ритуалов, без CLAUDE.md в трёх уровнях. Просто чтобы Claude перестал забывать контекст между сессиями. Плюс — опциональный knowledge-модуль (метод Карпати) для ingest'а внешних источников.

**Сколько времени займёт настройка:** 15 минут базового, +10 минут knowledge-модуля (опционально).

**Связанные документы (когда проект вырастет):**
- `framework.md` — полный фреймворк для команды 2–8 человек с ролями, ADR, ритуалами
- `knowledge.md` — полная версия knowledge-модуля (здесь — упрощённая секция)
- `non-tech.md` — гид для нетехнических соавторов

---

## Зачем это вообще нужно

Вы работаете с Claude Code. В понедельник договорились, что используем Supabase, а не Firebase. В пятницу новая сессия — Claude снова предлагает Firebase. Вы объясняете. В следующий вторник — опять. Это раздражает и съедает час в неделю на повторные объяснения.

Vault решает ровно это. Один раз записали — Claude читает каждую сессию.

Других проблем на этой стадии вам решать не нужно. ADR, роли, ритуалы, event taxonomy — это всё про команды, у которых есть координация. У вас координация одна: ваша голова.

**Бонус для pet-проектов и MVP:** опциональный knowledge-модуль (см. ниже, раздел «Когда нужны внешние источники») — автоматическая компиляция интервью / статей / research в связанные страницы. Для соло это экономит часы вручную.

## Минимальная структура

Одна папка рядом с проектом:

```
my-project/
├── .claude/         ← сам проект
├── src/
├── package.json
└── notes/           ← вот это
    ├── decisions.md
    └── todo.md
```

Два файла на старте. Всё. Не четыре, не пять.

### Что в каждом файле

**`decisions.md`** — решения, которые вы приняли, с причиной. Одна строка на решение:

```markdown
# Решения

## 2026-04-10
- Supabase, не Firebase (нужен Postgres + RLS)
- shadcn/ui для компонентов
- Zustand для state (Redux overkill)
- Деплой на Vercel

## 2026-04-15
- Pricing через Stripe Checkout, не свой чекаут
- Email через Resend

## 2026-04-20
- Отказались от real-time через WebSocket → SSE достаточно
- Image uploads на Cloudflare R2, не S3
```

Пишете, когда приняли решение. Не «архитектурный документ», а списком с датой.

**`todo.md`** — что делать дальше. Не менеджер задач, просто список:

```markdown
# TODO

## Сейчас
- [ ] Docs-страница с FAQ
- [ ] Email notifications для комментариев
- [ ] Rate limiting на API

## Потом
- [ ] Admin dashboard
- [ ] Export данных
- [ ] Webhook на новую подписку

## Идеи
- AI-помощник для онбординга
- Публичные профили
```

### Два других файла — по факту, не авансом

Эти файлы добавляются **когда появилось что туда положить**, не с первого дня. Принцип: не создавайте папок и файлов про запас.

**`patterns.md`** — создаёте, когда в третий раз копируете похожий кусок кода. Типовые решения в вашем коде:

```markdown
# Паттерны

## Формы
react-hook-form + zod. Пример — components/SignupForm.tsx

## Error handling на клиенте
Все fetch через lib/api.ts. Ошибки показываем через toast из sonner.

## Loading states
Skeleton-компоненты в components/skeletons/. 
Не используем spinners — дёргается.

## Auth
Supabase Auth. Проверка в middleware.ts, не в каждом route.
```

**`gotchas.md`** — создаёте, когда обожглись на первом неочевидном баге. Пока свежо, чтобы не забыть:

```markdown
# Gotchas

## Next.js App Router + Supabase SSR
Нужен createServerClient, не createClient. 
Иначе cookies не передаются, пользователь "не залогинен" на сервере.

## Tailwind в server components
Dynamic классы через clsx/cn не работают — Tailwind 
их не видит при сборке. Всегда статические строки.

## Resend в dev
Domain должен быть verified даже для test emails на свой адрес.
Иначе 403.
```

К концу первого месяца работы у вас обычно все четыре файла. Но не тратьте 5 минут на создание пустых файлов в день 1 — всё равно заполните в течение недели по мере работы.

## Как Claude Code это использует

Создайте файл `CLAUDE.md` в корне проекта:

```markdown
# Project notes

Перед работой прочти:
- notes/decisions.md — архитектурные решения
- notes/todo.md — приоритеты
- notes/patterns.md (если существует) — типовые паттерны кода
- notes/gotchas.md (если существует) — известные подводные камни

Когда принимаем значимое решение или обнаруживаем gotcha — 
предложи добавить в соответствующий файл. Не создавай файлы молча.
```

Всё. Claude Code при запуске в этой папке автоматически читает `CLAUDE.md`, видит ссылки на notes/ и работает в контексте.

## Рабочий процесс

### Когда писать в notes

**В `decisions.md`** — когда приняли решение, которое будет раздражать объяснять заново через месяц. Обычно это про:
- Выбор библиотеки или сервиса
- Архитектурное направление (auth, deployment, state)
- Отказ от какого-то подхода («пробовали X, не подошло»)

Не пишите про имена переменных и форматирование — это не решения, это стиль.

**В `patterns.md`** — когда поймали себя на том, что делаете что-то похожее в третий раз. Первый раз — просто код. Второй — «это может быть паттерн». Третий — точно паттерн, записываем.

**В `gotchas.md`** — сразу, как разобрались в странном баге. Пока свежо. Через неделю забудете детали.

**В `todo.md`** — свободно. Это ваш черновик приоритетов, не обязательство.

### Как часто обновлять

Нет расписания. Обновляете, когда заметили что стоит записать. Если за неделю не добавили ни строчки — ничего страшного. Если за день добавили пять — тоже нормально.

**Важно:** пишите коротко. Две строки вместо абзаца. Цель — чтобы вы и Claude быстро ориентировались, а не чтобы было «красиво документировано».

### Как удалять устаревшее

Никак. Просто замените новым решением. История не важна на этой стадии.

Если `decisions.md` вырос до 200 строк — это сигнал, что проект уже не маленький, пора смотреть полную версию (`framework.md`).

## Когда нужны внешние источники: knowledge-модуль

Если в вашем проекте накапливается **внешнее знание** — интервью с потенциальными пользователями, статьи про конкурентов, результаты user research, доменная литература — добавляйте knowledge-модуль.

Это основа метода Андрея Карпати «LLM wiki» в упрощённом виде: вы кладёте сырые источники в папку `raw/`, LLM компилирует их в связанные wiki-страницы с cross-references.

### Когда подключать

- У вас есть хотя бы 5 юзер-интервью или discovery-звонков
- Или вы собираете research по индустрии / конкурентам
- Или работаете над продуктом с нетривиальным доменом, где нужно накопить термины и концепты

До этого — не подключайте. Для pet-проекта «список дел для себя» это overkill.

### Когда особенно полезно для соло и MVP

Knowledge-модуль — это **автоматизация, которая реально работает в диалоге**. Вы бросили 3 интервью в `raw/`, сказали Claude «ingest» — получили compiled персону за 5 минут. Для соло, у которого нет команды аналитиков, это экономит часы ручного синтеза.

### Упрощённая структура

```
my-project/
└── notes/
    ├── decisions.md
    ├── todo.md
    ├── patterns.md
    ├── gotchas.md
    └── knowledge/               ← добавляется по факту
        ├── raw/                 ← всё сюда: транскрипты, статьи, заметки
        │   ├── 2026-04-10-user-interview-alice.md
        │   ├── 2026-04-12-competitor-notion-pricing.md
        │   └── 2026-04-15-podcast-on-plg.md
        ├── wiki/                ← Claude компилирует сюда
        │   ├── personas.md
        │   ├── competitors.md
        │   └── concepts.md
        └── CLAUDE.md            ← инструкция Claude как maintainer
```

Без подпапок в raw/ — плоско. Для соло классификация по типу — overkill.

### Минимальный CLAUDE.md для knowledge

Положите в `notes/knowledge/CLAUDE.md`:

```markdown
# Knowledge wiki

## Твоя роль
Ты поддерживаешь wiki/ как maintainer. 
Люди кладут raw-источники в raw/, ты компилируешь в wiki/.

## Принципы
1. raw/ — immutable. Никогда не редактируй.
2. wiki/ — твоя зона. Создавай и обновляй.
3. Каждое утверждение в wiki/ должно ссылаться на 
   raw-источник. Без источника — не пиши.
4. Прямые цитаты — дословно из raw/, не пересказ.
5. Противоречия — выноси в раздел "Contradictions",
   не сглаживай.

## Команды пользователя

### "ingest" или "ingest <файл>"
Прочитай новые raw-файлы. Обнови wiki. Сообщи что сделал.

### "что знаем про <X>"
Ищи в wiki/. Отвечай с цитатами и ссылками. 
Если в wiki/ нет — проверь raw/. Не выдумывай.

## Ограничения
- Не редактируй wiki/ в стилистическую сторону.
  Wiki отражает источники, не твой литературный вкус.
- Не удаляй wiki-страницы. Помечай deprecated во frontmatter.
```

Это минимум. Если захотите более развёрнутый набор команд (проверка противоречий, полные инструкции maintainer'а) — шаблон в `knowledge.md` раздел 4.2. Для solo-режима даже минимум работает.

### Рабочий цикл

1. В течение недели кладёте в `knowledge/raw/` транскрипты, статьи, заметки. По одному файлу с датой в имени.
2. Когда накопилось несколько — запускаете Claude Code в корне проекта:
   ```
   Прочти notes/knowledge/CLAUDE.md. 
   Ingest все новые файлы в raw/. Обнови wiki.
   Сообщи отчёт.
   ```
3. Claude компилирует, вы смотрите результат в `wiki/`.
4. Проверяете 2-3 цитаты против источников (галлюцинации случаются). Если есть ошибки — говорите Claude «вот тут неверно, перепроверь».
5. Дальше — можете задавать вопросы: «что знаем про enterprise-пользователей?», «как конкуренты решают pricing?». Claude идёт в wiki/, возвращает синтез.

**Честное предупреждение:** запускайте ingest регулярно. Если wiki не обновляется, но raw растёт — вы живёте в иллюзии знания. См. подробно в `knowledge.md` раздел 8.

## Чего НЕ делать

### Не создавайте папок про запас

Не нужно `notes/security/`, `notes/testing/`, `notes/deployment/`. Пустые папки — психологический балласт. Появится реальная нужда — добавите.

Это касается и knowledge/ — не создавайте его, пока не накопилось 5+ raw-источников.

### Не пишите ADR

Для команды 1–2 человека ADR — overkill. Одна строка в `decisions.md` с датой делает ту же работу в 10 раз быстрее.

### Не делайте ежедневный standup с собой

Это смешно, но люди пробуют. Если вы один — вы знаете, что делаете. Если вас двое — Slack/Discord работает без ритуала.

### Не автоматизируйте раньше времени

Никаких Cowork-скриптов, генерирующих weekly digests. У вас нет weekly, у вас есть «код и TODO». Автоматизация нужна, когда есть что автоматизировать.

Исключение — knowledge ingest, но он и так в диалоговом режиме, не в фоне.

### Не копируйте структуру больших команд

Увидели в каком-то блогпосте `domain/`, `product/`, `engineering/`, `design/`? Игнорируйте. Это для команд 5+. Для вас это кладбище пустых папок.

## Когда переходить на полную версию

Один из трёх сигналов:

1. **Нанимаете третьего человека.** Устные объяснения перестают работать, нужен контекст, который читается.
2. **Проект вышел в production и у вас есть пользователи.** Появляются incidents, feedback, метрики — это не помещается в четыре файла.
3. **`decisions.md` или `patterns.md` стал длиннее 300 строк.** Пора разбивать по темам.

До этого момента не усложняйте. Лайт-версия окупается с первой недели. Полная — только с третьего месяца при дисциплине. Для маленького проекта выбор очевиден.

При переходе на полную версию: knowledge-модуль, если уже был настроен, переезжает в vault как `knowledge/`. Но не совсем «просто»: в lite структура `raw/` плоская, а в полной версии — с подпапками (`user-interviews/`, `competitors/`, `industry/`, `calls/`). Придётся разложить существующие raw-файлы по подпапкам — обычно 30 минут работы на ~30 файлов. Wiki/ можно оставить как есть — LLM перекомпилирует при следующем ingest.

## Пример первого дня

Вы решили поднять новый проект. Идея — SaaS для трекинга привычек.

**Минута 1–5.** Создали репо, `npm create next-app`, открыли в редакторе.

**Минута 6–10.** Создали папку `notes/` с двумя файлами: `decisions.md`, `todo.md`. В `decisions.md` написали:

```markdown
# Решения

## 2026-04-18
- Next.js 15 App Router (знаю лучше всего)
- Supabase (auth + DB + storage в одном)
- Tailwind + shadcn/ui
- Деплой на Vercel
- TypeScript strict
```

**Минута 11–15.** Создали `CLAUDE.md` в корне со ссылками на notes/. Запустили Claude Code: «Собери основу: auth через Supabase, layout с навбаром, страницы /, /login, /dashboard».

Claude прочитал notes, использовал ваши решения, сгенерировал код. Вы работаете, не объясняя на пятый раз, что используется Supabase.

**Через неделю.** Обожглись на Next.js App Router + Supabase SSR cookies — потратили полчаса на отладку. Решив, создали `notes/gotchas.md` с одной записью. В следующей сессии Claude Code не пойдёт по этому же пути заново.

**Через месяц.** Начали discovery-звонки с потенциальными пользователями. После пятого — создали `notes/knowledge/raw/` и бросили туда все 5 транскриптов. Попросили Claude ingest — получили первую версию персоны и три концепта, на которых пользователи настаивают. Используете это при планировании следующих фич.

## Финальное

Это всё. Два файла на старте, один CLAUDE.md, 15 минут настройки. Дальше — просто дописываете по мере работы. Два других файла появляются по факту (паттерн третий раз / первый gotcha). Knowledge-модуль — когда накопились внешние источники.

Если через месяц вы не написали в notes ничего — скорее всего, у вас очень простой проект, и даже это не нужно. Если через месяц notes/ стала основным справочником, к которому вы возвращаетесь — отлично, это работает.

Усложняйте по факту боли, а не по плану.
