# Shubin Framework: lite version

**Полное название:** Репозиторий для маленьких проектов — лайт-версия
**Часть:** Shubin Framework (стартовый документ)
**Версия:** 3.0

**Для кого:** 1–2 человека, делающие MVP или pet-проект с Claude Code. Скорость важнее процесса. Нанимать пока не планируется.

**Что это:** минимальная практика использования markdown-папки как внешней памяти Claude. Без ADR, без ритуалов, без трёхуровневой иерархии `AGENTS.md`. Ровно столько, чтобы Claude перестал терять контекст между сессиями. Плюс — опциональный knowledge-модуль (метод Карпати) для ingest'а внешних источников.

**Время на настройку:** 15 минут базово, +10 минут на knowledge-модуль (опционально).

**Связанные документы (когда проект подрастёт):**
- `framework.md` — полный фреймворк для команд 2–8 с ролями, ADR, ритуалами
- `knowledge.md` — полная версия knowledge-модуля (упрощённая секция включена сюда)
- `non-tech.md` — гайд для нетехнических участников

---

## Зачем это вообще

Вы работаете с Claude Code. В понедельник договорились использовать Supabase, не Firebase. В пятницу, новая сессия — Claude снова предлагает Firebase. Объясняете. В следующий вторник — снова. Это бесит и съедает час в неделю на повторные объяснения.

Маленькая папка `notes/` решает ровно это. Записали один раз — Claude читает её каждую сессию.

Другие проблемы на этой стадии вам решать не нужно. ADR, роли, ритуалы, event taxonomy — это для команд, которым нужна координация. Ваша координация в одном месте: ваша голова.

**Бонус для pet-проектов и MVP:** опциональный knowledge-модуль (см. ниже, «Когда нужны внешние источники») — автоматическая компиляция интервью / статей / исследований в связные страницы. Для соло-разработчиков это экономит часы ручного синтеза.

## Минимальная структура

Одна папка рядом с проектом:

```
my-project/
├── .claude/         ← сам проект
├── src/
├── package.json
└── notes/           ← вот это
    ├── decisions.md
    └── todo.md
```

Два файла на старте. Всё. Не четыре, не пять.

### Что лежит в каждом файле

**`decisions.md`** — решения, которые приняли, с причинами. По одной строке на решение:

```markdown
# Decisions

## 2026-04-10
- Supabase, not Firebase (need Postgres + RLS)
- shadcn/ui для компонентов
- Zustand для state (Redux — overkill)
- Деплой на Vercel

## 2026-04-15
- Pricing через Stripe Checkout, не custom checkout
- Email через Resend

## 2026-04-20
- Отказ от real-time через WebSocket → SSE достаточно
- Загрузка изображений на Cloudflare R2, не S3
```

Пишете, когда приняли решение. Не «архитектурный документ», просто датированный список.

**`todo.md`** — что делать дальше. Не таск-менеджер, просто список:

```markdown
# TODO

## Now
- [ ] Docs page с FAQ
- [ ] Email notifications для комментариев
- [ ] Rate limiting на API

## Later
- [ ] Admin dashboard
- [ ] Data export
- [ ] Webhook на новую подписку

## Ideas
- AI onboarding helper
- Public profiles
```

### Ещё два файла — по нужде, не заранее

Эти файлы добавляются, **когда есть что в них положить**, а не в первый день. Принцип: не создавайте папки и файлы заранее.

**`patterns.md`** — создаётся, когда вы скопировали похожий кусок кода в третий раз. Типичные паттерны вашей кодовой базы:

```markdown
# Patterns

## Forms
react-hook-form + zod. Пример: components/SignupForm.tsx

## Client-side error handling
Все fetch'и идут через lib/api.ts. Ошибки показываются через toast из sonner.

## Loading states
Skeleton-компоненты в components/skeletons/.
Спиннеры не используем — они дёргаются.

## Auth
Supabase Auth. Проверка в middleware.ts, не в каждом роуте.
```

**`gotchas.md`** — создаётся, когда обожглись на первом неочевидном баге. Пишите по горячим следам, чтобы не забыть:

```markdown
# Gotchas

## Next.js App Router + Supabase SSR
Нужен createServerClient, не createClient.
Иначе куки не передаются, на сервере пользователь «не залогинен».

## Tailwind в server components
Динамические классы через clsx/cn не работают — Tailwind
их не видит на этапе билда. Всегда используйте статические строки.

## Resend в dev
Домен должен быть верифицирован, даже для тестовых писем на свой адрес.
Иначе 403.
```

К концу первого месяца у вас обычно есть все четыре файла. Но не тратьте 5 минут в первый день на создание пустых файлов — заполнятся естественно за первую неделю.

## Как Claude Code это использует

Создайте файл `AGENTS.md` в корне проекта:

```markdown
# Project notes

Перед работой прочти:
- notes/decisions.md — архитектурные решения
- notes/todo.md — приоритеты
- notes/patterns.md (если есть) — типичные паттерны кода
- notes/gotchas.md (если есть) — известные ловушки

Когда мы принимаем значимое решение или находим gotcha —
предложи добавить в соответствующий файл. Не создавай файлы молча.
```

Всё. `AGENTS.md` — кросс-инструментальный стандарт инструкций для AI-агентов; Claude Code читает его (вместе с `CLAUDE.md`) автоматически при запуске в этой папке, подхватывает ссылки на `notes/` и работает в контексте.

## Workflow

### Когда писать в notes

**В `decisions.md`** — когда приняли решение, объяснять которое через месяц вам будет лень. Обычно это:
- Выбор библиотеки или сервиса
- Архитектурное направление (auth, deployment, state)
- Отказ от какого-то подхода («попробовали X, не подошло»)

Не пишите про имена переменных и форматирование — это не решения, это стиль.

**В `patterns.md`** — когда поймали себя на том, что делаете похожее в третий раз. Первый раз — просто код. Второй — «возможно, это паттерн». Третий — точно паттерн, записывайте.

**В `gotchas.md`** — сразу после разбирательства со странным багом. Пока свежо. Через неделю забудете детали.

**В `todo.md`** — свободно. Это ваш черновик приоритетов, не commitment.

### Как часто обновлять

Без графика. Когда заметили что-то достойное записи. Если за неделю не добавили строки — нормально. Если за день добавили пять — тоже.

**Важно:** пишите коротко. Две строки вместо абзаца. Цель — чтобы вы и Claude быстро ориентировались, а не «красиво задокументировано».

### Как удалять устаревшее

Не удаляйте. Просто заменяйте новым решением. История на этой стадии не важна.

Если `decisions.md` перевалил за 200 строк — это сигнал, что проект уже не маленький, пора смотреть на полную версию (`framework.md`).

## Когда нужны внешние источники: knowledge-модуль

Если в проекте начинают накапливаться **внешние знания** — интервью с потенциальными пользователями, статьи о конкурентах, результаты user research, литература по домену — добавьте knowledge-модуль.

Это метод «LLM wiki» Андрея Карпати в упрощённой форме: вы кладёте сырые источники в папку `raw/`, LLM компилирует их в связные wiki-страницы с перекрёстными ссылками.

### Когда подключать

- У вас уже есть минимум 5 интервью с пользователями или discovery-звонков
- Или вы собираете industry / competitor research
- Или работаете над продуктом с нетривиальным доменом, где нужно накапливать термины и концепты

До этого — не подключайте. Для pet-проекта «todo-лист себе» это overkill.

### Особенно полезно для соло и MVP

Knowledge-модуль — это **автоматизация, которая реально работает в диалоге**. Кладёте 3 интервью в `raw/`, говорите Claude «ingest» — получаете компилированную персону за 5 минут. Для соло без аналитика это экономит часы ручного синтеза.

### Упрощённая структура

```
my-project/
└── notes/
    ├── decisions.md
    ├── todo.md
    ├── patterns.md
    ├── gotchas.md
    └── knowledge/               ← добавляется по нужде
        ├── raw/                 ← всё сюда: транскрипты, статьи, заметки
        │   ├── 2026-04-10-user-interview-alice.md
        │   ├── 2026-04-12-competitor-notion-pricing.md
        │   └── 2026-04-15-podcast-on-plg.md
        ├── wiki/                ← Claude компилирует сюда
        │   ├── personas.md
        │   ├── competitors.md
        │   └── concepts.md
        └── AGENTS.md            ← инструкции мейнтейнеру
```

Без подпапок в raw/ — плоско. Для соло классификация по типу — overkill.

### Минимальный AGENTS.md для knowledge

Положите в `notes/knowledge/AGENTS.md`:

```markdown
# Knowledge wiki

## Твоя роль
Поддерживаешь wiki/ как maintainer.
Люди кладут сырые источники в raw/, ты компилируешь в wiki/.

## Принципы
1. raw/ — иммутабельно. Не редактируй.
2. wiki/ — твоя зона. Создавай и обновляй.
3. Каждое утверждение в wiki/ должно ссылаться на raw-источник.
   Нет источника — не пиши.
4. Прямые цитаты — verbatim из raw/, без перефразирования.
5. Противоречия — выноси в секцию "Contradictions",
   не сглаживай.

## Команды пользователя

### "ingest" или "ingest <file>"
Прочитай новые raw-файлы. Обнови wiki. Отчитайся, что сделал.

### "что мы знаем о <X>"
Ищи в wiki/. Отвечай цитатами и ссылками.
Если в wiki/ нет — проверь raw/. Не выдумывай.

## Лимиты
- Не редактируй wiki/ из стилистических соображений.
  Wiki отражает источники, не твой литературный вкус.
- Не удаляй wiki-страницы. Помечай deprecated в frontmatter.
```

Это минимум. Если хотите более развёрнутый набор команд (проверка противоречий, полные maintainer-инструкции) — шаблон в `knowledge.md` секция 4.2. Для соло-режима даже минимум работает.

### Workflow

1. За неделю вы кидаете транскрипты, статьи, заметки в `knowledge/raw/`. Один файл на единицу с датой в имени.
2. Когда накопилось — запустите Claude Code в корне проекта:
   ```
   Прочитай notes/knowledge/AGENTS.md.
   Сделай ingest всех новых файлов в raw/. Обнови wiki.
   Отчитайся.
   ```
3. Claude компилирует; вы смотрите результат в `wiki/`.
4. Проверяете 2–3 цитаты против источников (галлюцинации бывают). Если нашли ошибки — скажите Claude «вот это место неверно, перепроверь».
5. Дальше — задаёте вопросы: «что мы знаем про enterprise-пользователей?», «как конкуренты работают с pricing?». Claude идёт в wiki/, возвращает синтез.

**Честное предупреждение:** запускайте ingest регулярно. Если wiki не обновляется, а raw продолжает расти — вы живёте в иллюзии знания. Подробности в `knowledge.md` секция 8.

## Чего НЕ делать

### Не создавайте папки заранее

Не нужно вам `notes/security/`, `notes/testing/`, `notes/deployment/`. Пустые папки — психологический балласт. Появится реальная нужда — добавите.

То же про knowledge/ — не создавайте, пока нет 5+ raw-источников.

### Не пишите ADR

Для команды 1–2 ADR — overkill. Однострочник в `decisions.md` с датой делает ту же работу в 10 раз быстрее.

### Не делайте daily standup сам с собой

Это смешно, но люди пробуют. Если вы один — вы знаете, что делаете. Если двое — Slack/Discord работает без ритуала.

### Не автоматизируйте слишком рано

Никаких Cowork-скриптов, генерящих недельные digest'ы. У вас нет недели — у вас есть «код и TODO». Автоматизация нужна, когда есть что автоматизировать.

Исключение: knowledge ingest, но он в режиме диалога, не в фоне.

### Не копируйте структуру больших команд

Увидели в каком-то блоге `domain/`, `product/`, `engineering/`, `design/`? Игнорируйте. Это для команд 5+. Для вас — кладбище пустых папок.

## Когда переходить на полную версию

Один из трёх сигналов:

1. **Вы наняли третьего человека.** Устные объяснения перестают работать, нужен читаемый контекст.
2. **Проект в production, есть пользователи.** Появились инциденты, фидбек, метрики — в четыре файла не помещается.
3. **`decisions.md` или `patterns.md` перевалил за 300 строк.** Пора делить по темам.

До этого — не усложняйте. Лайт-версия окупается с первой недели. Полная — только с 3-го месяца при дисциплине. Для маленького проекта выбор очевиден.

При переходе на полную версию: knowledge-модуль, если уже настроен, переезжает в репозиторий как `knowledge/`. Но не «просто переезжает»: в lite структура `raw/` плоская, в полной — с подпапками (`user-interviews/`, `competitors/`, `industry/`, `calls/`). Существующие raw-файлы придётся разложить по подпапкам — обычно 30 минут работы на ~30 файлов. Wiki/ можно оставить как есть — LLM перекомпилирует на следующем ingest'е.

## Пример первого дня

Вы решили начать новый проект. Идея — SaaS для habit tracking.

**Минуты 1–5.** Создали репо, `npm create next-app`, открыли в редакторе.

**Минуты 6–10.** Создали папку `notes/` с двумя файлами: `decisions.md`, `todo.md`. В `decisions.md` написали:

```markdown
# Decisions

## 2026-04-18
- Next.js 15 App Router (знаю лучше всего)
- Supabase (auth + DB + storage в одном)
- Tailwind + shadcn/ui
- Деплой на Vercel
- TypeScript strict
```

**Минуты 11–15.** Создали `AGENTS.md` в корне со ссылками на notes/. Запустили Claude Code: «Сделай каркас: auth через Supabase, layout с навбаром, страницы /, /login, /dashboard».

Claude прочитал notes, использовал ваши решения, сгенерировал код. Вы работаете, не объясняя в пятый раз, что используется Supabase.

**Через неделю.** Обожглись на Next.js App Router + Supabase SSR cookies — потратили полчаса на отладку. После решения создали `notes/gotchas.md` с одной записью. На следующей сессии Claude Code не пойдёт по той же дорожке.

**Через месяц.** Начали discovery-звонки с потенциальными пользователями. После пятого создали `notes/knowledge/raw/` и положили туда все 5 транскриптов. Попросили Claude сделать ingest — получили первую версию персоны и трёх концептов, на которых пользователи настаивали. Используете это, планируя следующие фичи.

## Финал

Это всё. Два файла на старте, один AGENTS.md, 15 минут настройки. Дальше — просто пополняйте по мере дела. Два других файла появятся по нужде (паттерн на третьей копии / первый gotcha). Knowledge-модуль — когда накопятся внешние источники.

Если за месяц вы ничего не написали в notes — наверное, ваш проект очень простой и даже это не нужно. Если за месяц notes/ стал основным справочником, к которому возвращаетесь, — отлично, работает.

Усложняйте по реальной боли, а не по плану.
