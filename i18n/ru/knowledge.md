**Для кого:** любой член команды — разработчик, PM, дизайнер, CEO, доменный эксперт. Не требуется технический опыт. Если вы умеете писать в markdown и нажимать commit в GitHub Desktop — этого достаточно.

**Что это:** модуль фреймворка, реализующий метод Андрея Карпати («LLM wiki») для командного knowledge management. LLM (Claude Code или Cowork) не просто читает вашу базу знаний — она **активно поддерживает её**: компилирует raw-источники в связанные страницы, обновляет cross-references, флагает противоречия.

**Что это НЕ:** не замена ADR, не замена feature-specs, не место для задач. Knowledge module отвечает только за **внешние и доменные знания**: юзер-интервью, конкуренты, персоны, отраслевые тренды, исследования.

**Связанные документы:**
- `framework.md` — полный фреймворк
- `lite.md` — упрощённая версия для соло/MVP
- `non-tech.md` — как работать с git без терминала

## Перед началом: что это и зачем

### Проблема, которую решает модуль

В команде накапливаются **сырые знания** о внешнем мире:
- Транскрипты интервью с пользователями
- Статьи про конкурентов
- Подкасты про индустрию
- Отчёты исследований
- Звонки с клиентами
- Скриншоты чужих продуктов

Это всё лежит в Google Drive, Notion, закладках браузера, чатах Slack. Через три месяца:
- Никто не помнит, в каком интервью пользователь сказал ключевую фразу про pricing
- Конкурент X и конкурент Y сливаются в голове в одного «какого-то конкурента»
- Персона «small business owner» есть у каждого в голове в своей версии
- Одни и те же доменные термины пересказываются заново каждому новому человеку

**Knowledge module это решает.** Вы бросаете raw-материалы в одну папку. LLM читает и компилирует их в структурированные страницы: персоны, концепты, конкуренты, цитаты из интервью. Страницы связаны `[[wikilinks]]`. При добавлении новых источников LLM обновляет существующие страницы и флагает противоречия.

### Откуда идея

В апреле 2026 Андрей Карпати опубликовал паттерн «LLM wiki» — базу знаний, которую LLM поддерживает сама. Три слоя:

1. **raw/** — immutable источники
2. **wiki/** — LLM-generated страницы с cross-references
3. **CLAUDE.md** — инструкция, как LLM поддерживает wiki

Для персонального knowledge management это работает уже у тысяч людей. Для команды подход адаптируется почти 1-в-1, с небольшими оговорками (ниже).

### Чем это отличается от остального фреймворка

В Shubin Framework есть другие слои:
- `active/decisions/` — ADR (наши решения). Пишут люди.
- `active/features/` — feature-specs. Пишут люди.
- `active/patterns/` — паттерны кода. Пишут люди.
- `log/feedback/` — raw-фидбек append-only. Пишут люди.

Это **human-maintained** слои. LLM читает их, но не модифицирует.

Knowledge module — **единственный слой, где LLM является maintainer'ом**. Люди бросают источники, LLM компилирует. Это принципиальное отличие.

```
    │   │   └── 2026-04-15-beta-startup.md

    │   │   └── 2026-04-18-linear-blog-on-ai.md
    │   ├── industry/
    │   │   └── 2026-04-05-gartner-report-summary.md
    │   └── calls/
    │       └── 2026-04-12-client-discovery-X.md
    │
    ├── wiki/                     ← LLM-maintained
    │   ├── personas/
    │   │   ├── small-business-owner.md
    │   │   └── enterprise-buyer.md
    │   ├── concepts/

    │   └── index.md              ← каталог wiki
    │

```

### 1.2. Два правила, которые нельзя нарушать

**Правило 1: raw/ — immutable.**
После того как вы положили файл в `raw/`, вы **не редактируете его содержимое**. Если нашли ошибку — добавьте новый файл с исправлением, старый не трогайте. Это критично: `raw/` — это ваша «фактическая история», и если её редактировать, LLM начнёт компилировать wiki на изменчивом фундаменте.

**Правило 2: wiki/ — maintained by LLM.**
Вы не пишете в `wiki/` руками, кроме редких случаев (исправить явную галлюцинацию). Wiki-страницы создаёт и обновляет LLM по вашим командам. Если вы видите в `wiki/personas/small-business-owner.md` что-то неверное — **исправьте источники в raw/ или попросите LLM обновить**, а не правьте wiki напрямую.

Эти два правила — фундамент. Нарушение первого убивает доверие к системе. Нарушение второго — превращает wiki в обычный документ-черновик, и LLM перестаёт понимать, что она делает.

## 2. Что именно кладёте в raw/

### 2.1. Форматы источников

Всё в markdown. Если источник не в markdown — конвертируйте перед загрузкой:

| Источник | Как конвертировать |
|---|---|
| Транскрипт Zoom / Meet | Скопировать в .md, добавить frontmatter |
| Статья из блога | Сохранить как markdown (browser extensions, MarkDownload) |
| PDF отчёт | Exp: Claude Code может прочитать PDF и создать markdown-резюме. Или используйте `pdftotext` и подправьте |
| Подкаст-транскрипт | С сервиса транскрипции (YouTube Premium, Otter.ai) скопировать |
| Скриншоты конкурентов | Обычно не нужны в raw/ как таковые. Лучше — заметка в markdown с описанием, ссылка на скриншот |
| Email переписка | Скопировать текст, замаскировать PII |
| Заметки с созвона | Пишите сразу в markdown во время созвона |

### 2.2. Frontmatter для raw-файлов

Минимум:

```markdown
---
type: user-interview     # user-interview | competitor | industry | call | other
source: Acme Corp        # компания/источник/название
date: 2026-04-10
author: @alice           # кто загрузил в vault
---
```

Это помогает LLM понять контекст при компиляции. Без frontmatter тоже работает, но хуже.

### 2.3. Что класть, что не класть

**Класть:**
- Всё, что содержит фактическое знание о внешнем мире
- Первичные транскрипты и заметки
- Статьи, тесты, отчёты

**Не класть:**
- Ваши внутренние ADR (они в `active/decisions/`)
- Feature specs (они в `active/features/`)
- Операционные документы команды (статусы, расписания)
- Чужие материалы без чистки PII (имена клиентов, номера телефонов, финансовые данные)
- Материалы под NDA, которые нельзя распространять внутри команды

### 2.4. Принцип «чем больше, тем лучше»

В обычной документации каждый документ — это стоимость поддержки. В knowledge module **каждый raw-источник — это актив**. LLM обрабатывает объём. 50 интервью дают качественно более полезную wiki, чем 5. Чем больше raw, тем точнее персоны и концепты.

Не бойтесь «захламить» raw/. Это не хлам — это данные.

## 3. Что получается в wiki/

### 3.1. Типы wiki-страниц

Основные четыре:

**Personas** (`wiki/personas/`). Компилированные описания пользовательских сегментов на основе интервью и звонков. Что у них болит, как они решают сейчас, что говорят дословно, какой у них workflow.

**Concepts** (`wiki/concepts/`). Ключевые идеи из домена и индустрии. «Что такое product-led growth», «как работает usage-based pricing», «что такое zero-party data». Одна страница = один концепт.

**Competitors** (`wiki/competitors/`). Профили конкурентов. Pricing, фичи, позиционирование, последние изменения, чьи преимущества и слабости. Компилируются из статей, блогов, скриншотов.

**Themes** (`wiki/themes/`). Трендовые темы из индустрии. «AI в productivity-tools», «shift to vertical SaaS», «consolidation в observability». Это синтез нескольких источников.

### 3.2. Формат wiki-страницы

Пример `wiki/personas/small-business-owner.md`:

```markdown
---
type: persona
last_updated: 2026-04-18
sources: 8
---

# Small Business Owner

## Характеристики
- 5-50 сотрудников
- Сам (а) принимает решения по tools
- Бюджет: $50-500/мес на tool
- Технически грамотный базово, но не разработчик

## Pain points
- Фрагментация инструментов: email, CRM, накладные — всё в разных местах
  (упоминается в [[raw/user-interviews/2026-04-10-acme-corp]], [[raw/calls/2026-04-12-client-discovery-X]])
- Нет времени на настройку сложных систем
  (упоминается в [[raw/user-interviews/2026-04-15-beta-startup]])

## Цитаты
> "Я готов платить больше, если это сэкономит мне час в неделю"
> — [[raw/user-interviews/2026-04-10-acme-corp]]

> "Попробовал Notion, HubSpot, Airtable. Всё слишком сложно для нас"
> — [[raw/calls/2026-04-12-client-discovery-X]]

## Related
- [[wiki/concepts/product-led-growth]]
- [[wiki/competitors/notion]]
- [[wiki/themes/tool-consolidation]]

## Contradictions / открытые вопросы
- В [[raw/user-interviews/2026-04-10-acme-corp]] сказано «бюджет до $100».
  В [[raw/calls/2026-04-12-client-discovery-X]] — «готовы платить $500».
  Возможно, два разных под-сегмента. Требует уточнения.
```

### 3.3. Что делает хорошую wiki-страницу

- **Каждое утверждение сослано на raw-источник.** Без этого страница — галлюцинация.
- **Цитаты — дословные из интервью.** Не пересказ, не reformulation.
- **Cross-links через [[wikilinks]]** — пользовательский механизм Obsidian для связей.
- **Раздел «Contradictions» — явный.** Если источники расходятся — это отмечено, не сглажено.
- **Обновляется при добавлении новых источников**, не переписывается с нуля.

## 4. CLAUDE.md модуля: инструкция maintainer

### 4.1. Зачем отдельный CLAUDE.md

В корне vault у вас уже есть `COMMON.md`. Но в `knowledge/` нужен **отдельный** `CLAUDE.md`, потому что поведение LLM здесь другое: не «читать при работе над кодом», а «активно поддерживать wiki».

Этот файл живёт в `knowledge/CLAUDE.md` и читается Claude Code / Cowork при работе с этой папкой.

### 4.2. Пример рабочего CLAUDE.md

```markdown
# Knowledge module: maintainer instructions

## Твоя роль

Ты поддерживаешь wiki/ как LLM-maintainer по методу Карпати.
Люди кладут raw-источники в raw/, ты компилируешь их в wiki/.

## Принципы

1. raw/ — immutable. Никогда не редактируй файлы в raw/.
2. wiki/ — твоя зона. Создавай и обновляй страницы там.
3. Каждое утверждение в wiki/ должно ссылаться на raw-источник
   через [[wiki-link]]. Без источника — не пиши.
4. Прямые цитаты — дословно из raw/. Никогда не переформулируй.
5. Противоречия — выноси в явный раздел "Contradictions",
   не сглаживай.

## Команды пользователя и как реагировать

### "ingest <файл> в knowledge"
1. Прочитай файл в raw/.
2. Определи, какие wiki-страницы уже существуют и связаны с темой.
3. Обнови существующие страницы новыми фактами и цитатами.
4. Если нужны новые страницы (новая персона, новый концепт, 
   новый конкурент) — создай их с полным набором секций.
5. Обнови index.md.
6. Сообщи пользователю: какие страницы затронуты, какие новые,
   какие противоречия выявлены.

### "что мы знаем про <X>"
1. Начни с поиска wiki-страниц по теме.
2. Если страница есть — ответь в её терминах с цитатами и ссылками.
3. Если страницы нет, но в raw/ есть релевантное —
   предложи создать wiki-страницу.
4. Никогда не выдумывай факты, которых нет в raw/ или wiki/.

### "обнови <страницу>"
1. Прочитай все raw-источники, на которые она ссылается.
2. Прочитай все новые raw/ файлы с даты last_updated.
3. Обнови секции, сохрани структуру.
4. Обнови last_updated и sources count.

### "проверь wiki на противоречия"
1. Пройди по всем wiki-страницам.
2. Найди утверждения, которые противоречат друг другу через
   разные raw-источники.
3. Отметь в разделе "Contradictions" на соответствующих страницах.
4. Верни пользователю короткий отчёт.

## Ограничения

- Не создавай wiki-страницы молча. Сообщай, что создал.
- Не удаляй wiki-страницы. Если страница устарела —
  помести "deprecated" в frontmatter и сообщи.
- Если raw-источники противоречат друг другу — не выбирай
  "правильный". Фиксируй оба и флагай.
- Если пользователь просит "переписать" или "улучшить стиль"
  wiki-страницы — откажись. Wiki отражает источники, 
  а не твой литературный вкус.
```

### 4.3. Что здесь тонкое

Инструкции написаны явно «ты», с конкретными командами и триггерами. Это не общий style-guide. Claude Code следует таким явным инструкциям лучше, чем абстрактным принципам.

Критически важно: **запрет выдумывать факты и обязанность ссылаться на источник**. Без этого wiki за неделю превратится в hallucinated-болото, которому никто не доверяет.

## 5. Рабочий процесс

### 5.1. Еженедельный цикл

**Понедельник (любой):** положили новые raw-источники за неделю. Интервью, звонки, статьи, отчёты.

**Ingest-сессия (20-40 минут, 1-2 раза в неделю):**
1. Откройте Claude Code в папке vault (или Cowork, если предпочитаете).
2. Команда: «Зайди в knowledge/CLAUDE.md, прочти инструкции. Потом ingest все новые файлы в raw/ за последнюю неделю».
3. Claude читает каждый новый raw-файл, обновляет wiki.
4. После каждого файла — commit в git («ingest: <файл>»). Так вы видите историю компиляции.
5. После всей сессии — короткий отчёт от Claude: что обновил, что создал, какие противоречия нашёл.

**Раз в месяц:** команда «проверь wiki на противоречия». Это выявляет накопленные нестыковки между источниками.

### 5.2. Adhoc-запросы

В течение недели, когда кому-то нужно знание:

- «Что мы знаем про пользователей из e-commerce?» → Claude идёт в wiki/personas/, возвращает синтез.
- «Как конкурент X подаёт pricing?» → Claude идёт в wiki/competitors/, возвращает.
- «Кто из пользователей упоминал интеграции со Slack?» → Claude грепает по raw/ и wiki/, возвращает цитаты со ссылками.

Это главная ценность модуля: команда получает доступ к общему знанию в ответ на естественно-языковые вопросы.

### 5.3. Кто в команде какие source кладёт

| Роль | Что обычно загружает |
|---|---|
| PM / Product | Интервью с пользователями, звонки discovery |
| Sales / Success | Звонки клиентов, их возражения и wishlists |
| CEO / Founder | Разговоры с инвесторами, интервью индустрии |
| Дизайнер | Usability-тесты, отраслевые тренды UX |
| Разработчик | Технические статьи, релизы конкурентов |
| Маркетолог | Отраслевые отчёты, позиционирование конкурентов |

Каждый — owner своей категории raw/. Все вместе — reader wiki/.

### 5.4. Когда не использовать Knowledge module

- **У вас в vault живут только технические решения и ADR.** Тогда модуль не нужен, он про внешний мир, а не внутренние решения.
- **У вас меньше 10 raw-источников за первые 2 месяца.** Модуль окупается на объёме. Настраивать его ради трёх интервью — overkill.
- **Команда не готова раз в неделю проводить ingest-сессию.** Без этой дисциплины wiki будет ложно-устаревшей, и хуже, чем её отсутствие.

## 6. Настройка: пошагово для всех ролей

### 6.1. Технический человек (разработчик, tech lead): 20 минут

1. В vault-репо: создать папку `knowledge/` с подпапками `raw/` и `wiki/`.
2. Создать `knowledge/CLAUDE.md` со ссылкой на шаблон из раздела 4.2.
3. Создать `knowledge/raw/user-interviews/`, `raw/competitors/`, `raw/industry/`, `raw/calls/` — по реальным нуждам команды.
4. Создать `knowledge/wiki/index.md` с пустым списком — LLM заполнит.
5. Commit + push.
6. Объяснить остальной команде, куда класть raw (см. 6.2).

### 6.2. Нетехнический человек (PM, дизайнер, CEO): 15 минут

Этот раздел предполагает, что у вашей команды уже настроен Claude Code (у разработчика на машине) и/или Cowork (desktop-приложение для работы с vault). Если это не так — сначала попросите разработчика настроить. Для нетех-контрибьютинга в raw/ достаточно одного из двух; ingest-сессию обычно запускает разработчик через Claude Code, но вы можете запрашивать wiki через Cowork независимо.

Если вы ещё не настроили git — сначала `non-tech.md`, разделы 2-3.

Дальше:
1. В вашем клоне vault откройте `knowledge/raw/<подпапка>/`.
2. Создайте новый файл: `2026-04-18-<что это>.md`.
3. Добавьте frontmatter (шаблон в разделе 2.2).
4. Вставьте содержимое (транскрипт, текст статьи, заметки).
5. Commit в GitHub Desktop с сообщением `raw: <что это>`.
6. Push.

Через какое-то время (в следующую ingest-сессию) Claude сам обновит wiki/. Вы можете посмотреть результаты в `knowledge/wiki/`.

Если нужна срочная справка «что мы знаем про X» — попросите разработчика запустить Claude Code с запросом. Или подключите Cowork и попросите его сами.

### 6.3. Запустить ingest впервые

Когда в raw/ накопилось 5-10 файлов:

1. Откройте Claude Code в корне vault (разработчик).
2. Промпт:
   ```
   Прочти knowledge/CLAUDE.md. Потом выполни ingest всех 
   файлов в knowledge/raw/. Начни с user-interviews/.
   После каждого обновления wiki — сделай commit в git 
   с коротким описанием.
   Если найдёшь противоречия между источниками — 
   отметь в соответствующих wiki-страницах.
   Сообщи мне план перед началом и отчёт после.
   ```
3. Claude предложит план. Посмотрите, что вас устраивает. Скажите «go».
4. Через 5-15 минут получите отчёт: «создал X страниц, обновил Y, нашёл Z противоречий».
5. Просмотрите wiki/. Если что-то явно галлюцинировано — скажите Claude «такое-то утверждение неверно, вот правильный источник».

## 7. Lite-версия модуля (для соло/MVP)

Полный модуль для команды 1-2 человек — overkill. Упрощённая версия:

```
notes/
├── decisions.md
├── patterns.md
├── gotchas.md
├── todo.md
└── knowledge/                ← добавляется, когда появилось >5 raw
    ├── raw/
    │   └── *.md              ← плоско, без подпапок
    └── wiki/
        └── *.md              ← плоско
```

**Отличия от командной версии:**
- Нет подпапок в raw/ — всё плоско. Для соло классификация по типу (interview/article/call) — overkill.
- wiki/ тоже плоская — 5-10 страниц, не иерархия.
- CLAUDE.md для knowledge/ делается копией шаблона из 4.2, но без команд вроде "проверь на противоречия" — для 5-10 страниц это избыточно.
- Ingest — раз в 2-3 недели, не еженедельно.

**Когда добавлять модуль к lite:**
- Начали делать discovery-интервью или user research
- Накопилось >5 raw-источников, которые хочется «свести вместе»
- Работаете над продуктом с нетривиальным доменом, и нужна база терминов/концептов

До этого — не добавляйте. Четыре базовых файла lite-версии достаточны.

**Почему это важно для lite.** Для pet-проекта или MVP ingest-цикл через Claude **полностью автоматизирован**. Вы бросили 3 интервью в `raw/`, сказали «ingest» — получили compile'д персону за 5 минут. Вам не нужно садиться и писать summary руками. Для соло это экономит часы.

## 8. Честные предупреждения

### 8.1. LLM галлюцинирует

Даже с инструкцией «ссылайся на источник» Claude иногда:
- Приписывает цитату не тому интервью
- Делает обобщение, которого нет в источниках
- Создаёт связь между источниками, которая поверхностна

**Противоядие:** правило **каждую wiki-страницу читает человек после первого создания**. Дальше — при значительных обновлениях. Проверяете 2-3 цитаты против источников. Если есть галлюцинации — говорите Claude «вот тут неверно, перепроверь».

### 8.2. Wiki устаревает молча

Источники добавляются, но ingest не запускается → wiki становится stale. Признак: `last_updated` во frontmatter старше 2 недель при активной работе команды.

**Противоядие:** ingest-сессия — это ритуал, не «сделаем когда будет время». Раз в неделю, 30 минут, независимо от того, сколько новых источников. Если ничего нет — Claude вернёт «нет новых источников», это нормально.

### 8.3. Соблазн редактировать wiki руками

Сидит человек, видит в `wiki/personas/small-business-owner.md` неправильную фразу. Думает: «сейчас поправлю». Через месяц половина wiki — ручные правки, LLM-compilation перестала совпадать с реальностью.

**Противоядие:** если что-то в wiki неверно — **найдите raw-источник**, который это породил, исправьте его (или добавьте correction-файл в raw/), запустите ingest. Или явно попросите Claude «на странице X утверждение Y неверно, перепроверь и обнови». Никогда не редактируйте wiki напрямую.

Исключение: если Claude hallucinated что-то *отсутствующее в источниках* — удалите это руками и зафиксируйте в commit message. Это отладка, не контент-правка.

### 8.4. Это не RAG и не search

Ряд ожиданий, которые модуль НЕ выполняет:
- Не отвечает на «найди все упоминания X в raw/» за миллисекунды (для этого — просто grep)
- Не генерирует SQL-like запросы к источникам
- Не заменяет semantic search по большому корпусу документов

Knowledge module — это **compilation, не retrieval**. Ценность не в быстром поиске, а в том, что knowledge синтезирован в читаемые страницы с источниками.

### 8.5. Нет автоматического re-compile

Когда вы добавляете raw-файл, wiki **не обновляется автоматически**. Вы должны явно запустить ingest. Можно настроить GitHub Action, который на каждый push запускает ingest, — но это уже automation поверх, не часть базового модуля. Не обещайте команде «просто положи файл и wiki обновится сама», если не настроили automation.

## 9. Интеграция с остальным фреймворком

### 9.1. Связь knowledge → ADR

В ADR можно ссылаться на wiki-страницы: «исходя из потребностей [[wiki/personas/small-business-owner]], мы выбираем X». Это честное обоснование, с прослеживаемостью до реальных интервью.

В шаблон ADR можно добавить опциональную секцию **Research context**:
```markdown
## Research context
- Персона: [[wiki/personas/small-business-owner]]
- Конкурент-бенчмарк: [[wiki/competitors/notion]]
```

### 9.2. Связь knowledge → feature

В feature-page секция Why может ссылаться на wiki-персоны вместо расплывчатых описаний целевой аудитории:

```markdown
## Why
Фича закрывает pain point из [[wiki/personas/small-business-owner]]:
«фрагментация инструментов». 
Success criteria валидированы на интервью 
[[knowledge/raw/user-interviews/2026-04-10-acme-corp]].
```

### 9.3. Связь knowledge → feedback loop

Когда в `log/feedback/` появляются новые пользовательские сигналы (тикеты, отзывы, usage-наблюдения), их можно периодически promote'ить в `knowledge/raw/` если они несут доменное знание, а не конкретный баг-репорт.

Различие:
- `log/feedback/` — операционный поток (тикет, отзыв, наблюдение)
- `knowledge/raw/` — материал для компиляции в знание о мире

Не всё из feedback попадает в knowledge. Решает человек (обычно PM).

## 10. Финальное

Knowledge module — это **единственный слой Shubin Framework, где LLM не просто читает, а поддерживает**. Это ближе к оригинальной идее Карпати, и именно здесь AI-агент даёт эффект, который сложно получить вручную: **compilation at scale**.

Если у вас команда, которая делает много discovery-работы (интервью, research, competitive intel) — модуль окупается на 10-20 raw-источниках. Если нет — не включайте. Пустой или полу-пустой knowledge-слой хуже отсутствующего: он создаёт впечатление знания, которого нет.

**Дисциплина, которая реально работает:**
1. Кладите raw честно — сырой контент, без фильтрации «красиво/некрасиво».
2. Запускайте ingest по расписанию, не по настроению.
3. Читайте wiki глазами хотя бы раз в месяц, проверяйте галлюцинации.
4. Никогда не правьте wiki руками (за редкими исключениями debug).
