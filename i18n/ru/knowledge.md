# Shubin Framework: Knowledge Module

**Полное название:** LLM-wiki для команд, адаптация метода Карпати
**Часть:** Shubin Framework (companion-документ)
**Версия:** 2.0

**Для кого:** любой член команды — разработчик, PM, дизайнер, CEO, доменный эксперт. Технический бэкграунд не требуется. Если умеете писать markdown и нажимать commit в GitHub Desktop — этого достаточно.

**Что это:** модуль фреймворка, реализующий метод «LLM wiki» Андрея Карпати для управления командными знаниями. LLM (Claude Code или Cowork) не просто читает базу знаний — она **активно её поддерживает**: компилирует сырые источники в связные страницы, обновляет cross-references, отмечает противоречия.

**Чем не является:** не заменой ADR, не заменой feature-спек, не таск-бордом. Knowledge-модуль отвечает только за **внешние и доменные знания**: интервью пользователей, конкурентов, персон, индустриальных трендов, исследований.

**Связанные документы:**
- `framework.md` — полный фреймворк
- `lite.md` — упрощённая версия для соло/MVP
- `non-tech.md` — как работать с git без терминала

---

## Перед стартом: что это и зачем

### Какую проблему решает модуль

В команде накапливаются **сырые знания о внешнем мире**:
- Транскрипты интервью с пользователями
- Статьи о конкурентах
- Подкасты об индустрии
- Research-отчёты
- Звонки с клиентами
- Скриншоты других продуктов

Всё это лежит в Google Drive, Notion, закладках браузера, Slack-чатах. Через три месяца:
- Никто не помнит, в каком интервью была ключевая цитата пользователя про pricing
- Конкуренты X и Y сливаются в «какого-то конкурента» в голове у всех
- У всех своя версия персоны «small business owner»
- Те же доменные термины переобъясняются каждому новому человеку

**Knowledge-модуль это решает.** Вы кладёте сырые материалы в одну папку. LLM читает и компилирует их в структурированные страницы: персоны, концепты, конкуренты, цитаты из интервью. Страницы связаны через `[[wikilinks]]`. Когда приходят новые источники, LLM обновляет существующие страницы и отмечает противоречия.

### Откуда идея

В апреле 2026 Андрей Карпати опубликовал паттерн «LLM wiki» — базу знаний, которую поддерживает сама LLM. Три слоя:

1. **raw/** — иммутабельные источники
2. **wiki/** — страницы, сгенерированные LLM, со ссылками между собой
3. **AGENTS.md** — инструкции, как LLM поддерживает wiki

Для personal knowledge management этим уже пользуются тысячи людей. Для команд подход адаптируется почти 1-в-1, с маленькими нюансами (см. ниже).

### Чем это отличается от остального фреймворка

В Shubin Framework есть другие слои:
- `decisions/` — ADR (наши решения). Пишут люди.
- `features/` — feature-спеки. Пишут люди.
- `patterns/` — code-паттерны. Пишут люди.
- `journal/feedback/` — сырой append-only feedback. Пишут люди.

Это **human-maintained** слои. LLM их читает, не модифицирует.

Knowledge-модуль — **единственный слой, где LLM является мейнтейнером**. Люди кладут источники, LLM компилирует. В этом фундаментальное различие.

## 1. Архитектура модуля

### 1.1. Структура папок

```
your-product/
├── decisions/
├── features/
├── patterns/
├── journal/
└── knowledge/                    ← этот модуль
    ├── raw/                      ← иммутабельно, люди пишут/кладут
    │   ├── user-interviews/
    │   │   ├── 2026-04-10-acme-corp.md
    │   │   └── 2026-04-15-beta-startup.md
    │   ├── competitors/
    │   │   ├── 2026-04-02-notion-pricing-changes.md
    │   │   └── 2026-04-18-linear-blog-on-ai.md
    │   ├── industry/
    │   │   └── 2026-04-05-gartner-report-summary.md
    │   └── calls/
    │       └── 2026-04-12-client-discovery-X.md
    │
    ├── wiki/                     ← LLM-maintained
    │   ├── personas/
    │   │   ├── small-business-owner.md
    │   │   └── enterprise-buyer.md
    │   ├── concepts/
    │   │   ├── product-led-growth.md
    │   │   └── usage-based-pricing.md
    │   ├── competitors/
    │   │   ├── notion.md
    │   │   └── linear.md
    │   ├── themes/
    │   │   └── ai-in-productivity-tools.md
    │   └── index.md              ← каталог wiki
    │
    └── AGENTS.md                 ← инструкции для LLM-мейнтейнера
```

### 1.2. Два правила, которые нельзя нарушать

**Правило 1: raw/ иммутабельно.**
Положив файл в `raw/`, вы **не редактируете его содержимое**. Нашли ошибку? Добавьте новый файл с правкой; старого не трогайте. Это критично: `raw/` — ваша «фактическая история», и если её редактировать, LLM компилирует wiki на сдвигающемся фундаменте.

**Правило 2: wiki/ поддерживает LLM.**
Вы не пишете в `wiki/` руками, кроме редких случаев (правка очевидной галлюцинации). Wiki-страницы создаёт и обновляет LLM по вашим командам. Если вы видите что-то неправильное в `wiki/personas/small-business-owner.md` — **исправляйте источники в raw/ или попросите LLM обновить**, не редактируйте wiki напрямую.

Эти два правила — основа. Нарушение первого убивает доверие к системе. Нарушение второго превращает wiki в обычный черновик, и LLM перестаёт понимать, что она делает.

## 2. Что именно идёт в raw/

### 2.1. Форматы источников

Всё в markdown. Если источник не в markdown, конвертируйте перед загрузкой:

| Источник | Как конвертировать |
|---|---|
| Транскрипт Zoom / Meet | Скопируйте в .md, добавьте frontmatter |
| Статья из блога | Сохраните как markdown (расширения браузера, MarkDownload) |
| PDF-отчёт | Claude Code умеет читать PDF и делать markdown-summary. Или `pdftotext` и почистите |
| Подкаст-транскрипт | Скопируйте из транскрипционного сервиса (YouTube Premium, Otter.ai) |
| Скриншоты конкурентов | Обычно не нужны в raw/ как есть. Лучше — markdown-заметка с описанием и ссылкой на скриншот |
| Email-thread | Скопируйте текст, отредачьте PII |
| Заметки звонка | Пишите прямо в markdown во время звонка |

### 2.2. Frontmatter для raw-файлов

Минимум:

```markdown
---
type: user-interview     # user-interview | competitor | industry | call | other
source: Acme Corp        # компания/источник/имя
date: 2026-04-10
author: @alice           # кто загрузил в репозиторий
---
```

Это помогает LLM понять контекст при компиляции. Работает и без frontmatter, но хуже.

### 2.3. Что класть, что не класть

**Класть:**
- Всё, что содержит фактическое знание о внешнем мире
- Первичные транскрипты и заметки
- Статьи, исследования, отчёты

**Не класть:**
- Внутренние ADR (они в `decisions/`)
- Feature-спеки (они в `features/`)
- Командные операционные документы (статусы, графики)
- Чужие материалы без скрабинга PII (имена клиентов, номера телефонов, финансовые данные)
- NDA-материалы, которые нельзя распространять внутри команды

### 2.4. Принцип «больше — лучше»

В обычной документации каждый документ — стоимость поддержки. В knowledge-модуле **каждый raw-источник — актив**. LLM работает с объёмом. 50 интервью дают качественно более полезный wiki, чем 5. Чем больше raw, тем точнее персоны и концепты.

Не бойтесь «захламить» raw/. Это не хлам — это данные.

## 3. Что выходит в wiki/

### 3.1. Типы wiki-страниц

Четыре основных типа:

**Personas** (`wiki/personas/`). Скомпилированные описания пользовательских сегментов на основе интервью и звонков. Что болит, как сейчас решают, что говорят дословно, какой workflow.

**Concepts** (`wiki/concepts/`). Ключевые идеи из домена и индустрии. «Что такое product-led growth», «как работает usage-based pricing», «что такое zero-party data». Одна страница = один концепт.

**Competitors** (`wiki/competitors/`). Профили конкурентов. Pricing, фичи, позиционирование, недавние изменения, их сильные и слабые стороны. Компилируется из статей, блогов, скриншотов.

**Themes** (`wiki/themes/`). Трендовые темы из индустрии. «AI in productivity tools», «shift to vertical SaaS», «consolidation in observability». Синтез по нескольким источникам.

### 3.2. Формат wiki-страницы

Пример: `wiki/personas/small-business-owner.md`:

```markdown
---
type: persona
last_updated: 2026-04-18
sources: 8
---

# Small Business Owner

## Characteristics
- 5–50 сотрудников
- Сам принимает решения о выборе инструментов
- Бюджет: $50–500/мес на инструмент
- Технически грамотен на базовом уровне, но не разработчик

## Pain points
- Фрагментация инструментов: email, CRM, invoicing — всё в разных местах
  (упоминается в [[raw/user-interviews/2026-04-10-acme-corp]], [[raw/calls/2026-04-12-client-discovery-X]])
- Нет времени настраивать сложные системы
  (упоминается в [[raw/user-interviews/2026-04-15-beta-startup]])

## Quotes
> «Готов платить больше, если это сэкономит мне час в неделю»
> — [[raw/user-interviews/2026-04-10-acme-corp]]

> «Пробовал Notion, HubSpot, Airtable. Всё слишком сложно для нас»
> — [[raw/calls/2026-04-12-client-discovery-X]]

## Related
- [[wiki/concepts/product-led-growth]]
- [[wiki/competitors/notion]]
- [[wiki/themes/tool-consolidation]]

## Contradictions / open questions
- [[raw/user-interviews/2026-04-10-acme-corp]] говорит «бюджет до $100».
  [[raw/calls/2026-04-12-client-discovery-X]] говорит «готов платить $500».
  Возможно, два разных под-сегмента. Требует уточнения.
```

### 3.3. Что делает wiki-страницу хорошей

- **Каждое утверждение ссылается на raw-источник.** Без этого страница — галлюцинация.
- **Цитаты — verbatim из интервью.** Без перефразирования и реформулирования.
- **Cross-links через [[wikilinks]]** — нативный механизм Obsidian.
- **Секция «Contradictions» явная.** Если источники расходятся — это отмечено, а не сглажено.
- **Обновляется при появлении новых источников**, не переписывается с нуля.

## 4. AGENTS.md модуля: инструкции мейнтейнеру

### 4.1. Зачем отдельный AGENTS.md

В корне репозитория уже есть `AGENTS.md`. Но `knowledge/` нужен **отдельный** `AGENTS.md`, потому что поведение LLM здесь другое: не «читать при работе с кодом», а «активно поддерживать wiki».

Этот файл живёт в `knowledge/AGENTS.md` и читается Claude Code / Cowork при работе с этой папкой.

### 4.2. Рабочий пример AGENTS.md

```markdown
# Knowledge module: maintainer instructions

## Твоя роль

Ты поддерживаешь wiki/ как LLM-maintainer по методу Карпати.
Люди кладут сырые источники в raw/, ты компилируешь их в wiki/.

## Принципы

1. raw/ — иммутабельно. Никогда не редактируй файлы в raw/.
2. wiki/ — твоя зона. Создавай и обновляй страницы.
3. Каждое утверждение в wiki/ должно ссылаться на raw-источник
   через [[wiki-link]]. Нет источника — не пиши.
4. Прямые цитаты — verbatim из raw/. Никогда не перефразируй.
5. Противоречия — выноси в явную секцию «Contradictions»,
   не сглаживай.

## Команды пользователя и реакция на них

### "ingest <file> в knowledge"
1. Прочитай файл в raw/.
2. Определи, какие wiki-страницы уже существуют и относятся к теме.
3. Обнови существующие страницы новыми фактами и цитатами.
4. Если нужны новые страницы (новая персона, новый концепт,
   новый конкурент) — создай со всем набором секций.
5. Обнови index.md.
6. Отчитайся: какие страницы тронул, какие новые,
   какие противоречия выявил.

### "что мы знаем о <X>"
1. Сначала ищи wiki-страницы по теме.
2. Если страница есть — отвечай в её терминах, цитатами и ссылками.
3. Если страницы нет, но в raw/ есть релевантное —
   предложи создать wiki-страницу.
4. Никогда не выдумывай факты, которых нет в raw/ или wiki/.

### "обнови <страницу>"
1. Прочитай все raw-источники, на которые она ссылается.
2. Прочитай все новые raw/ файлы с момента last_updated.
3. Обнови секции, сохрани структуру.
4. Обнови last_updated и sources count.

### "проверь wiki на противоречия"
1. Пройдись по всем wiki-страницам.
2. Найди утверждения, противоречащие друг другу
   через разные raw-источники.
3. Помечай в секции «Contradictions» на соответствующих страницах.
4. Верни короткий отчёт пользователю.

## Лимиты

- Не создавай wiki-страницы молча. Отчитайся, что создал.
- Не удаляй wiki-страницы. Если страница устарела —
  помечай «deprecated» в frontmatter и сообщай.
- Если raw-источники противоречат друг другу — не выбирай
  «правильный». Записывай оба и помечай.
- Если пользователь просит «переписать» или «улучшить стиль»
  wiki-страницы — отказывайся. Wiki отражает источники,
  а не твой литературный вкус.
```

### 4.3. Что здесь тонко

Инструкции написаны как явное «ты», с конкретными командами и триггерами. Это не общий style guide. Claude Code следует таким явным инструкциям лучше, чем абстрактным принципам.

Критично: **запрет выдумывать факты и обязательность ссылаться на источники**. Без этого wiki за неделю превращается в галлюцинированное болото, и никто ему не доверяет.

## 5. Workflow

### 5.1. Недельный цикл

**В любой день недели:** вы накидали новых raw-источников за прошедшую неделю. Интервью, звонки, статьи, отчёты.

**Ingest-сессия (20–40 минут, 1–2 раза в неделю):**
1. Откройте Claude Code в папке репозитория (или Cowork, если предпочитаете).
2. Команда: «Иди в knowledge/AGENTS.md, прочитай инструкции. Затем сделай ingest всех новых файлов в raw/ за прошедшую неделю».
3. Claude читает каждый новый raw-файл, обновляет wiki.
4. После каждого файла — git commit («ingest: <file>»). Это даёт историю компиляции.
5. После всей сессии — короткий отчёт от Claude: что обновлено, что создано, какие противоречия найдены.

**Раз в месяц:** запускайте команду «проверь wiki на противоречия». Это поднимает накопленные несоответствия между источниками.

### 5.2. Ad-hoc запросы

В течение недели, когда кому-то нужно знание:

- «Что мы знаем про e-commerce пользователей?» → Claude идёт в wiki/personas/, возвращает синтез.
- «Как конкурент X позиционирует pricing?» → Claude идёт в wiki/competitors/, возвращает ответ.
- «Какие пользователи упоминали интеграции со Slack?» → Claude грепает по raw/ и wiki/, возвращает цитаты со ссылками.

В этом основная ценность модуля: команда получает доступ к общему знанию через вопросы на естественном языке.

### 5.3. Кто в команде кладёт что за источник

| Роль | Что обычно загружают |
|---|---|
| PM / Product | Интервью пользователей, discovery-звонки |
| Sales / Success | Звонки с клиентами, их возражения и wishlists |
| CEO / Founder | Разговоры с инвесторами, индустриальные интервью |
| Designer | Usability-тесты, тренды UX |
| Developer | Технические статьи, релизы конкурентов |
| Marketer | Industry-отчёты, позиционирование конкурентов |

Каждый — владелец своей категории raw/. Все вместе читают wiki/.

### 5.4. Когда НЕ использовать knowledge-модуль

- **В вашем репозитории только технические решения и ADR.** Тогда модуль не нужен — он про внешний мир, не про внутренние решения.
- **У вас меньше 10 raw-источников за первые 2 месяца.** Модуль окупается на объёме. Настраивать ради трёх интервью — overkill.
- **Команда не готова делать ingest-сессию раз в неделю.** Без этой дисциплины wiki будет ложно-устаревшим, что хуже, чем wiki вовсе нет.

## 6. Setup: пошагово для всех ролей

### 6.1. Технический человек (разработчик, тех-лид): 20 минут

1. В репозитории: создайте папку `knowledge/` с подпапками `raw/` и `wiki/`.
2. Создайте `knowledge/AGENTS.md` на основе шаблона из 4.2.
3. Создайте `knowledge/raw/user-interviews/`, `raw/competitors/`, `raw/industry/`, `raw/calls/` — по реальным потребностям команды.
4. Создайте `knowledge/wiki/index.md` с пустым списком — LLM заполнит.
5. Commit + push.
6. Объясните остальной команде, куда класть raw (см. 6.2).

### 6.2. Нетехнический человек (PM, дизайнер, CEO): 15 минут

Эта секция предполагает, что у вашей команды уже настроен Claude Code (на машине разработчика) и/или Cowork (десктоп-приложение для работы с репозиторием). Если нет — попросите разработчика настроить сначала. Для нетехнического вклада в raw/ хватает любого из двух; ingest-сессия обычно запускается разработчиком через Claude Code, но запросы к wiki можно делать через Cowork самостоятельно.

Если ещё не настроен git — начните с `non-tech.md`, секции 2–3.

Затем:
1. В клоне репозитория откройте `knowledge/raw/<подпапка>/`.
2. Создайте новый файл: `2026-04-18-<что-это>.md`.
3. Добавьте frontmatter (шаблон в 2.2).
4. Вставьте содержимое (транскрипт, текст статьи, заметки).
5. Сделайте commit в GitHub Desktop с сообщением `raw: <что это>`.
6. Push.

Через какое-то время (на следующей ingest-сессии) Claude обновит wiki/ сам. Результат можно посмотреть в `knowledge/wiki/`.

Если нужен срочный «что мы знаем про X» — попросите разработчика запустить Claude Code с запросом. Или подключите Cowork и спросите сами.

### 6.3. Запуск ingest впервые

Когда в raw/ накопилось 5–10 файлов:

1. Откройте Claude Code в корне репозитория (разработчик).
2. Промпт:
   ```
   Прочитай knowledge/AGENTS.md. Затем сделай ingest всех файлов
   в knowledge/raw/. Начни с user-interviews/.
   После каждого обновления wiki делай git commit
   с коротким описанием.
   Если найдёшь противоречия между источниками —
   помечай в соответствующих wiki-страницах.
   Сначала покажи план, потом отчёт.
   ```
3. Claude предлагает план. Проверьте, что выглядит разумно. Скажите «go».
4. Через 5–15 минут получаете отчёт: «создано X страниц, обновлено Y, найдено Z противоречий».
5. Просмотрите wiki/. Если что-то очевидно галлюцинировано — скажите Claude «вот это утверждение неверно, вот правильный источник».

## 7. Lite-версия модуля (для соло/MVP)

Полный модуль для команды 1–2 — overkill. Упрощённая версия:

```
notes/
├── decisions.md
├── patterns.md
├── gotchas.md
├── todo.md
└── knowledge/                ← добавляется при >5 raw
    ├── raw/
    │   └── *.md              ← плоско, без подпапок
    └── wiki/
        └── *.md              ← плоско
```

**Отличия от командной версии:**
- Без подпапок в raw/ — всё плоско. Для соло классификация по типу (interview/article/call) — overkill.
- wiki/ тоже плоско — 5–10 страниц, не иерархия.
- AGENTS.md для knowledge/ — копия шаблона из 4.2, минус команды вроде «проверь противоречия» — избыточно для 5–10 страниц.
- Ingest — раз в 2–3 недели, не еженедельно.

**Когда добавлять модуль в lite:**
- Начали делать discovery-интервью или user research
- Накопилось 5+ raw-источников, которые хотите «связать»
- Работаете над продуктом с нетривиальным доменом, нужна база терминов/концептов

До этого — не добавляйте. Четырёх базовых файлов lite-версии достаточно.

**Почему это важно для lite.** Для pet-проектов или MVP ingest-loop через Claude **полностью автоматизирован**. Кладёте 3 интервью в `raw/`, говорите «ingest» — получаете компилированную персону за 5 минут. Не нужно сидеть и писать summary руками. Для соло это экономит часы.

## 8. Честные предупреждения

### 8.1. LLM галлюцинируют

Даже с инструкцией «всегда ссылайся на источник» Claude иногда:
- Атрибутирует цитату не тому интервью
- Делает обобщение, которого нет в источниках
- Создаёт связь между источниками, которая поверхностна

**Антидот:** правило **человек читает каждую wiki-страницу после первого создания**. Затем — на значимых обновлениях. Проверяйте 2–3 цитаты против источников. Если есть галлюцинации — скажите Claude «это неверно, перепроверь».

### 8.2. Wiki молча устаревает

Источники добавляются, но ingest не запускается → wiki становится устаревшим. Сигнал: `last_updated` в frontmatter старше 2 недель при том, что команда активно работает.

**Антидот:** ingest-сессия — ритуал, а не «сделаем, когда будет время». Раз в неделю, 30 минут, независимо от количества новых источников. Если ничего нет — Claude скажет «новых источников нет», это нормально.

### 8.3. Соблазн редактировать wiki руками

Человек садится, видит неправильное предложение в `wiki/personas/small-business-owner.md`. Думает: «я просто исправлю». Через месяц половина wiki отредактирована руками, и LLM-компиляция перестаёт соответствовать реальности.

**Антидот:** если что-то в wiki неверно — **найдите raw-источник**, который это породил, исправьте (или добавьте файл-правку в raw/), запустите ingest. Или явно попросите Claude: «на странице X утверждение Y неверно, перепроверь и обнови». Не редактируйте wiki напрямую.

Исключение: если Claude галлюцинировал что-то, *чего нет в источниках* — удалите руками и запишите в commit message. Это отладка, не редактирование контента.

### 8.4. Это не RAG и не поиск

Ожидания, которые модуль НЕ закрывает:
- Не отвечает на «найди все упоминания X в raw/» за миллисекунды (для этого — grep)
- Не генерирует SQL-запросы по источникам
- Не заменяет семантический поиск по большому корпусу документов

Knowledge-модуль — **компиляция, не retrieval**. Ценность не в быстром поиске, а в синтезированном знании на читаемых страницах со ссылками.

### 8.5. Нет автоматической перекомпиляции

Когда вы добавляете raw-файл, wiki **не обновляется автоматически**. Нужно явно запустить ingest. Можно настроить GitHub Action, который запускает ingest на каждый push, — но это автоматизация поверх, не часть базового модуля. Не обещайте команде «просто кидайте файл и wiki обновится сам», если автоматизация не настроена.

## 9. Интеграция с остальным фреймворком

### 9.1. Связь knowledge → ADR

ADR могут ссылаться на wiki-страницы: «опираясь на потребности [[wiki/personas/small-business-owner]], выбираем X». Это честное обоснование с прослеживаемостью до реальных интервью.

Шаблон ADR может содержать опциональную секцию **Research context**:
```markdown
## Research context
- Persona: [[wiki/personas/small-business-owner]]
- Competitor benchmark: [[wiki/competitors/notion]]
```

### 9.2. Связь knowledge → feature

В feature-странице секция Why может ссылаться на wiki-персоны вместо размытых описаний целевой аудитории:

```markdown
## Why
Эта фича закрывает боль из [[wiki/personas/small-business-owner]]:
«фрагментация инструментов».
Success criteria подтверждены интервью
[[knowledge/raw/user-interviews/2026-04-10-acme-corp]].
```

### 9.3. Связь knowledge → feedback loop

Когда новые сигналы пользователей появляются в `journal/feedback/` (тикеты, отзывы, наблюдения за usage), их можно периодически промоутить в `knowledge/raw/`, если они несут доменное знание, а не конкретный bug-репорт.

Различие:
- `journal/feedback/` — операционный поток (тикет, отзыв, наблюдение)
- `knowledge/raw/` — материал для компиляции в знание о мире

Не каждый feedback попадает в knowledge. Решает человек (обычно PM).

## 10. Финал

Knowledge-модуль — **единственный слой Shubin Framework, где LLM не просто читает, а поддерживает**. Это ближе всего к оригинальной идее Карпати, и именно здесь AI-агент даёт эффект, который сложно получить руками: **компиляция на масштабе**.

Если у вас команда, которая много занимается discovery (интервью, research, competitive intel) — модуль окупается на 10–20 raw-источниках. Если нет — не включайте. Пустой или полупустой knowledge-слой хуже, чем его отсутствие: создаёт впечатление знания, которого нет.

**Дисциплина, которая реально работает:**
1. Класть raw честно — сырое содержимое, без фильтрации «красиво/некрасиво».
2. Запускать ingest по расписанию, не по настроению.
3. Читать wiki глазами хотя бы раз в месяц, проверять на галлюцинации.
4. Никогда не редактировать wiki руками (с редкими debug-исключениями).
