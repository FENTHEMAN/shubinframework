# Shubin Framework: Intro

**Полное название:** Context-First Framework для маленьких команд с AI — hub
**Версия:** 1.0
**Время чтения:** 10 минут

Этот документ — entry point в Shubin Framework. Если вы впервые слышите про фреймворк — начните здесь. Если уже читали основной — можете пропустить.

---

## 1. Что это

Shubin Framework — это набор практик работы с markdown-vault как **общим контекстом команды** для людей и AI-агентов (Claude Code, Cowork).

Центральная идея: **контекст команды должен жить в читаемом людьми и AI виде, в git, вне чьей-то головы**.

Когда практики реально работают, команда получает:
- Claude Code перестаёт переоткрывать отвергнутые решения каждую сессию
- Новый член команды онбордится заметно быстрее, чем через устные объяснения
- Уход ключевого человека — серьёзная боль, но не катастрофа
- Команда не теряет по 10 минут на каждое «а почему мы это сделали вот так?»

Когда не работает:
- Команда пишет в vault, но никто не читает
- Tech lead не продавливает практики
- Документы обновляются формально, без реальной связи с решениями

**Без дисциплины всё мертво.** Это не «поставил и забыл» — это долгосрочная практика, которая окупается на 2–3 месяце при реальном использовании.

---

## 2. Откуда

Фреймворк основан на двух источниках:

**Метод Андрея Карпати «LLM wiki»** — паттерн personal knowledge management, где LLM поддерживает markdown-wiki: компилирует raw-источники в структурированные страницы, обновляет cross-references. В Shubin Framework этот метод реализуется как **опциональный модуль** (`knowledge/`), применимый к командному research и knowledge management.

**Классическая engineering-практика** — ADR (Architecture Decision Records, Michael Nygard), feature-specs, runbooks, event taxonomy. Это core большинства глав фреймворка.

Комбинация: core engineering-дисциплина + опциональная LLM-wiki для domain knowledge.

---

## 3. Для кого и не для кого

**Для кого:**
- Команды 2–8 человек на стадии от MVP до early traction
- Использующие AI-агентов (Claude Code, Cowork) в ежедневной работе
- Имеющие хотя бы одного человека, готового **продавливать практики** (tech lead, founder)
- Работающие с продуктом, где накопление контекста даёт отдачу (SaaS, B2B tools, complex B2C)

**Не для кого:**
- Команды 15+ человек — нужны более тяжёлые процессы, enterprise practices
- Команды без tech lead'а, готового ревьюить и подталкивать — умрёт за квартал
- Pet-проекты и MVP у соло-разработчика — см. lite-версию
- Агентства с flow клиентских проектов — другой формат knowledge flow

Если вы соло или в команде 1–2 человек — начните с `lite.md`, полная версия избыточна.

## 4. Навигация по документам

Фреймворк состоит из 5 документов. Каждый — для конкретной потребности.

### `framework.md` — **основной документ, для tech lead**
Всё, что нужно для настройки и ведения vault команды 2–8 человек. Vault-структура, ADR-процесс, итерационный цикл фичи, роли, разработка, дизайн с Figma, инфраструктура, тестирование, ритуалы, анти-паттерны, поэтапное внедрение. Читать первым, если вы tech lead.

### `lite.md` — **для соло и команд 1–2 человек на MVP**
Упрощённая версия: два файла на старте (`decisions.md`, `todo.md`), `CLAUDE.md` в корне проекта, 15 минут настройки. Без ADR, без ритуалов, без ролей. Плюс — упрощённый knowledge-модуль для MVP с внешними источниками.

### `knowledge.md` — **LLM-wiki модуль для командного knowledge management**
Реализация метода Карпати для команды. Нужен, когда вы накапливаете внешние источники (интервью, статьи про конкурентов, research), и хотите, чтобы LLM их автоматически компилировала в связанные wiki-страницы. Читают все роли — разработчики, PM, дизайнеры, CEO.

### `non-tech.md` — **для PM, дизайнера, CEO без инженерного опыта**
Как работать с git и vault без терминала. GitHub Desktop, Obsidian, 4 операции (clone/pull/commit/push), типичные ситуации, что можно и что нельзя. 45 минут на настройку.

### `non-engineering.md` — **минимум по Finance/Legal/Security/Data/Support**
Зоны, которые не относятся к core engineering, но всё равно нужны. Минимальный набор практик для команды 2–8: что делать, что отложить до traction. Чтобы не было ни cargo cult enterprise, ни катастрофы от «security это потом».

## 5. Быстрый решатель «с чего начать»

**Я соло или нас двое, делаем MVP или pet-проект.**
→ Только `lite.md`. Остальное не нужно.

**Я tech lead команды 3–8, настраиваем процесс с нуля.**
→ `intro.md` (этот) → `framework.md` → по мере появления соответствующих нужд — остальные.

**Я PM / дизайнер / founder, команда уже внедрила фреймворк, я должен работать с vault.**
→ Только `non-tech.md`. Если команда использует knowledge-модуль — плюс раздел 8 там же.

**У нас команда разработчиков, и мы начинаем накапливать юзер-исследования, конкурентный анализ.**
→ К основному фреймворку добавить `knowledge.md`. Читают все роли.

**Я основатель, приближается первый enterprise-клиент / security-вопросы.**
→ `non-engineering.md`. Это минимум. При серьёзных compliance-требованиях — нужны специалисты.

## 6. Обзор концепции vault

```
vault/ (один git-репозиторий)
│
├── stable/              меняется редко
│   ├── stack.md         технологический стек
│   ├── team.md          кто за что
│   └── glossary.md      доменные термины
│
├── active/              основная работа
│   ├── features/        каждая фича — папка со специ- и прогрессом
│   ├── decisions/       ADR (сквозная нумерация)
│   └── patterns/        накопленные инженерные паттерны
│
├── log/                 append-only, не редактируется
│   ├── feedback/        сигналы от пользователей
│   ├── incidents/       постмортемы
│   └── retrospectives/  итоги ретро
│
├── knowledge/           ← опциональный модуль (метод Карпати)
│   ├── raw/             immutable внешние источники
│   ├── wiki/            LLM-compiled страницы с cross-refs
│   └── CLAUDE.md        инструкция LLM как maintainer
│
├── COMMON.md            схема для AI-агентов
├── README.md
└── index.md             каталог
```

Ключевые отличия зон:

| Зона | Кто пишет | Кто редактирует | Характер |
|---|---|---|---|
| stable/ | люди | люди (редко) | декларативное |
| active/ | люди | люди | процессное |
| log/ | люди | никто (append-only) | хроника |
| knowledge/raw/ | люди | никто (immutable) | источники |
| knowledge/wiki/ | LLM | LLM | compiled |

---

## 7. Два принципа фреймворка

**Принцип 1: Append-only для истории, mutable для состояния.**
Лог решений, фидбек, incidents — только дописываются. Roadmap, feature-страницы, статусы ADR — меняются. Структура папок прямо отражает этот принцип: `stable/` и `active/` — mutable, `log/` — append-only, `knowledge/raw/` — immutable.

**Принцип 2: Одна правда на каждый факт.**
Каждое решение, каждая фича, каждый компонент имеют ровно одно каноническое место жительства. Дубли расходятся. Если что-то описано в Linear, Figma и vault — через месяц три описания разойдутся.

Остальные практики — следствия этих двух.

## 8. Чего ожидать по времени

- **Первый месяц:** настройка, первая фича в полном workflow. Видимого выигрыша нет.
- **2–3 месяц:** ритуалы становятся привычкой, появляются первые возвраты (Claude Code перестаёт переоткрывать решения, команда начинает ссылаться на ADR).
- **4–6 месяц:** compounding — накопленный контекст даёт реальное ускорение. Новый человек онбордится за 1–2 недели. Velocity выше, чем без фреймворка.
- **После 6 месяцев:** расширение по факту потребности (enterprise clients, compliance, масштабирование).

Если через 3 месяца vault не используется — это не провал фреймворка, это сигнал о более глубокой проблеме: команде не хватает ясности в том, что вы строите и зачем. Никакие инструменты эту ясность не заменят.

## 9. Критические анти-паттерны

Читайте эти анти-паттерны в основном документе раздел «Анти-паттерны». Но если можно выделить самые частые:

1. **Vault как write-only medium** — пишут, никто не читает
2. **Один писатель** — только tech lead реально поддерживает
3. **AI генерирует, никто не проверяет** — половина ADR с hallucinated trade-offs
4. **Ritual theater** — встречи проходят, решения не принимаются

Противоядие общее: **дисциплина tech lead'а + видимые ссылки на vault в ежедневной работе (PR, обсуждения)**. Без этого фреймворк — кладбище документов.

## 10. Следующий шаг

Определите по разделу 5, какой документ читать следующим. Обычно это один из:
- `lite.md` (если соло/MVP)
- `framework.md` (если tech lead команды 3–8)
- `non-tech.md` (если вы не разработчик в уже-внедрившей команде)

Не пытайтесь прочитать всё сразу. Документы читаются под конкретную задачу: «настраиваю vault команды», «не знаю git, надо контрибьютить», «подключаю knowledge-модуль». Читайте под задачу.
