# Shubin Framework

**Полное название:** Context-First Framework для маленьких команд с AI
**Автор:** Fen
**Дата:** 2026-05-02

**Для кого:** команды 2–8 человек, делающие продукт от MVP до early traction, с AI-агентами (Claude Code, Cowork) как полноценными участниками.

**Что это:** строительные леса практик, которые окупаются на 2–3 месяце использования при наличии дисциплины.

**Чем не является:** готовым рецептом из коробки, заменой инженерного вкуса, серебряной пулей, готовым enterprise-плейбуком.

---

## Место в семье документов

Это основной документ, для тех-лида команды. Компаньоны закрывают конкретные нужды:

- `lite.md` — соло-разработчики и команды 1–2 на MVP
- `knowledge.md` — LLM-wiki модуль по методу Карпати
- `non-tech.md` — работа с репозиторием и git без терминала (PM, дизайнер, CEO)
- `non-engineering.md` — минимум по Finance, Legal, Security, Data, Support для команды 2–8
- `guides/feature-cycle-example.md` — полный разобранный пример feature-цикла из главы 5

Этот файл покрывает архитектуру репозитория, процесс ADR, цикл итерации фичи, разработку, инфраструктуру и ритуалы. По под-теме — смотри соответствующего компаньона.

---

## Перед стартом: три честных предупреждения

**1. Фреймворк не окупается мгновенно.** В первый месяц вы тратите время на настройку без видимого выигрыша. На 2–3 месяце он начинает окупаться. К 6-му месяцу команда движется быстрее, чем без фреймворка. Если вы не готовы к этому лагу — не начинайте.

**2. Без дисциплины ничего не работает.** Репозиторий, который никто не читает, — кладбище. ADR без ревью — формальность. Ритуалы без последствий — театр. Ничего из этого не работает в режиме «настроил и забыл».

**3. Не всё это для вас.** Документ описывает зрелое состояние. Стартуете с минимума и расширяетесь по мере роста. Расширяйтесь только когда реально упёрлись в отсутствие практики, а не «фреймворк сказал».

## 1. Базовая идея

### 1.1. Проблема

Маленькая команда с AI-агентами упирается в две боли:

- **Амнезия AI-агента.** Каждая сессия Claude Code начинается с чистого листа. Решения, принятые вчера, нужно объяснять заново завтра.
- **Потеря человеческого контекста.** Через три месяца никто не помнит, почему API возвращает camelCase, почему выбрана именно эта БД, почему в onboarding четыре шага, а не один.

Обе проблемы — одна и та же: **контекст живёт в головах, а не в читаемой форме**.

### 1.2. Решение

Вынести командный контекст в **единый markdown-репозиторий**, читаемый одинаково людьми и AI-агентами. Всё важное записано там. Что не записано — не существует ни для новичка, ни для Claude Code.

Преимущества markdown-в-git перед Notion или Confluence: работают plaintext-поиск и grep; AI-агенты читают и редактируют нативно; версионирование из коробки; нет vendor lock-in; нет интеграций и нет API rate limits.

### 1.3. Два принципа, на которых стоит всё

**Принцип 1: Append-only для истории, mutable для состояния.** Decision log, feedback, инциденты — только дописываются. Roadmap, feature-страницы, статусы ADR — меняются. Это разделение даёт честную историю без хаоса.

Уточнение этого принципа в v2: **mutable state живёт в GitHub, а не в markdown frontmatter**. Status, ownership, priority, due dates, прогресс — это GitHub Issues, Milestones и PR-таймлайны. Markdown держит *спецификацию*; GitHub держит *где оно сейчас*. Последствия проходят через главы 4 и 5.

**Принцип 2: Один источник правды на факт.** Каждое решение, каждая фича, каждый компонент имеет ровно один канонический дом. Дубликаты расходятся. Если что-то описано в Linear, Figma и репозитории — через месяц у вас три расходящихся описания.

Других принципов нет. Всё остальное — следствие.

### 1.4. Связь с методом Карпати

Ядро репозитория (главы 2–12) — классическая инженерная практика: ADR, feature spec, паттерны, runbook'и. Пишут люди, LLM читает.

Отдельный **опциональный модуль** `knowledge/` реализует метод «LLM wiki» Андрея Карпати: люди кладут сырые источники (интервью, статьи, исследования), LLM компилирует их в связные wiki-страницы (персоны, концепты, конкуренты). Это единственный слой, где LLM — мейнтейнер.

Модуль вынесен в отдельный документ: `knowledge.md`. Подключайте, если команда много занимается discovery и хочет общую синтезированную память о внешнем мире. До этого — нет.

## 2. Структура репозитория

### 2.1. Минимум, чтобы начать

Не создавайте двенадцать папок в первый день. Стартуйте с канонических файлов в корне и двух рабочих папок плюс журнал:

```
your-product/
├── ARCHITECTURE.md       карта системы
├── STACK.md              стек
├── GLOSSARY.md           глоссарий доменных терминов
├── TEAM.md               кто за что отвечает (опционально)
├── AGENTS.md             инструкции для AI-агентов
├── README.md
│
├── decisions/            ADR, сквозная нумерация
├── patterns/             накопленные инженерные паттерны
│
└── journal/              append-only
    ├── feedback/         датированные сырые feedback-заметки
    ├── incidents/        постмортемы
    └── retros/           недельные и квартальные retro-саммари
```

Заметьте, чего здесь *нет*: папки `features/`. Фичи живут в GitHub Issues, не в markdown — см. главу 4.1.

Опционально: `knowledge/` (LLM-wiki модуль, см. `knowledge.md`); `guides/` (длинные примеры и командные рецепты); `runbooks/` (инцидент-плейбуки).

Папки добавляются по реальной необходимости. `runbooks/` появляется после первого инцидента. `guides/` — когда есть разобранный пример длиннее двух страниц. Не раньше.

### 2.2. Почему такая раскладка

Три осознанных выбора.

**Плоский корень вместо вложенных обёрток.** Никакого `vault/` или другой обвёрточной директории. Сам репозиторий — обёртка. Лишний префикс на каждом пути добавляет шума без информации; `grep` и `git log` работают чище на плоской структуре.

**UPPER_CASE для канона, lower_case для рабочих папок.** Контрибьютор, скользящий взглядом по корню, должен моментально узнавать meta-документы (`ARCHITECTURE.md`, `STACK.md`, `GLOSSARY.md`, `AGENTS.md`) без вчитывания. Рабочие папки — `decisions/`, `patterns/`, `journal/` — в нижнем регистре, потому что это *содержимое*, не канон.

**Классификация по режиму изменений, а не по домену.** `journal/` назван по поведению (append-only хроника), а не по типу контента. Альтернатива — `product/`, `engineering/`, `design/` — ассоциируется с орг-ролями и провоцирует силосы: «это от engineering, я туда не лезу». Схема по режиму изменений напрямую отражает Принцип 1.

Если команда приходит из устоявшегося Notion-пространства с доменными названиями — альтернативная раскладка допустима. Главное: выбрать одну и не смешивать. Менять структуру через полгода — болезненная миграция.

### 2.3. Что где живёт

**Корневые канонические файлы** меняются редко. `ARCHITECTURE.md` — высокоуровневая карта продукта, bird's-eye view, границы модулей, инварианты; matklad-style, прозой, держится в пределах ~300 строк. `STACK.md` — выбранный стек с однострочными причинами. `GLOSSARY.md` — доменные термины, общие для всей команды; модуль-специфичные термины — в глоссариях модулей (глава 3). `TEAM.md` — кто за что отвечает (опционально в команде 2–3 человека). `AGENTS.md` — правила для AI-агентов (см. 2.5).

**Рабочие папки** меняются активно. `decisions/` — ADR, со сквозной нумерацией (`0001-...md`); нумерация никогда не сбрасывается, без подпапок. `patterns/` — инженерные code-паттерны и gotchas, рождённые из реальной повторяемости (7.4). У фичей рабочей папки нет специально: каждая фича — это GitHub Issue, а не markdown-файл (4.1).

**Журнал** — append-only: записи датированы и никогда не редактируются. `journal/feedback/` несёт файлы вида `2026-04-18-user-X-interview.md`; `journal/incidents/` — постмортемы; `journal/retros/` — недельные и квартальные саммари ретро.

Доменные знания о внешнем мире (персоны, конкуренты, концепты) не живут в `GLOSSARY.md`. Они либо накапливаются в обычных markdown-файлах, если команда не использует knowledge-модуль, либо в `knowledge/wiki/`, если использует.

### 2.4. Почему не Notion

Справедливый вопрос: чем это отличается от хорошо организованного Notion-воркспейса? Четырьмя вещами. AI-агенты читают и пишут нативно, без API и токенов — просто файловые операции. Версионирование git даёт `git log` и `git blame` — археологию решений, которую в Notion не воспроизвести. Grep и ripgrep работают мгновенно, в том числе из CLI для агентов. Нет vendor lock-in — переезд на другую платформу — это `git clone`.

Notion MCP существует, но markdown-в-git быстрее, дешевле и надёжнее для команды с Claude Code на каждой машине.

### 2.5. AGENTS.md как схема

Один файл в корне репозитория, описывающий правила для AI-агентов. `AGENTS.md` — кросс-инструментальный стандарт, принятый Claude Code, Cowork и другими, — мы используем это имя, а не изобретаем своё.

Минимальный пример:

```markdown
# Repository schema

## Структура

- decisions/ — ADR, immutable после Accepted
- patterns/ — инженерные паттерны, mutable
- journal/ — append-only история, не редактировать
- knowledge/ — опциональный LLM-wiki модуль (см. knowledge/AGENTS.md)

Фичи живут в GitHub Issues, не в этом репо. Issue — это спека;
структуру обеспечивает Issue Template.

## При работе с кодом

1. Прочти связанный GitHub Issue по фиче in-flight
2. Проверь decisions/ на релевантные ADR
3. Проверь patterns/ на похожие паттерны
4. На новое значимое решение — предложи создать ADR

## При значимой находке

Если нашёл gotcha, паттерн или решение, которое стоит сохранить —
спроси пользователя, записать ли в patterns/.
Не создавай файлы молча.

## Чего не делать

- Не редактировать journal/ (append-only)
- Не создавать markdown-страницу спеки для фичи — спека и есть Issue
- Не создавать пустые папки «на будущее»
- Не работать с knowledge/ по общим правилам — у него свой AGENTS.md
```

`AGENTS.md` растёт по мере того, как команда учится. Честно: в первый месяц он короткий и часто обновляется. К 3-му месяцу стабилизируется.

### 2.6. Репозиторий как отдельный репо

Один markdown-репозиторий на продукт, независимый от кода. Другие репозитории подключают его одним из трёх способов: git submodule (сложнее, но версии запиннены); git clone как peer (проще, но у всех локальные копии); symlink после clone (чтобы `AGENTS.md` каждого код-репозитория ссылался на канонический).

Выбор фиксируется в первом ADR команды. Смешивать методы внутри одной команды — плохо: выбираете один и придерживаетесь.

## 3. Масштабирование: когда одного репо мало

### 3.1. По умолчанию: один репо на продукт

Один продукт, один markdown-репозиторий со структурой выше. Это работает, пока команда держит единую ментальную модель продукта, любой участник в курсе всех значимых работ in-flight, а фичи регулярно пересекают границы доменов.

Большинство команд 2–8 остаются здесь навсегда. Не оптимизируйте под масштаб, которого нет.

### 3.2. Анти-паттерн: дробление по тех-слою

`backend-vault/`, `frontend-vault/`, `design-vault/` — **нет**. Знание о фиче целостно: у фичи транзакций есть бизнес-инвариант, схема БД, API, UI и UX — всё в одной голове. Разделение по тех-слою создаёт силосы и стоит больше, чем экономит.

### 3.3. Триггеры для split на модули

Дробить на модули, только когда срабатывает **минимум 2 из 3**:

1. **Команда поделилась.** Две команды со своими циклами и своими PM — не «один на бэке, второй на фронте».
2. **Расходятся релизные каденции.** Модули деплоятся независимо, на разных графиках.
3. **Доменный язык реально различен.** Термины конфликтуют между модулями или общие определения в одном глоссарии становятся неудобными.

Один триггер сам по себе — недостаточно. Одна команда с двумя релизными пайплайнами всё ещё одна команда.

### 3.4. Гибрид: монорепо с `modules/`

Когда триггеры срабатывают, держим один репозиторий и делим внутри:

```
product/
├── ARCHITECTURE.md           карта системы и границы модулей
├── GLOSSARY.md               только cross-module термины
├── decisions/                cross-module ADR
├── patterns/                 cross-module паттерны
├── modules/
│   ├── transactions/
│   │   ├── ARCHITECTURE.md   внутренности модуля
│   │   ├── GLOSSARY.md       authorize/capture/refund — только здесь
│   │   ├── decisions/
│   │   └── patterns/
│   └── notifications/
│       └── ...
```

Один репозиторий, два слоя: корень несёт cross-module контекст; каждый модуль — свой. Фичи по-прежнему живут в GitHub Issues — обычно в код-репо релевантного модуля; cross-module работа открывает параллельные Issues с явными ссылками между ними.

### 3.5. Правило роутинга

Решая, куда идёт документ, задайте три вопроса:

- Если завтра присоединится совершенно новый модуль — нужен ли ему этот документ? → корень.
- Если переименовать модуль — пойдёт ли документ за ним? → модуль.
- Если этот модуль выделить в отдельный продукт — останется ли документ внутри? → модуль.

Вопросы строже, чем кажутся. Большинство документов попадает в одно ведро однозначно.

### 3.6. Глоссарий: самое строгое правило

Корневой `GLOSSARY.md` несёт только термины, используемые между модулями. Доменно-специфичные термины — `authorize`, `capture`, `refund` — внутри модуля, который ими владеет. Без этого правила корневой глоссарий за год пухнет до 200 терминов, и его никто не читает.

### 3.7. Паттерны: то же правило

Если паттерн повторяется в трёх модулях, поднимаем в корневой `patterns/`. Иначе живёт в модуле. Периодический рефакторинг расположения паттернов — норма: паттерны мигрируют по мере роста системы.

## 4. Единица работы: Feature + ADR

### 4.1. Фича как GitHub Issue

Каждая фича — это **GitHub Issue** в релевантном код-репозитории. В этом репозитории нет markdown-страницы спеки и нет папки `features/`.

Для фичей P0 и P1 структуру обеспечивает Issue Template. Шаблон лежит в `.github/ISSUE_TEMPLATE/feature.md` каждого код-репо:

```markdown
## Why
Один абзац: на кого влияет, какую проблему решает, почему сейчас.

## Scope
Границы: что мы строим. Что НЕ строим.

## Success criteria
Измеримый чек, что фича готова. Один пункт на критерий.

## Decisions
Ссылки на ADR в markdown-репо:
- [ADR-0024: OAuth library choice](../../<markdown-repo>/decisions/0024-oauth-library.md)

## Design
Ссылка на Figma-frame, если применимо.
```

Для P2-работы и багфиксов шаблон опционален.

Что Issue даёт нативно, без markdown:

- **Статус** — open / closed.
- **Владелец** — assignee.
- **Приоритет** — лейблы (`P0`, `P1`, `P2`).
- **Даты создания и цели** — `created_at` плюс due date milestone.
- **Прогресс-таймлайн** — комментарии и панель Linked Pull Requests.
- **Декомпозиция** — sub-issues или task lists в теле.
- **Итерация** — milestone.
- **Обсуждение** — комментарии.

Когда имплементирующий PR мержится с `Closes #N`, Issue закрывается автоматически. Закрытый Issue — канонический документ того, что построили: исходные Why, Scope и Success criteria, обсуждение, связанные PR и резолюция — всё в одном месте.

Что живёт в markdown, а не в Issue:

- **Решения (ADR)** — иммутабельны; Issues мутабельны и могут быть удалены. ADR, написанный год назад, должен читаться сегодня дословно, даже если Issue, который его триггернул, отредактирован или закрыт.
- **Паттерны** — переиспользуемое инженерное знание, не привязанное к одной фиче.
- **Архитектура, глоссарий, стек** — устойчивое знание.
- **Журнал** — feedback, инциденты, retro: append-only хроника.

Связь от Issue к ADR однонаправленная: тело Issue ссылается на файл ADR. ADR не обязан ссылаться на Issue — ADR может пережить любой конкретный Issue.

Markdown держит **знание**. GitHub держит **workflow**. Жёсткое правило: ни один markdown-файл не отражает state Issue. Файл `features/<slug>.md`, повторяющий тело Issue, разойдётся с ним за несколько дней; файл без state'а его задублирует. Так или иначе — вторая копия гниёт. Удаление markdown-слоя целиком закрывает дилемму.

### 4.2. ADR

Architectural Decision Record — документ о решении, которое сложно отменить, влияет на несколько частей системы и которое потребует объяснения через три месяца.

Хорошая граничная эвристика: если решение можно откатить за день работы одного человека — ADR не нужен. Если откат — неделя и больше — ADR нужен.

Формат:

```markdown
# ADR-0015: <короткое название решения>

## Status

Accepted — 2026-04-17

## Context

Какая ситуация, какие ограничения.

## Options considered

Варианты с плюсами и минусами. Даже отклонённые полезны: через шесть месяцев
вспомните, почему не пошли туда.

## Decision

Выбранный вариант плюс одно-два предложения о том, почему.

## Consequences

Что теряем, что выигрываем, что становится сложнее.

## Related

Ссылки на фичи и сервисы.
```

Нумерация сквозная и никогда не сбрасывается. Устаревшие ADR получают статус `Deprecated` или `Superseded by ADR-NNNN`.

**ADR не получают GitHub Issue для tracking.** ADR — единица *истории решений*, а не работы in-flight. Поле `Status` (`Accepted`, `Superseded by ADR-NNNN`) — часть иммутабельного документа, а не state, который трекается где-то ещё. Это единственное исключение из правила «state живёт в GitHub» — и оно исключение, потому что state'а нет, есть только иммутабельная запись.

Если команда использует knowledge-модуль, ADR может опционально содержать секцию **Research context** со ссылками на `knowledge/wiki/personas/` и `knowledge/wiki/competitors/` — это даёт прослеживаемость до реальных интервью и исследований.

### 4.3. ADR-first — идеал, к которому стремимся

Идеально — значимое решение начинается с PR, меняющего только ADR. Обсуждение, апрув, мерж. Затем код.

На практике в маленькой команде тех-лид часто решает в голове и пишет ADR, когда часть кода уже написана. Это нормально. Важно, чтобы ADR в итоге существовал, даже если написан постфактум.

Опасность: ADR ради ADR. Сорок ADR в первом квартале — половина из них про выбор HTTP-клиента — и практика превращена в bikeshedding. Рабочая норма — 5–15 ADR за первые шесть месяцев активной разработки.

## 5. Цикл итерации фичи

В цикле шесть шагов, не девять. Каждый шаг лёгкий, потому что state живёт там, где ему естественно (GitHub), а знание — там, где оно сохраняется (markdown). Параллельной markdown-спеки, которую надо синхронизировать с Issue, не существует.

Разобранный пример с конкретными числами, PR и цитатами — в `guides/feature-cycle-example.md`.

### 5.1. Обзор цикла

1. **Idea → Issue** — открыть GitHub Issue по Feature Template.
2. **Discussion (опционально)** — комментарии в Issue.
3. **ADR (если нужен)** — открыть PR в markdown-репо с `decisions/NNNN-<slug>.md`.
4. **Implementation** — ветка, код, PR с `Closes #N`.
5. **Merge** — PR мержится, Issue закрывается автоматически; закрытый Issue — исторический документ.
6. **Retrospective signal (если нужен)** — запись в `journal/incidents/` или `journal/retros/`, если что-то пошло не так.

### 5.2. Шаг 1 — Idea → Issue

Любой — инженер, PM, дизайнер, фаундер — открывает GitHub Issue по Feature Template (глава 4.1). Заполняет Why, Scope и Success criteria. Лейблы приоритета и milestone могут проставляться сразу или после обсуждения. Ссылки на дизайн идут в body, когда готовы.

Issue — это спека. Никакого отдельного документа, никакого `features/<slug>/index.md`, никакого PR в markdown-репо на этом шаге.

### 5.3. Шаг 2 — Discussion (опционально)

Комментарии в Issue. PM и один разработчик прогоняют sanity-check по scope и success criteria; дизайнер подключается на UX-тяжёлых фичах. Если обсуждение порождает нетривиальное техническое решение — переходим к шагу 3. Иначе пропускаем.

### 5.4. Шаг 3 — ADR (если нужен)

Срабатывает, когда работа содержит реальный выбор между опциями со значимыми trade-off'ами (эвристика обратимости из 4.2). Разработчик, который будет имплементить, открывает PR в markdown-репо с `decisions/NNNN-<slug>.md` по шаблону ADR. Ревьюверы комментируют PR. Мержим, когда принято.

ADR затем линкуется обратно из секции `## Decisions` тела Issue. Связь однонаправленная: Issue ссылается на ADR; ADR не обязан упоминать Issue.

Большинство фичей не требуют ADR — не пишите его на тривиальную работу. ADR-инфляция — отдельный анти-паттерн (12.11).

### 5.5. Шаг 4 — Implementation

Разработчик ассайнится на Issue, создаёт ветку в код-репо, запускает Claude Code с телом Issue, связанным ADR и Figma-ссылкой как контекстом, генерирует и ревьюит код, пишет тесты на критичную логику. Код-PR открывается с `Closes #N`, чтобы Issue закрылся автоматически на merge.

PR появляется в панели «Linked pull requests» Issue автоматически — никакой ручной перекрёстной ссылки. Если в имплементации обнаруживается необходимость нового ADR — пишем (назад к шагу 3) и линкуем из Issue.

На каждый push крутится PR-пайплайн и деплоится preview-environment, если у команды он есть (глава 9). Дизайнер и PM проверяют на preview; расхождения — комментарии в PR. Обычно 1–3 итерации.

### 5.6. Шаг 5 — Merge

Approve от одного-двух ревьюверов, merge в main, автоматический deploy в production где применимо. `Closes #N` авто-закрывает Issue. Закрытый Issue — с исходной спекой, обсуждением, связанными PR и резолюцией — становится историческим документом фичи.

Никакого ручного обновления статуса нигде, потому что поля статуса вне Issue не существует.

### 5.7. Шаг 6 — Retrospective signal (если нужен)

Если что-то пошло не так — инцидент, пропущенный дедлайн, blow-up по scope, неожиданная реакция пользователей — открываем запись в `journal/incidents/` или `journal/retros/` с date-префиксом (например, `2026-04-18-<slug>.md`). Ссылаемся на Issue и любые ADR.

Если команда проверяет post-release метрики (success criteria, PostHog, фидбек пользователей) и результаты важны для будущей работы, те же папки `journal/` — место, где это записать. Если используется knowledge-модуль, эти записи идут в следующий ingest-цикл wiki.

Если фича уехала гладко и повела себя как ожидалось — **запись в журнал не нужна**. Закрытый Issue уже и есть документ. Не пишите retro ради retro.

### 5.8. Когда реальность расходится с планом

Типично: на шаге 4 разработчик понимает, что подход, выбранный в ADR, не работает. Не молчать и не править код тихо, забыв про ADR. А: обновить статус исходного ADR на `Superseded by ADR-NNNN`, написать новый ADR с новым решением и причиной отказа от старого, обновить body Issue, если изменился scope, и оставить комментарий в Issue с резюме разворота. Пятнадцать минут. Через шесть месяцев никто не чешет затылок, почему код говорит одно, а ADR — другое.

## 6. Роли и AI-агенты

### 6.1. Базовые роли

В команде 2–8 роли — это функции, не должности. Один человек может нести 2–3 роли; важно, что каждая функция явная.

| Функция     | Что делает                       | Чем владеет                                             |
|-------------|----------------------------------|---------------------------------------------------------|
| Product     | Держит «зачем мы это делаем»     | Issue Why/Scope/Success criteria, `journal/feedback/`   |
| Design      | Визуальная и UX-логика           | Figma-frames, линкуемые из Issue                        |
| Engineering | Код и архитектура                | `patterns/`, `decisions/`, код-репо                     |
| Ops         | Инфра, деплой, инциденты         | `runbooks/`, `infra/`                                   |

Специализированные роли (DevOps, QA, Security, Data) добавляются по реальной нужде, а не по плану. Сигнал «пора» — когда существующая команда стабильно тратит больше 20% времени в этой зоне.

### 6.2. AI-агенты как часть команды

У каждого AI-агента есть **конфигурация**, которая делает его дисциплинированным членом команды, а не general-purpose чат-ботом.

- **Claude Code в каждом код-репо** — со своим `AGENTS.md`, который ссылается на корневой `AGENTS.md` репозитория.
- **Cowork** — настроен один раз на машине человека, который им пользуется.
- **Claude Design** — опционально и осторожно; только для одноразовых прототипов и презентаций (см. 8).

Это не «шесть виртуальных должностей в таблице». Это **один-два активных агента на человека в день**, настроенных читать репозиторий и следовать его конвенциям.

### 6.3. Границы делегирования AI

AI-агенты делают: генерируют код по спеке; читают репозиторий и отвечают в контексте; пишут черновики ADR по запросу; помогают с рутинными изменениями; поддерживают knowledge-wiki, если модуль используется.

AI-агенты не делают: продуктовых решений; не действуют в production без human-in-the-loop; не пишут ADR без человеческого ревью (получите галлюцинированные trade-off'ы); не коммитят в main без PR и ревью.

## 7. Разработка

### 7.1. Выбор стека

Фреймворк **не диктует стек**. Выбор зависит от опыта команды, требований продукта, hiring в вашем регионе, бюджета runway. Главное правило: написать первый ADR команды о стеке. Какой бы ни был выбор, он должен быть аргументирован и записан.

**Прагматичный default для веб-CRUD:** TypeScript/Node.js. Один язык на стек, зрелая экосистема, проще нанимать. Скучный выбор — обычно правильный. Исключения: тяжёлые вычисления (ML-serving, видео и аудио), realtime со строгими требованиями к latency, embedded, системное программирование. Там другие факторы перевешивают скорость разработки.

### 7.2. Монолит vs микросервисы

Для команды 2–8 — **начинать с модульного монолита**. Микросервисы решают независимое масштабирование и независимую работу нескольких команд. Ни того, ни другого у вас на старте нет.

Признаки, что пора делить: больше 4–5 активных разработчиков наступают друг другу на ноги в одной кодовой базе, реальная нагрузка требует независимого скейла, требования регуляторной изоляции.

### 7.3. AGENTS.md на проект

Каждый код-репозиторий несёт свой `AGENTS.md` в корне, который ссылается на корневой `AGENTS.md` markdown-репозитория для общего контекста. Проект-специфичные правила — typescript strict, no `any`, правила форматирования, правила тестирования — в локальном файле проекта. Общие конвенции — в markdown-репо.

### 7.4. Паттерны

`patterns/` накапливает решения типичных задач: `data-fetching.md`, `forms.md`, `error-handling.md`, `auth.md`, `gotchas.md`. Паттерны рождаются **из реальных решений**, а не из общих best practices. Первая версия `forms.md` пишется, когда вы скопировали один и тот же подход в третий раз, не раньше.

### 7.5. API-контракты

Если у вас есть backend-сервис с HTTP API, его контракт фиксируется в machine-readable форме (OpenAPI для REST, GraphQL schema для GraphQL). Контракт — источник правды. Frontend-типы, клиенты для других сервисов и mock'и для тестов — всё генерируется из него. Расхождение фактических роутов и контракта ловится в CI.

## 8. Дизайн-процесс

Figma — источник правды для UI. Figma Professional ($15–25/мес за Dev seat); free-план не позволяет полезной работы с MCP-сервером (лимит 6 вызовов в месяц). Честная оценка Claude Design на 2026 год: ещё сырой — годится для одноразовых презентаций, ранних мокапов, одноразовых лендингов, но не для реальных продуктовых экранов и не для design system. Ситуация может измениться через 3–6 месяцев.

Design system живёт **в двух местах одновременно** — в Figma (component library, переменные для токенов) и в коде (`shared/design-system/` с компонентами, Tailwind config или CSS-переменные с теми же токенами) — и эти два синхронизируются вручную, но по согласованию: новый компонент сначала в Figma, потом в коде; design-изменения, которых ещё нет в коде → дизайнер открывает PR или таску; раз в месяц — 30-минутная встреча, проверяющая выравнивание. Детальная механика — структура файлов, Tokens Studio пайплайн, handoff workflow — варьируется от команды к команде; зафиксируйте под себя в `patterns/design-system.md`. Стартовый рецепт — в `guides/design-process.md`, если и когда понадобится.

Figma MCP в Claude Code позволяет разработчику вставить ссылку на Figma-frame и сгенерировать код по этому frame. Настройка: включить Dev Mode MCP Server в Figma; OAuth через `/mcp` в Claude Code. Лимиты: на платном Dev seat — 200 вызовов в день; ~25k токенов контекста на ответ (работайте поframe'но); точность около 80% — отступы и компоненты обычно правильно, edge cases (анимации, состояния) часто пропускаются.

Когда дизайнер пропускается: внутренние admin-страницы, простые формы из готовых компонентов, системные страницы (404, 500), транзакционные email. Когда дизайнер обязателен: пользовательские экраны не из готовых компонентов, изменения design system, лендинги, критические потоки. Серая зона обсуждается на недельной встрече.

## 9. Инфраструктура и деплой

**Платите за managed-сервисы, пока можете.** Self-managed Kubernetes-кластер на старте — это ещё один наёмный человек, которого у вас нет.

Типичный стартовый стек для веб-приложения: фронтенд-хостинг (Vercel, Cloudflare Pages, Netlify), бэкенд-хостинг (Railway, Fly.io, Render), БД (Supabase, Neon, PlanetScale), error tracking (Sentry), аналитика (PostHog для продукта, Plausible или Fathom для веба), CI/CD (GitHub Actions), секреты (Doppler, 1Password Secrets Automation, эквиваленты), uptime-мониторинг (Better Stack, UptimeRobot). Реалистичная стоимость на ранней стадии: $150–600/мес в зависимости от планов и нагрузки. Разобранную конфигурацию см. в `guides/infrastructure-starter-stack.md`.

Окружения: `dev` (локально, docker-compose) → `preview` (per-PR, изолированная БД через branching) → `staging` (опционально, копия production для ручного QA) → `production` (deploy только из main). Preview-environment'ы — огромный win для маленькой команды: дизайнер и PM видят фичу до merge.

PR-пайплайн, реалистичный минимум: lint и type-check, unit-тесты, build, deploy preview, опционально E2E против preview. Девятишаговый пайплайн красив на бумаге, но в маленькой команде становится flaky, блокирует мержи и в итоге выключается. Добавляйте шаги по мере роста, когда они ловят реальные баги.

Минимум observability: структурные логи (JSON) с trace ID, error tracking, uptime-мониторинг, базовые platform-метрики. Tracing через OpenTelemetry полезен, но требует времени — не в первый месяц. `runbooks/` пишется после первого инцидента соответствующего рода, не раньше. AI-агенты полезны для **диагностики** во время инцидента — они быстро проходят runbook. В production действуют только люди.

Секреты, железные правила: никогда не в git, никогда в чатах, разные для dev/staging/prod, ротация для production, журнал доступа к production-секретам. Реальность маленькой команды: ротация пропускается. Лучше «раз в полгода и реально делаем», чем «раз в 30 дней и никогда».

## 10. Тестирование

Универсальной test pyramid не существует. Ваша стратегия тестирования — это ADR: обсудить, записать, пересматривать раз в полгода.

Независимо от стратегии, всегда покрываем: критические пользовательские пути (5–10 в e2e — signup, payment, критические happy paths), бизнес-логику с деньгами или безопасностью (высокое покрытие), сложные алгоритмы (unit-тесты). Не покрываем: простой CRUD UI без логики, тривиальные геттеры и сеттеры, моки на моках.

Честный совет про e2e: начинайте с 2–3 тестов на самые важные потоки. Растите по мере того, как ловите реальные баги. Не пытайтесь покрыть всё — flakiness убьёт сьют.

В маленькой команде без QA-инженера тестирование — общая ответственность: разработчик пишет тесты на свою фичу, дизайнер ревьюит preview, PM проверяет функциональность перед промоушеном в production. Нанимать QA — когда стабильно видите больше одного значимого бага в неделю в production или релизы откладываются на ручное тестирование.

Баги — в GitHub Issues. Но их **причины и предотвращение** — в репозитории: `journal/incidents/` для постмортема, `patterns/gotchas.md` для «не делать так больше». Что сломалось / root cause / что изменили.

## 11. Ритуалы

Минимум, который реально работает.

- **Ежедневно (асинхронно, 2 мин/чел):** standup в Slack или Discord — что сделал, что делаю, блокеры.
- **Еженедельно (sync, 45–60 мин):** одна встреча всей команды — ретро прошлой недели + планирование следующей. Саммари ретро ложится в `journal/retros/`.
- **Ежемесячно (sync, 1 час):** финансы и продуктовые метрики вместе.
- **Ежеквартально (sync, полдня):** strategy offsite — пересмотр корневых канонических файлов, roadmap, North Star.

Если используется knowledge-модуль — добавьте 20–40-минутную **ingest-сессию** раз в 1–2 недели.

Чего избегать: ежедневные синхронные standup'ы (избыточно для четырёх человек), пятничные просмотры graph view репозитория (никто не смотрит на graph view дольше 30 секунд), отдельные квартальные security/compliance/data ревью (всё это ложится в один offsite), плановые bug-bash'и (откладываются на 6-й неделе, удаляются на 10-й).

AI-автоматизации вроде «еженедельный digest от Cowork» звучат красиво, но ломаются после трёх месяцев пренебрежения. Исключение — knowledge-модуль, где ingest **и есть** автоматизация и идёт в диалоге, не в фоне.

## 12. Анти-паттерны

Конкретные способы, которыми команды ломают этот фреймворк.

### 12.1. Репозиторий как write-only

Самый распространённый failure mode. Документы пишутся, но **никто не читает**. Через три месяца репозиторий — кладбище. Признаки: ADR никогда не цитируются в PR, новички в первую неделю не упоминают репозиторий, в обсуждениях звучит «где это написано?» — «не помню».

Антидот: ссылки на репозиторий в каждом значимом PR; регулярные «что говорит репозиторий по этому поводу?» в обсуждениях; онбординг через репозиторий, а не через устные объяснения.

### 12.2. Один писатель

Только один человек поддерживает репозиторий (обычно фаундер или тех-лид). Когда он в отпуске или уходит — всё замирает.

Антидот: в первые три месяца каждый член команды должен написать хотя бы одну страницу. Без этого привычка не формируется.

### 12.3. AI генерирует, никто не ревьюит

Claude Code пишет ADR по запросу, никто не ревьюит. Через полгода половина ADR содержит галлюцинированные обоснования.

Антидот: ADR проходят code review как любой PR. AI-генерация — черновик, не финал.

### 12.4. Театр ритуалов

Все формально приходят на встречи, пишут digest'ы, проводят retro. От этого ничего не меняется.

Антидот: каждая встреча заканчивается списком следующих действий с владельцами. Следующая встреча начинается с проверки этого списка.

### 12.5. Репозиторий как замена Notion

В репозитории появляются задачи, календари, kanban-доски. Это превращает его в операционный хаос.

Антидот: жёсткое разделение. Задачи — в GitHub Issues или Linear. Знания — в репозитории. Между ними — ссылки, не дубликаты.

### 12.6. Реизобретение GitHub

Хранение `status`, `owner`, `priority`, `due_date` и руками набитой секции `## Log` в markdown-frontmatter, когда GitHub Issues, Milestones и PR-таймлайны уже трекают это нативно. Симптом: feature-страницы со `status: in-progress` трёхмесячной давности.

Антидот: workflow живёт в Issue, знание — в markdown. Никакой frontmatter не держит workflow-state.

### 12.7. Feature page как кэш Issue

Markdown-файл `features/<slug>/index.md` (или по любому другому пути), который «спек-теневит» GitHub Issue — копирует Why, Scope и Success criteria из тела Issue в репо. Намерение обычно благородное: «хотим grep-нуть спеки», «хотим читать без открытия браузера», «AI-агентам легче найти». Результат — гарантированное гниение: как только тело Issue правится, а markdown — нет, два источника расходятся, и читатель не знает, какой актуален.

Антидот: фичи живут только в Issues. Markdown-слоя, который мог бы разойтись, нет. AI-агенты читают Issue через `gh issue view` или GitHub API; люди — на github.com. Issue Template обеспечивает те же поля, что обеспечил бы markdown-скелет, без второй копии для поддержки.

### 12.8. Vault на тех-слой

Один продукт делится на `backend-vault/`, `frontend-vault/`, `design-vault/`. Знание о фиче размазано по слоям, команде приходится триангулировать на любой вопрос.

Антидот: один репозиторий на продукт. Доменно-модульный split — только когда срабатывают 2 из 3 триггеров (3.3).

### 12.9. Преждевременный split на модули

Команда из трёх человек делит репозиторий на четыре модуля «потому что архитектурно так правильно». Создаёт навигационный оверхед там, где у реального workflow его не было.

Антидот: split — только когда срабатывают 2 из 3 триггеров одновременно. До этого — один репозиторий, плоско.

### 12.10. Дорогой старт

Команда из трёх, но Kubernetes, микросервисы, Kafka, data warehouse. Бюджет сложности исчерпан до первой строки бизнес-логики.

Антидот: начинать с монолита на managed-сервисах. Делить и усложнять — на реальной боли.

### 12.11. ADR-инфляция

Сорок ADR в первом квартале, половина — про выбор HTTP-клиента.

Антидот: эвристика «откатываемо за день — ADR не нужен».

### 12.12. Идеальный репозиторий в первый месяц

Команда тратит две недели на настройку структуры до того, как пишет код.

Антидот: стартуйте с двух рабочих папок и нескольких канонических файлов. Расширяйтесь по нужде.

### 12.13. Knowledge без ingest

Настроили knowledge-модуль, кладёте сырые источники, ingest никогда не запускаете. Wiki устаревший, команда думает «у нас есть knowledge base», но на деле — нет.

Антидот: ingest-сессия — ритуал, а не «когда будет время». См. `knowledge.md` 5.1.

### 12.14. Security/QA/аналитика «на потом»

«Сначала PMF, потом всё это». Результат: первый инцидент или утечка становятся катастрофой.

Антидот: минимум с первого дня — секреты в менеджере, error tracking, event taxonomy, privacy policy. Это одна-две недели работы, а не один день и не три месяца. Детали в `non-engineering.md`.

## 13. Поэтапный rollout

Реалистичные сроки для команды 2–8, стартующей с нуля. **Оценки сверху**: реальность будет медленнее.

### Месяц 1: Foundation

Цель: минимальная инфра и первая фича через полный workflow.

- Репозиторий, корневые канонические файлы (`STACK.md`, `GLOSSARY.md`, `AGENTS.md`, `README.md`), две рабочих папки (`decisions/`, `patterns/`), `journal/`.
- Первые 2–3 ADR: стек, базовая архитектура, monorepo vs polyrepo.
- Инфра: GitHub, фронтенд-хостинг, бэкенд-хостинг, БД, error tracking, secrets manager.
- Issue Template (`.github/ISSUE_TEMPLATE/feature.md`) в каждом код-репо.
- Первая фича — открыта как GitHub Issue, проведена через полный шестишаговый цикл (глава 5).
- CI-пайплайн на PR: lint, type check, тесты, preview.

Что реально происходит: настройка сервисов занимает дольше плана (первый неожиданный quirk съедает день); первый ADR про стек — черновик (реальное понимание приходит через 2–3 недели); первая фича обнажает несколько пробелов, которые придётся закрывать.

**Не подключайте** knowledge-модуль в первый месяц, даже если кажется «полезным заранее». Модуль окупается на 10+ сырых источниках, которых у вас почти точно ещё нет.

### Месяцы 2–3: Стабилизация

Цель: ритуалы и паттерны становятся привычкой.

- Design system с 5–10 компонентами в коде.
- `patterns/` с 3–5 написанными паттернами.
- Недельный командный ритуал стабилен.
- Первый инцидент с runbook'ом и постмортемом.
- Event taxonomy в `STACK.md` или отдельном `events.md` (см. `non-engineering.md`).
- Privacy policy, если нужна.

Возникают споры «это в markdown или в issue tracker» — решаем case by case через Принцип 2. Соблазн расширить структуру репозитория — сопротивляемся. Кто-то начинает игнорировать `AGENTS.md` — обсуждаем.

Если команда занимается discovery-работой, knowledge-модуль подключается на 2–3 месяце. До 10 сырых источников он не нужен.

### Месяцы 4–6: Compounding

Цель: накопленный контекст начинает работать на команду.

- База ADR (10–20) реально используется в обсуждениях.
- Новый человек онбордится за 1–2 недели.
- Claude Code в сессиях ведёт себя в стиле команды без постоянных напоминаний.
- Паттерны переиспользуются в новых фичах.
- Velocity команды выше, чем в месяцах 1–2.

Это точка, где фреймворк окупается. Если в этот момент команда сопротивляется репозиторию — пересматривайте подход или бросайте фреймворк. Он подходит не всем.

### После 6 месяцев

Расширение по нужде: enterprise-клиенты → компоненты из `non-engineering.md` (compliance, security review); 10K+ пользователей → роль Data и отдельный security review; 8+ человек → формальные спринты, выделенный QA, DevOps; несколько модулей расходятся → подумать о split на модули по главе 3.

Репозиторий растёт органически, не по плану.

## Финал

Этот фреймворк — строительные леса, не истина. Он окупается, если у команды есть дисциплина поддерживать репозиторий и уважение к накопленному контексту. Без этого — просто ещё одна папка устаревших документов.

Главное, что фреймворк даёт, — **операционный язык для людей, работающих с AI-агентами**. Когда репозиторий, `AGENTS.md`, паттерны и ADR настроены, вы перестаёте каждый раз заново объяснять Claude Code, как ваша команда принимает решения. Он читает и действует в контексте.

Второе по важности — **переживать уход людей**. В маленькой команде с одним экспертом в области его уход — катастрофа. С репозиторием, где решения записаны, — серьёзная боль, но не катастрофа.

Третье — **более быстрый онбординг**. Новый человек становится продуктивным за 1–2 недели, потому что командный контекст читаем, а не пересказываем.

Всё остальное — следствие.

---

**Вопросы, сомнения, контрпримеры** — их должно быть много. Фреймворк адаптируется, не копируется. Если через месяц использования вы обнаруживаете, что какая-то часть вам не подходит, — это не ошибка, это сигнал.
