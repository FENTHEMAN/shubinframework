## Перед тем как начать: три честных предупреждения

**1. Фреймворк окупается не сразу.** В первый месяц вы будете тратить время на настройку без видимого выигрыша. На 2–3-м месяце начинает окупаться. На 6-м — команда двигается быстрее, чем без фреймворка. Если вы не готовы к этому лагу — не начинайте.

**2. Без дисциплины всё мертво.** Vault без читателей — кладбище. ADR без ревью — формальность. Ритуалы без последствий — театр. Это не те практики, где «поставил и забыл».

**3. Не всё отсюда вам нужно.** Документ описывает зрелое состояние. Вы начинаете с минимума и расширяете по мере роста. Расширяться нужно, только когда реально упираетесь в отсутствие практики, а не когда «по фреймворку положено».

## 1. Центральная идея

### 1.1. Проблема

Маленькая команда с AI-агентами сталкивается с двумя болями:

- **Амнезия AI-агентов.** Каждая сессия Claude Code — с нуля. Решения, принятые вчера, завтра нужно объяснять заново.
- **Потеря контекста людьми.** Через 3 месяца никто не помнит, почему API возвращает camelCase, почему выбрали именно эту БД, почему онбординг в 4 шага, а не в 1.

Обе проблемы — одна и та же: **контекст живёт в чужих головах, а не в читаемом виде**.

### 1.2. Решение

Вынести контекст команды в **один markdown-репозиторий (vault)**, который читают одинаково и люди, и AI-агенты. Всё важное пишется туда. Что не написано — не существует для новых участников и для Claude Code.

Преимущества markdown в git перед Notion/Confluence:

- Plaintext-поиск и grep работают
- AI-агенты читают и редактируют нативно
- Версионирование из коробки
- Нет vendor lock-in
- Никаких интеграций, никакого API rate limit

### 1.3. Два принципа, на которых всё стоит

**Принцип 1: Append-only для истории, mutable для состояния.**
Лог решений, фидбек, incidents — только дописываются. Roadmap, feature-страницы, статусы ADR — меняются. Это разделение даёт честную историю без хаоса.

**Принцип 2: Одна правда на каждый факт.**
Каждое решение, каждая фича, каждый компонент имеют ровно одно каноническое место жительства. Дубли расходятся. Если что-то описано в Linear, Figma и vault — через месяц три описания разойдутся.

Больше принципов нет. Всё остальное — следствия.

### 1.4. Связь с методом Карпати

Ядро vault (главы 2-11) — это классическая engineering-практика: ADR, feature-specs, паттерны, runbooks. Люди пишут, LLM читает.

Отдельный **опциональный модуль** `knowledge/` реализует метод Андрея Карпати «LLM wiki»: люди кладут raw-источники (интервью, статьи, исследования), LLM компилирует их в связанные wiki-страницы (персоны, концепты, конкуренты). Это единственный слой, где LLM является maintainer'ом.

Модуль вынесен в отдельный документ: `knowledge.md`. Подключайте, если ваша команда делает много discovery-работы и хочет общую синтезированную память о внешнем мире. До этого — не нужен.

## 2. Vault: структура знаний команды

### 2.1. Минимальная структура для старта

Не создавайте 12 папок в первый день. Начинайте с **трёх категорий**:

```
vault/
├── stable/              редко меняется
│   ├── stack.md
│   ├── team.md
│   └── glossary.md
│
├── active/              основная работа
│   ├── features/        каждая фича — папка
│   ├── decisions/       ADR
│   └── patterns/        накопленные инженерные паттерны
│
├── log/                 только дописывается
│   ├── feedback/        папка с raw-фидбеком по датам
│   ├── incidents/       постмортемы
│   └── retrospectives/
│
├── README.md
├── COMMON.md            инструкции для AI-агентов
└── index.md
```

Опциональная четвёртая зона — `knowledge/` (модуль LLM-wiki, см. `knowledge.md`). Добавляется когда команда начинает накапливать внешние источники (интервью, research, competitive intel) и хочет их синтезировать.

Папки внутри трёх базовых зон добавляются, когда реально нужны. `design/`, `specs/openapi/`, `runbooks/` появляются, когда накопилось 2–3 файла, которые просятся туда. Не раньше.

### 2.2. Почему `stable/active/log`, а не `domain/product/engineering/feedback`

Это сознательный выбор режимной (по характеру изменений) классификации вместо доменной (по типу содержания). Обоснование:

- Режимная схема прямо отражает Принцип 1 («append-only для истории, mutable для состояния»). Структура подсказывает поведение.
- Меньше категорий — три вместо пяти-шести. Меньше моментов «куда положить».
- Доменная схема (`product/`, `engineering/`, `design/`) ассоциируется с организационными ролями и провоцирует силосы: «это от engineering, я туда не лезу».

**Альтернатива.** Если ваша команда привыкла мыслить доменно или приходит из устоявшегося Notion-пространства — режимные имена могут показаться абстрактными. В этом случае допустимо использовать `foundation/`, `domain/`, `product/`, `engineering/`, `feedback/`. Главное — **выбрать один подход и не смешивать**. Смена структуры через 6 месяцев — болезненная миграция.

Для новых команд по умолчанию рекомендую `stable/active/log`. Для команд, мигрирующих из Notion — обсудите явно в первом ADR.

### 2.3. Что живёт в каждой зоне

**stable/** (меняется редко):

- `stack.md` — технологический стек
- `team.md` — кто за что отвечает
- `glossary.md` — словарь доменных терминов
- `north-star.md` — главная метрика (при появлении)
- `events.md` — event taxonomy (при появлении, см. companion non-engineering)

**active/** (активная работа):

- `features/` — одна папка на фичу, см. главу 3
- `decisions/` — ADR, сквозная нумерация
- `patterns/` — инженерные паттерны кода, gotchas

**log/** (append-only, никогда не редактируется):

- `feedback/` — папка с файлами-заметками по датам: `2026-04-18-user-X-interview.md`
- `incidents/` — постмортемы
- `retrospectives/` — итоги еженедельных/ежеквартальных retro

Доменные знания о внешнем мире (персоны, конкуренты, концепты) не живут в stable/. Они либо накапливаются кратко в `stable/glossary.md`, либо — если команда использует knowledge-модуль — в `knowledge/wiki/`.

### 2.4. Почему это не Notion

Разумный вопрос: чем отличается от organized Notion workspace? Разница в четырёх вещах:

1. **AI-агенты читают и пишут нативно** — без API, без токенов, через обычные file operations.
2. **Git-версионирование** — видно кто, когда, что изменил. `git log` + `git blame` дают археологию решений.
3. **Grep и ripgrep** — поиск по тексту работает мгновенно, в том числе из CLI для агентов.
4. **Нет vendor lock-in** — переехать на другую платформу = `git clone`.

Notion MCP существует, но markdown в git быстрее, дешевле и надёжнее для команды с Claude Code на каждой машине.

### 2.5. COMMON.md как схема

Один файл в корне, описывающий правила работы с vault для AI-агентов. Инжектируется симлинком или импортом в каждый проект, где работает Claude Code.

Минимальный пример:

```markdown
# Vault Schema

## Структура

- stable/ — редко меняется, источник правды
- active/ — активная работа над фичами и решениями
- log/ — append-only история
- knowledge/ — опциональный LLM-wiki модуль (см. knowledge/CLAUDE.md)

## При написании кода

1. Читай active/features/<текущая-фича>/index.md
2. Проверь active/decisions/ на релевантные ADR
3. Проверь active/patterns/ на похожие паттерны
4. При новом значимом решении — предлагай создать ADR

## При значимой находке

Если обнаружил gotcha, паттерн или решение, которое стоит сохранить —
спроси у пользователя, не стоит ли зафиксировать в active/patterns/.
Не создавай файлы молча.

## Чего не делай

- Не редактируй log/ (append-only)
- Не создавай пустые папки "на будущее"
- Не работай с knowledge/ по общим правилам —
  там свой CLAUDE.md с другой механикой maintainer'а
```

COMMON.md растёт по мере того, как команда учится. Честно: в первый месяц он короткий и часто обновляется. К 3-му месяцу стабилизируется.

### 2.6. Vault как отдельный репозиторий

Один репозиторий vault, независимый от кода. Другие репозитории подключают его одним из способов:

- git submodule (сложнее, но версии фиксируются)
- git clone как peer (проще, но все держат локально)
- symlink после clone (для CLAUDE.md → COMMON.md)

Выбор — в первом же ADR команды. Смешивать способы в одной команде — плохо, выберите один и держитесь.

## 3. Единица работы: Feature + ADR

### 3.1. Feature как каноническая страница

Каждая фича живёт в `active/features/<slug>/index.md`:

```markdown
---
status: in-progress       # idea | validated | in-progress | shipped | killed
owner: @alice
priority: P1              # P0 | P1 | P2 | P3
created: 2026-04-17
target: 2026-05-01
---

# User Auth

## Why

Одно предложение, зачем это нужно и кому.

## Scope

Что именно делаем. Границы: что НЕ делаем.

## Success criteria

Измеримая проверка, что задача решена.

## Technical

Ссылки на [[decisions/0015-auth-strategy]]

## Design

Ссылка на Figma-фрейм.

## Implementation

PR: #123

## Log

- 2026-04-17: created
- 2026-04-20: backend ready
- 2026-04-22: shipped
```

Это единица работы, не тикет в Linear. Тикет ссылается сюда, не наоборот.

### 3.2. ADR

Architectural Decision Record — документ о решении, которое:

- Трудно откатить
- Влияет на несколько частей системы
- Потребует объяснения через 3 месяца

Хорошая эвристика границы: **если решение можно откатить за день работы одного человека — ADR не нужен**. Если откат — это неделя или больше — нужен.

Формат:

```markdown
# ADR-0015: <краткое название решения>

## Status

Accepted — 2026-04-17

## Context

Что за ситуация, какие ограничения.

## Options considered

Варианты с плюсами/минусами. Даже отвергнутые — полезны:
через полгода вы вспомните, почему не пошли этим путём.

## Decision

Выбранный вариант + одно-два предложения почему.

## Consequences

Что теряем, что приобретаем, что теперь стало труднее.

## Related

Ссылки на фичи и сервисы.
```

Нумерация сквозная, никогда не удаляется. Устаревшие — status: `Deprecated` или `Superseded by ADR-NNNN`.

Если команда использует knowledge-модуль, ADR может опционально содержать секцию **Research context** со ссылками на `knowledge/wiki/personas/` и `knowledge/wiki/competitors/` — это даёт прослеживаемость до реальных интервью и исследований.

### 3.3. ADR-first — идеал, к которому стремимся

В идеале значимое решение начинается с PR, меняющего только ADR. Обсуждение, апрув, merge. Потом — код.

На практике в маленькой команде tech-лид часто принимает решение в голове, пишет ADR уже после того, как код частично написан. Это нормально. Главное — чтобы ADR в итоге существовал, даже если он пишется задним числом.

Опасность: ADR ради ADR. Если у вас 40 ADR за первый квартал — скорее всего, половина про выбор HTTP-клиента, и вы превратили практику в bikeshedding. Рабочая норма — 5–15 ADR за первые 6 месяцев активной разработки.

## 4. Итерационный цикл фичи

Это центральная глава фреймворка. Здесь сходятся vault, Figma, Claude Code и git в одном рабочем процессе. Полный цикл от идеи до production со всеми артефактами.

### 4.1. Обзор цикла

```
       ┌─────────────────────────────────┐
       │  1. Идея в vault                │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  2. Спецификация (Why + Scope)  │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  3. ADR (если нужен)            │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  4. Дизайн в Figma              │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  5. Реализация (Claude Code)    │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  6. Preview + review            │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  7. Merge + deploy production   │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  8. Закрытие в vault            │
       └────────────┬────────────────────┘
                    ▼
       ┌─────────────────────────────────┐
       │  9. Feedback loop               │
       └─────────────────────────────────┘
```

Каждый шаг производит конкретные артефакты в конкретных местах.

> **Замечание про сквозной пример.** В этой главе используется вымышленная фича «онбординг» с конкретными цифрами (drop-off 60%, время 145с и т.д.). Все цифры — иллюстративные, не основаны на реальных данных. Цель — показать формат артефактов и переходы между шагами, не дать benchmark.

### 4.2. Шаг 1: Идея

**Кто:** обычно PM или founder. Может — любой член команды.
**Где:** новая папка `active/features/<slug>/` в vault, файл `index.md`.
**Артефакт:** черновик с YAML-frontmatter и одним предложением «зачем».

```markdown
---
status: idea
owner: @alice
priority: P2
created: 2026-04-18
---

# Onboarding redesign

## Why

Текущий онбординг — drop-off 60% на втором экране. Гипотеза:
слишком много полей сразу.

## Scope

TBD после обсуждения.
```

**Что в git:** создаём branch `feature/onboarding-redesign` в репо vault, коммитим черновик, открываем PR.
**Триггер следующего шага:** обсуждение на еженедельной встрече, решение «делаем».

### 4.3. Шаг 2: Спецификация

**Кто:** PM + один из разработчиков (для технической реалистичности) + дизайнер (для UX).
**Где:** тот же `index.md`, дополняется секциями.
**Артефакт:** заполненные секции Scope, Success criteria.

```markdown
## Scope

- 4 экрана вместо 5
- Прогресс-бар сверху
- Возможность пропустить onboarding
- Сохранение прогресса (если ушёл — продолжить)

НЕ делаем в этой итерации:

- Видео-туториал
- Персонализация по plan'у
- A/B тест разных вариантов

## Success criteria

- Drop-off на 2-м экране < 30% (сейчас 60%)
- Время прохождения медиана < 90 сек
- Замер через 2 недели после релиза
```

**Что в git:** обновление того же PR в vault, status меняется на `validated`.
**Триггер следующего шага:** approve PR, merge в main vault'а.

### 4.4. Шаг 3: ADR (если нужен)

**Когда нужен:** если в спецификации есть нетривиальное техническое решение, которое влияет на >1 часть системы. В нашем примере: «сохранение прогресса» — где хранить? localStorage? БД? Это решение → ADR.

**Кто:** разработчик, который будет реализовывать.
**Где:** `active/decisions/00NN-onboarding-progress-storage.md`.

```markdown
# ADR-0023: Хранение прогресса onboarding

## Status

Accepted — 2026-04-19

## Context

Юзер может покинуть onboarding и вернуться позже. Нужно сохранять,
на каком шаге остановился, что уже заполнил.

## Options considered

### 1. localStorage

- Простота, ноль серверной логики
  − Теряется при смене устройства

### 2. Поле в users table

- Кросс-устройственно
  − Migration

### 3. Отдельная таблица onboarding_progress

- Чистая separation
  − Лишняя таблица для одной фичи

## Decision

Поле `onboarding_state JSONB` в users table.

## Consequences

- Кросс-устройственно
  − Migration (см. PR в backend repo)
```

**Что в git:** PR в vault, тот же или отдельный, в зависимости от удобства.

### 4.5. Шаг 4: Дизайн

**Кто:** дизайнер. Если дизайнера нет — разработчик с готовыми компонентами.
**Где:** Figma-проект продукта, файл `[Feature] Onboarding`. Детали — глава 7.

**Артефакт:**

- Фреймы 4-х экранов в section `In progress`
- Прототип переходов
- Использование существующих компонентов из Design System

**В vault:** в `active/features/onboarding/index.md` секция Design дополняется ссылкой на Figma-страницу:

```markdown
## Design

Figma: https://figma.com/file/.../Onboarding

- Section "In progress" — финальные экраны
- Prototype: Play mode для демо переходов
```

### 4.6. Шаг 5: Реализация

**Кто:** разработчик + Claude Code.

**Подготовка:**

- `git pull` в репо vault — получить последние изменения
- `git checkout -b feature/onboarding` в репо кода

**Сессия Claude Code:**

```
cd ~/projects/my-app
claude

> Реализуй фичу onboarding. Контекст:
> - active/features/onboarding/index.md
> - active/decisions/0023-onboarding-progress-storage.md
> - Figma: [ссылка из index.md]
>
> Используй Figma MCP для чтения фреймов.
> Используй компоненты из shared/design-system.
> Сверься с active/patterns/forms.md.
```

Claude Code читает vault-страницы, ADR, через Figma MCP читает фреймы, генерирует код. Разработчик ревьюит, правит edge cases (Claude часто пропускает loading states и error handling), пишет тесты на критичную логику.

**Артефакт:** PR в репо кода с реализацией.

**В vault:** разработчик добавляет ссылку на PR в секцию Implementation:

```markdown
## Implementation

- Frontend PR: my-app#123
- Backend PR: api#45 (migration + endpoint)
```

### 4.7. Шаг 6: Preview + review

Автоматически: на каждый push в feature-branch — preview environment деплоится.

Дизайнер открывает preview, сравнивает с Figma. Расхождения — комментариями к PR. PM проверяет, что Scope выполнен. Разработчик правит, push'ит. Цикл повторяется, обычно 1–3 итерации.

### 4.8. Шаг 7: Merge + deploy production

Approve от 1–2 человек (зависит от размера команды; в команде 2 — от второго, в команде 4+ — от двух). Merge в main → автоматический deploy в production.

В этот момент фича жива для пользователей.

### 4.9. Шаг 8: Закрытие в vault

**Кто:** обычно тот, кто делал merge.
**Где:** `active/features/onboarding/index.md`.

```markdown
---
status: shipped     ← было: in-progress
shipped: 2026-04-28
---

...

## Log

- 2026-04-18: created
- 2026-04-19: ADR-0023 принят
- 2026-04-22: дизайн готов
- 2026-04-28: shipped, deploy v1.4.2
```

**Что в git:** маленький PR в vault только с обновлением статуса.

### 4.10. Шаг 9: Feedback loop

Через 2 недели после релиза (по success criteria из шага 2) PM или дизайнер:

1. Проверяет метрики в PostHog.
2. Создаёт `log/feedback/2026-05-12-onboarding-results.md` с цифрами, цитатами пользователей, conclusions.
3. Если по итогам нужны изменения — новая фича в `active/features/`.

Если команда использует knowledge-модуль — значимые инсайты из этого feedback-файла попадают в следующий ingest-цикл wiki.

### 4.11. Что происходит в каждом репозитории

За один цикл фичи среднего размера в vault-repo появляется 5–8 коммитов:

1. `feat: add onboarding redesign idea`
2. `spec: scope and success criteria for onboarding`
3. `adr: 0023 onboarding progress storage`
4. `design: add figma link for onboarding`
5. `impl: add PR links for onboarding`
6. `status: onboarding shipped`
7. `feedback: onboarding results`

В frontend-repo: branch `feature/onboarding`, PR с реализацией. В backend-repo: branch `feature/onboarding-progress`, PR с migration и endpoint.

Коммиты в vault маленькие и информативные. Это норма, не «слишком много коммитов».

### 4.12. Что делать, если реальность отклонилась от плана

Типичная ситуация: на шаге 5 разработчик понял, что выбранный в ADR подход не работает.

**Не делайте:** тихо менять код, забыв про ADR.

**Делайте:**

1. Обновите ADR-0023: status → `Superseded by ADR-0024`.
2. Создайте ADR-0024 с новым решением и причиной отказа от старого.
3. Обновите `active/features/onboarding/index.md`, если изменился scope.

Это занимает 15 минут. Через полгода никто не будет ломать голову, почему в коде одно, а в ADR — другое.

## 5. Роли и AI-агенты

### 5.1. Базовые роли

В команде 2–8 человек роли — это функции, не должности. Один человек может нести 2–3 роли; важно, чтобы каждая функция была явной.

| Функция     | Что делает                   | Владеет в vault                                               |
| ----------- | ---------------------------- | ------------------------------------------------------------- |
| Product     | Держит «зачем мы это делаем» | active/features/\* (секция Why), log/feedback/                |
| Design      | Визуальная и UX-логика       | active/features/\*/design                                     |
| Engineering | Код и архитектура            | active/patterns, active/decisions, features/\*/implementation |
| Ops         | Инфра, деплой, инциденты     | runbooks/, infra/                                             |

Специализированные роли (DevOps, QA, Security, Data) — добавляются при реальной потребности, не по плану. Сигнал «пора» — когда существующая команда тратит на эту зону больше 20% времени стабильно.

### 5.2. AI-агенты как часть команды

Ключевая идея: каждый AI-агент имеет **конфигурацию**, которая делает его дисциплинированным членом команды, а не универсальным чат-ботом.

- **Claude Code в каждом репозитории кода** — свой CLAUDE.md, который ссылается на COMMON.md vault
- **Cowork** — настраивается один раз на машине того, кто им пользуется
- **Claude Design** — опционально и осторожно: только для разовых прототипов и презентаций (см. 7.1)

Это не «шесть виртуальных должностей в таблице». Это **один-два активных агента на человека в день**, настроенных так, чтобы читать vault и следовать его конвенциям.

### 5.3. Границы делегирования AI

AI-агенты делают:

- Генерируют код по спецификации
- Читают vault и отвечают в контексте
- Пишут ADR-драфты по запросу
- Помогают с routine-изменениями
- Поддерживают knowledge-wiki (при использовании модуля) — компилируют raw в wiki

AI-агенты НЕ делают:

- Не принимают продуктовые решения
- Не действуют в проде без human-in-the-loop
- Не пишут ADR без ревью человека (иначе получаете hallucinated trade-offs)
- Не коммитят в main без PR и ревью

## 6. Разработка

### 6.1. Выбор стека

Фреймворк **не навязывает стек**. Выбор зависит от опыта команды, требований продукта, найма в вашем регионе, бюджета runway.

Главное правило: **пишите первый ADR команды о стеке**. Какой бы выбор ни был, он должен быть обоснован и зафиксирован.

Ключевые вопросы для ADR стека:

- Кто из команды уже знает технологию продуктивно?
- Реалистично ли нанять ещё одного разработчика на этот стек в вашем регионе?
- Какая экосистема библиотек и managed services вокруг?
- Какие компромиссы в compile time, error messages, learning curve?

**Прагматичный default для web-CRUD:** TypeScript/Node.js. Причины: один язык на весь стек, зрелая экосистема, найм проще. Скучный выбор — и обычно правильный.

**Исключения:** тяжёлые вычисления (ML serving, обработка видео), realtime с жёсткими требованиями по latency, embedded, системное программирование — там другие факторы перевешивают скорость разработки.

### 6.2. Монолит vs микросервисы

Для команды 2–8 человек **начинайте с модульного монолита**. Микросервисы решают проблему независимого масштабирования частей системы и независимой работы нескольких команд над одной системой. У вас нет ни того, ни другого на старте.

Признаки, что пора делить:

- > 4–5 активных разработчиков, мешающих друг другу в одной кодовой базе
- Реальная нагрузка, требующая независимого масштабирования частей
- Регуляторные требования, требующие изоляции

Пока этого нет — модульный монолит с чёткими границами модулей внутри.

### 6.3. CLAUDE.md для проекта

Ключевой конфигурационный файл в корне каждого репозитория кода. Содержит:

```markdown
# <Project name>

## Context

Общий контекст: ../vault/COMMON.md
Текущие фичи: ../vault/active/features/

## Stack

См. ../vault/stable/stack.md

## Перед написанием кода

1. Найти фичу в ../vault/active/features/
2. Проверить ../vault/active/decisions/ на релевантные ADR
3. Проверить ../vault/active/patterns/ на похожие паттерны
4. Проверить ../vault/active/patterns/gotchas.md

## Правила

- (специфичные правила проекта: typescript strict, no any, etc)
- (правила форматирования, линтинга)
- (правила тестирования)

## Deploy

См. ../vault/infra/deployment.md (если есть)
```

### 6.4. Паттерны в vault

По мере разработки в `active/patterns/` накапливаются решения типичных задач. Минимум для веб-приложения:

- `data-fetching.md` — как работаете с сервером
- `forms.md` — валидация и отображение ошибок
- `error-handling.md` — стратегия ошибок
- `auth.md` — как проверяется авторизация
- `gotchas.md` — на чём обожглись (самое ценное со временем)

Паттерны рождаются **из реальных решений**, не из best practices общего плана. Первая версия `forms.md` пишется, когда вы третий раз копируете один и тот же подход к форме.

### 6.5. API-контракты

Если у вас есть бэкенд-сервис с HTTP API, его контракт фиксируется в машиночитаемом виде (OpenAPI для REST, GraphQL schema для GraphQL).

Контракт — источник правды. Из него генерируются типы для фронтенда, клиенты для других сервисов, моки для тестов. Расхождение между реальными роутами и контрактом ловится на CI.

## 7. Дизайн-процесс: Figma + дизайн-система в коде

### 7.1. Реальное состояние Claude Design (апрель 2026)

Честная оценка после нескольких недель в production: **Claude Design сырой**. Это качественные наблюдения ранних пользователей — у вас на конкретном продукте опыт может отличаться. Типичные проблемы: генерирует «средний» UI без характера, искажает дизайн-систему в нетривиальных кейсах, handoff в код требует переписывания значительной части.

**Где Claude Design ОК:** одноразовые презентации, ранние мокапы в обсуждении, лендинги-однодневки.

**Где не использовать в production:** реальные продуктовые экраны, работа с дизайн-системой, дизайн, имеющий значение для UX.

Ситуация может измениться через 3–6 месяцев. Сейчас Figma остаётся единственным серьёзным инструментом дизайна.

### 7.2. Figma как источник правды для UI

**Figma Professional** ($15–25/мес за Dev seat). Бесплатный план не позволяет нормально работать с MCP-сервером (лимит 6 вызовов в месяц).

**Что в Figma:** дизайн-система с компонентами и токенами, финальные макеты нетривиальных экранов, prototyping для юзер-тестов.

**Что НЕ в Figma:** простой UI (формы из 3 полей), решения «почему» (это в vault), компоненты дизайн-системы как первоисточник (они в коде, см. 7.3).

### 7.3. Структура Figma-файлов и дизайн-система

Один проект в Figma на продукт. Внутри:

```
📁 Product Name
├── 📄 Design System         ← компоненты, токены
├── 📄 [Feature] User Auth   ← по странице на крупную фичу
├── 📄 [Feature] Billing
└── 📄 Archive               ← старые итерации
```

В каждом файле фичи — section'ы `Current`, `In progress`, `Explorations`. Смешение в одном фрейме = главный источник «дизайнер сделал, разработчик сделал не то».

Дизайн-система живёт **в двух местах одновременно**, и они синхронизируются вручную, но согласованно.

**В Figma:** component library с вариантами, variables для токенов.
**В коде:** `shared/design-system/` с React-компонентами, Tailwind config или CSS variables с теми же токенами.

**Правила синхронизации:**

- Новый компонент сначала в Figma, потом в код.
- Изменение в Figma без изменения кода → дизайнер открывает PR или создаёт задачу.
- Раз в месяц — sync-встреча 30 минут: «всё ли совпадает».

Детальная организация Figma-файлов, Tokens Studio pipeline, handoff workflow — если вам это активно нужно, фиксируйте в `active/patterns/design-system.md` под вашу команду. Общий рецепт здесь бесполезен, варьируется сильно.

### 7.4. Figma MCP в Claude Code

Интеграция, которая позволяет Claude Code читать Figma-файлы напрямую.

**Установка:**

1. В Figma: Settings → Enable Dev Mode MCP Server.
2. В Claude Code: `/mcp` → figma → OAuth.

**Использование:** дизайнер выделяет фрейм → Copy link to selection. Разработчик вставляет в Claude Code: «Реализуй этот экран: [ссылка]». Claude через MCP читает фрейм, генерирует код.

**Лимиты:**

- Бесплатный Figma: 6 вызовов в месяц (недостаточно).
- Платный Dev seat: 200 вызовов в день.
- Контекст на ответ: 25k токенов — работайте по фреймам, не по целым страницам.
- Точность ~80%: spacing и компоненты обычно верные, edge cases (анимации, состояния) часто пропускаются.

### 7.5. Когда дизайнер пропускается

Не каждый UI-элемент требует дизайнера. Прагматично для команды 2–8:

**Без дизайнера, разработчик сам:** внутренние админ-страницы, простые формы на готовых компонентах, системные страницы (404, 500), transactional email.

**С дизайнером обязательно:** любой экран конечного пользователя, не сделанный из готовых компонентов; изменения в дизайн-системе; лендинги; critical flows.

Серая зона — обсуждается на еженедельной встрече.

## 8. Инфраструктура и деплой

### 8.1. Философия

**Платите за managed services, пока можете себе позволить.** Самостоятельный Kubernetes-кластер на старте — это найм ещё одного человека, которого у вас нет.

### 8.2. Стартовый стек

Типичный набор для веб-приложения:

- **Хостинг frontend** — Vercel, Cloudflare Pages или Netlify
- **Хостинг backend** — Railway, Fly.io или Render
- **Database** — Supabase, Neon, PlanetScale
- **Error tracking** — Sentry
- **Analytics** — PostHog (product), Plausible/Fathom (web)
- **CI/CD** — GitHub Actions
- **Secrets** — Doppler, 1Password Secrets Automation или аналог
- **Monitoring** — Better Stack, Axiom или встроенные возможности хостинга

Реалистичная стоимость на ранней стадии: $150–600/мес в зависимости от выбранных планов и нагрузки. Разные сервисы масштабируются по-разному (managed БД обычно делают резкие скачки между тирами, хостинг frontend растёт медленнее).

### 8.3. Когда думать о Kubernetes

Практически никогда, пока не появится:

- Отдельный DevOps-инженер
- 5+ сервисов с разными требованиями
- Нагрузка, которую managed-платформы не тянут
- Требование self-hosted по compliance

Если да на всё выше — managed Kubernetes (GKE, EKS, AKS), не self-hosted.

### 8.4. Environments

- **dev** — локально, docker-compose
- **preview** — на каждый PR, изолированная БД через branching
- **staging** — опционально, копия production для ручного QA
- **production** — деплой только из main

Preview environments — огромный выигрыш для маленькой команды. Дизайнер и PM видят фичу до merge.

### 8.5. Pipeline на PR

Реалистичный минимум:

1. Lint + type check
2. Unit tests
3. Build
4. Deploy preview
5. (Опционально) E2E против preview

Полный pipeline из 9+ шагов — красиво на бумаге, но в маленькой команде часто становится flaky, блокирует merge и в итоге отключается. Добавляйте шаги по мере роста, когда действительно ловят реальные баги.

### 8.6. Observability

Минимум:

- Структурированные логи (JSON) с trace ID
- Error tracking (Sentry)
- Uptime monitoring (Better Stack, UptimeRobot)
- Базовые метрики из хостинг-платформы

Tracing через OpenTelemetry — полезно, но требует времени на настройку. Не первый месяц.

### 8.7. Runbooks

`infra/runbooks/` — страницы «что делать, когда сломалось X». Пишутся после первого incident'а соответствующего типа, не до.

AI-агенты полезны для **диагностики** во время incident'а — могут быстро пройтись по runbook, проверить логи, предложить гипотезы. Но **действуют в проде только люди**. Автономный incident response через AI — это не про 2026 год.

### 8.8. Secrets

Железные правила: никогда в git, никогда в чатах, разные на dev/staging/prod, rotation для production (частоту определяете сами), access log для production secrets.

Реальность маленькой команды: rotation часто пропускается. Лучше «раз в 6 месяцев и реально делать», чем «раз в 30 дней и не делать никогда».

## 9. Тестирование

### 9.1. Стратегия

Не существует одной правильной пирамиды тестов для всех команд. Выбор зависит от типа системы:

- UI-heavy с простой логикой — unit-тесты на утилитах и hooks, e2e на критических флоу
- API-heavy со сложной логикой — много integration, меньше unit
- Микросервисы — contract tests между сервисами

Ваша стратегия — это ADR. Обсудите, зафиксируйте, пересматривайте раз в полгода.

### 9.2. Что покрывать всегда

Независимо от стратегии:

- **Критические user journeys** — 5–10 штук в e2e (регистрация, оплата, критический happy path)
- **Бизнес-логика с деньгами или безопасностью** — покрытие 80–90%+
- **Сложные алгоритмы** — unit-тесты

Не покрывать: простой CRUD UI без логики, тривиальные getters/setters, mocks на mocks.

### 9.3. E2E: реальность vs идеал

Playwright и аналоги — мощные, но flaky-тесты требуют времени на стабилизацию. Визуальная регрессия часто даёт false positives.

Честная рекомендация: начинайте с 2–3 e2e на самые важные флоу. Растите по мере того, как ловите реальные баги. Не покрывайте всё подряд.

### 9.4. QA без QA-инженера

В маленькой команде тестирование — общая ответственность. Разработчик пишет тесты на свою фичу. Дизайнер ревьюит preview environment. PM проверяет функциональность перед promote в production.

Когда нанимать отдельного QA: стабильно >1 значимый баг в неделю в production, релизы задерживаются из-за ручного тестирования.

### 9.5. Bug tracking

Баги — в Linear или GitHub Issues. Но **их причины и prevention** — в vault. Значимый баг → короткая страница в `log/incidents/` или строчка в `active/patterns/gotchas.md` со структурой: что сломалось / первопричина / что изменили в паттерне.

## 10. Ритуалы

### 10.1. Минимум, который реально работает

Список намеренно короткий. Больше ритуалов = больше времени в синхронных встречах = меньше реальной работы.

**Ежедневно (async, 2 минуты на человека):** standup в Slack/Discord — что сделал, что делаю, блокеры.

**Еженедельно (sync, 45–60 минут):** одна встреча всей командой — ретроспектива прошлой недели + планирование следующей. Сюда включается и sprint-ритуал, если используете спринты.

**Ежемесячно (sync, 1 час):** финансы + продуктовые метрики вместе — где мы по сравнению с месяцем назад.

**Ежеквартально (sync, полдня):** strategy offsite — пересмотр `stable/`, roadmap, North Star.

Если используете knowledge-модуль, добавьте раз в 1-2 недели **ingest-сессию** (20–40 минут): разработчик запускает Claude Code на новых raw-источниках.

### 10.2. Чего избегать

Ритуалы, которые красиво звучат и умирают через месяц:

- Ежедневный синхронный standup (для команды 4 человека — избыточно)
- Пятничный review vault в graph view (никто не смотрит на graph view дольше 30 секунд)
- Отдельные security/compliance/data review ежеквартально (в маленькой команде это в одном offsite)
- Bug bash по расписанию (через 6 недель переносится, через 10 удаляется)

### 10.3. AI-автоматизации: честно

В теории Cowork может по понедельникам читать vault, делать digest, флагать аномалии. На практике это нужно написать и отладить, API интеграции требуют поддержки, за 3 месяца без внимания автоматизация часто ломается.

Если у вас есть человеко-ресурс на построение и поддержку — отлично, эти автоматизации окупаются. Если нет — делайте вручную. Не включайте «автоматический weekly digest» в план внедрения как данность.

Исключение — knowledge-модуль, где ingest через LLM **и есть** основная автоматизация, и она работает в диалоговом режиме, не в фоне. Это единственная автоматизация, которую рекомендую.

## 11. Анти-паттерны

Конкретные способы, которыми команды ломают этот фреймворк.

### 11.1. Vault как write-only

Самый частый failure mode. Документы пишутся, но **никто не читает**. Через 3 месяца vault — кладбище.

Признаки: в PR никогда не цитируются ADR, новые члены команды не упоминают vault в первую неделю, в обсуждениях «а где это записано?» — «не помню».

Противоядие: ссылки на vault-страницы в каждом значимом PR, регулярное «а что в vault про это?» в обсуждениях, onboarding через vault — не через устные объяснения.

### 11.2. Один писатель

Vault поддерживает только один человек (обычно founder или tech lead). Когда он в отпуске или уходит — всё замирает.

Противоядие: в первые 3 месяца каждый член команды должен написать минимум одну страницу в vault. Без этого привычка не сформируется.

### 11.3. AI генерирует, никто не проверяет

Claude Code пишет ADR по запросу, никто не ревьюит. Через полгода половина ADR содержит hallucinated обоснования.

Противоядие: ADR идут через code review как любой другой PR. AI-генерация — драфт, не финал. Для knowledge-модуля правило ещё строже — см. соответствующий документ.

### 11.4. Ritual theater

Все формально ходят на встречи, пишут digest'ы, делают ретроспективы. Но ничего не меняется от этого.

Противоядие: каждая встреча заканчивается списком next actions с owner'ами. Следующая встреча начинается с проверки этого списка.

### 11.5. Vault как Notion-replacement

В vault появляются задачи, календари, kanban-доски. Это превращает его в операционный chaos.

Противоядие: жёсткое разделение. Задачи — в Linear/GitHub Issues. Знания — в vault. Ссылки между ними, но не дубли.

### 11.6. Дорогой старт

Команда 3 человека, но Kubernetes, микросервисы, Kafka, data warehouse. Complexity budget исчерпан до первой строчки бизнес-логики.

Противоядие: начинайте с монолита на managed services. Разбивайте и усложняйте по факту боли.

### 11.7. ADR inflation

40 ADR за первый квартал, половина — про выбор HTTP-клиента.

Противоядие: эвристика «если можно откатить за день — без ADR».

### 11.8. Идеальный vault в первый месяц

Команда 2 недели настраивает структуру vault до написания кода.

Противоядие: начинайте с 3 папок. Расширяйтесь по факту.

### 11.9. Knowledge без ingest

Настроили knowledge-модуль, кладёте raw-источники, но ingest не запускаете. Wiki stale, команда думает «у нас есть база знаний», но на деле её нет.

Противоядие: ingest-сессия — ритуал, не «сделаем когда будет время». См. `knowledge.md` раздел 5.1.

### 11.10. Security/QA/аналитика «потом»

«Сначала PMF, потом всё это». В результате первый же incident или breach становится catastrophic.

Противоядие: минимум с первого дня — secrets в менеджере, error tracking, event taxonomy, privacy policy. Это 1–2 недели работы, не 1 день (как часто обещают), но и не 3 месяца. Детали — в `non-engineering.md`.

## 12. Поэтапное внедрение

Реалистичные сроки для команды 2–8, стартующей с нуля. Это **оценки сверху**: в реальности пойдёт медленнее.

### Месяц 1: Foundation

Цель: минимальная инфраструктура и первая фича в полном workflow.

- Vault-репозиторий, 3 папки (stable, active, log), README, COMMON.md
- Первые 2–3 ADR: стек, базовая архитектура, monorepo vs polyrepo
- Инфра: GitHub, хостинг frontend, хостинг backend, БД, error tracking, secrets manager
- Первая фича в `active/features/`, прошедшая через полный workflow (глава 4)
- CI pipeline на PR: lint, type check, tests, preview

Что реально произойдёт:

- Настройка сервисов займёт больше, чем запланировано (первый unexpected quirk съест день).
- Первый ADR по стеку будет черновиком — реальное понимание придёт через 2–3 недели.
- Первая фича покажет несколько дыр в vault, которые нужно заполнить.

**Чего НЕ делаете в месяц 1:** не подключаете knowledge-модуль, даже если кажется «полезно заранее». Модуль окупается на 10+ raw-источниках, которых у вас в первый месяц почти наверняка ещё нет.

### Месяцы 2–3: Stabilization

Цель: ритуалы и паттерны становятся привычкой.

- Дизайн-система с 5–10 компонентами в коде
- `active/patterns/` с 3–5 написанными паттернами
- Еженедельный ритуал команды стабилен
- Первый incident с runbook'ом и постмортемом
- Event taxonomy в `stable/events.md`
- Privacy policy, если нужна (см. non-engineering)

Что реально произойдёт:

- Возникнут споры «это в vault или в Linear» — решайте по месту.
- Будет соблазн расширять структуру vault — сопротивляйтесь.
- Кто-то в команде начнёт игнорировать COMMON.md — обсудите.

Если команда делает discovery-работу (интервью, research) — на 2–3 месяце подключайте knowledge-модуль. До 10 raw-источников он не нужен, но когда их становится 10+, модуль даёт значимый эффект.

### Месяц 4–6: Compounding

Цель: накопленный контекст начинает работать на команду.

- ADR-база (10–20 штук) реально используется при обсуждениях
- Новый человек в команде онбордится за 1–2 недели
- Claude Code в сессиях ведёт себя в стиле команды без постоянных напоминаний
- Паттерны используются повторно в новых фичах
- Velocity команды выше, чем была на 1–2 месяце

Это точка, где фреймворк окупается. Если в этот момент команда сопротивляется vault — пересмотрите подход или откажитесь от фреймворка. Не подходит всем.

### После 6 месяцев

Расширение по факту потребности:

- Появились enterprise-клиенты → компоненты из non-engineering (compliance, security review)
- > 10K пользователей → появляется Data-роль и отдельный security review
- > 8 человек → формальные спринты, отдельный QA, DevOps
- Несколько сервисов мешают друг другу → разделение на микросервисы

Vault растёт органически, не по плану.

## Финальное

Этот фреймворк — опорный каркас, не истина. Он окупается, если у команды есть дисциплина поддерживать vault и уважение к накопленному контексту. Без этого — это просто ещё одна папка с устаревшими документами.

Главное, что фреймворк даёт — это **операционный язык общения людей с AI-агентами**. Один раз настроив vault, CLAUDE.md, паттерны и ADR, вы перестаёте каждый раз объяснять Claude Code, как ваша команда принимает решения. Он читает и делает в контексте.

Второе по важности — **переживаемость ухода людей**. В маленькой команде с одним экспертом в области его уход — катастрофа. С vault, где решения записаны — серьёзная боль, но не катастрофа.

Третье — **ускорение onboarding**. Новый человек за 1–2 недели становится продуктивным, потому что контекст команды читается, а не пересказывается в устных разговорах.

Всё остальное — следствия.

---

**Вопросы, сомнения, контрпримеры** — их и должно быть много. Фреймворк адаптируется, а не копируется. Если после месяца использования вы обнаружили, что какая-то часть не работает для вашей команды — это не ошибка, это сигнал.
