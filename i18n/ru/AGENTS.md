# AGENTS.md

Правила для AI-агентов (Claude Code, Codex, Cursor и других), работающих над этим репозиторием.

## Что это за репо

Shubin Framework — documentation-only репозиторий, описывающий конвенции для использования markdown-репы плюс GitHub Issues как общего командного контекста.

Здесь нет кода, нет билда, нет тестов. «Код» — это markdown. «Билд» — рендеринг на GitHub.

Следствие: большинство AI-агент-навыков в этом репо — про *редактирование прозы*, а не имплементацию фич. Авторский голос — прямой и честный про caveats; сохраняйте его.

## Что вы читаете первым

1. `README.md` — обзор проекта и reading router.
2. `ARCHITECTURE.md` — layout репозитория и инварианты.
3. `framework.md` — основная спецификация.
4. `decisions/` — принятые решения, особенно недавние (этой директории пока не существует; когда появится — читайте).

До предложения любой структурной правки — прочитайте секцию Инварианты в `ARCHITECTURE.md`. Многие «улучшения» их нарушают.

## Что вы делаете

- Предлагаете правки в spec-документы (`framework.md`, `lite.md`, `knowledge.md`, `projects.md`, `non-tech.md`, `non-engineering.md`), когда вас просят.
- Помогаете написать новый ADR, когда пользователь описывает решение, достойное записи. Используйте формат из `framework.md` главы 4.4.
- Обновляете cross-references, когда файлы переезжают, главы перенумеровываются или секция переименовывается.
- Держите русский перевод в `i18n/ru/` в уме: изменение в root-spec создаёт дрейф. Обновляйте `i18n/ru/` в том же PR или фиксируйте дрейф в описании PR.

## Что вы не делаете

- Не добавляйте директорию `features/`. Фичи живут в GitHub Issues. Это один из самых сильных инвариантов в `ARCHITECTURE.md`.
- Не добавляйте директорию `knowledge/` в *этот* репозиторий. Фреймворк предписывает `knowledge/` для adopting team; мы сознательно не применяем фреймворк к собственной мета-работе. Пример AGENTS.md для knowledge-модуля живёт в `templates/knowledge/AGENTS.md`.
- Не добавляйте stateful frontmatter (status, owner, priority, due date) ни в один документ. State живёт в GitHub.
- Не реорганизуйте top-level layout без явной инструкции.
- Не переводите на другие языки, пока не попросят. Русский живёт в `i18n/ru/`; другие языки добавили бы `i18n/<lang>/` плюс `README.<lang>.md` в корне, но только по запросу.
- Не добавляйте emoji в канонические (UPPER_CASE) документы. Lower-case spec-документы могут содержать emoji только внутри цитируемых внешних источников.
- Не пушьте в `main` во время миграции v2. Вся работа идёт на ветке `v2` с префиксом `v2:`; см. `conventions/GIT_CONVENTION.md`.

## Стиль ADR

При написании ADR используйте формат из `framework.md` главы 4.4:

- `# ADR-NNNN: <короткий заголовок решения>`
- `## Status` — `Accepted — YYYY-MM-DD`, или `Superseded by ADR-NNNN`
- `## Context` — ситуация и ограничения
- `## Options considered` — pros и cons по каждому, включая отвергнутые
- `## Decision` — выбранный вариант и предложение о том, почему
- `## Consequences` — что мы теряем, что выигрываем, что становится тяжелее
- `## Related` — ссылки на другие ADR и на GitHub Issue, если есть

ADR иммутабельны после acceptance. Чтобы изменить — пишите новый ADR со статусом `Supersedes ADR-NNNN` и обновляйте старый ADR на `Superseded by ADR-NNNN`.

## Markdown-стиль

- Английский в root-spec-документах; русский только в `i18n/ru/`.
- Проза вместо буллет-листов, где разумно. Буллеты — для настоящих перечислений, не для разрыва предложений по строкам.
- Бэктики для имён файлов, путей и идентификаторов.
- Никаких emoji в канонических документах.
- Соблюдайте авторский голос: прямой, готовый назвать, где практика проваливается, честный про caveats. Избегайте корпоративного hedging'а и AI-slop'а («furthermore, it is important to note that...»).

## Шаблоны

Когда фреймворк предписывает конкретный файл, который копирует adopting team, каноническая версия живёт в `templates/`:

- `templates/.github/ISSUE_TEMPLATE/` — Issue Templates для кодовых репо.
- `templates/knowledge/AGENTS.md` — auto-ingest spec для опционального knowledge-модуля.

Если `framework.md` вводит новый copy-target, добавляйте под `templates/`, а не вшивайте полный файл в спеку; спека ссылается на шаблон.

## Когда сомневаетесь

Спрашивайте до деструктивных операций: удаления файлов, переноса между директориями, крупных переписываний прозы. Когда не уверены в конвенции — проверяйте Инварианты в `ARCHITECTURE.md` сначала; если ответа там нет — спросите пользователя, не угадывайте.
