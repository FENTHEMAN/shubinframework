# Shubin Framework: репозиторий для нетехнических членов команды

**Полное название:** Гайд по работе с репозиторием и git без терминала
**Часть:** Shubin Framework (companion-документ)
**Версия:** 3.0

**Для кого:** PM, дизайнер, маркетолог, фаундер, доменный эксперт — любой член команды, которому приходится работать с командным markdown-репозиторием, но кто никогда не открывал терминал.

**Время чтения:** 30 минут.
**Время на первичную настройку:** 45 минут.

**Что НЕ нужно знать перед чтением:**
- Программирование
- Командную строку
- Что такое git, до сегодняшнего дня

**Что нужно знать про контекст:**
Ваша команда использует общий markdown-репозиторий, в котором живёт командное знание — архитектура, решения, глоссарий, журналы, ресёрч. Он синхронизируется через git — систему, которая хранит историю всех изменений и позволяет работать вместе без конфликтов.

Вы вносите изменения в этот репозиторий — добавляете термин в глоссарий, логируете интервью, кладёте источник в ресёрч, фиксируете feedback. Сами фичи в v2 в этом репозитории не живут: они — GitHub Issues, открываются в браузере. Этот документ учит работе с репозиторием без боли; работа с Issues описана в секции 6.

**Связанные документы:**
- `framework.md` — полный фреймворк для тех-лида команды. Опциональное чтение.
- `knowledge.md` — LLM-wiki модуль. Если ваша команда его использует — понадобится секция 9.

## 1. Что такое git и зачем он нужен

Очень коротко:

**Git** — система, которая хранит историю изменений ваших файлов. Каждое изменение — это «commit» с автором, датой и описанием. Вы можете откатиться к любой предыдущей версии. Можете работать одновременно с другими людьми без конфликтов.

**GitHub / GitLab / Bitbucket** — сайты, на которых хранится копия вашего git-репозитория, доступная всей команде.

**Repository (repo)** — папка проекта, отслеживаемая git'ом. У вашей команды, скорее всего, их несколько: markdown-репозиторий (где живёт знание), фронтенд, бэкенд и т. д.

Чтобы работать с markdown-репозиторием, нужно понимать четыре операции:
- **clone** — скачать репозиторий с GitHub на компьютер (один раз на старте)
- **pull** — обновить локальную копию с GitHub (получить чужие изменения)
- **commit** — сохранить свои изменения локально с описанием
- **push** — отправить свои коммиты на GitHub (чтобы остальные увидели)

Эти четыре операции покрывают 95% работы.

## 2. Инструмент: GitHub Desktop

Разработчики работают с git из командной строки. Вам это не нужно — используйте GUI-клиент.

**Рекомендуемые инструменты:**

**GitHub Desktop** ([desktop.github.com](https://desktop.github.com)) — бесплатный, простой интерфейс, идеально для старта. Работает с GitHub из коробки. Все примеры в этом документе на нём.

**Obsidian с git-плагином** — если уже редактируете репозиторий в Obsidian, плагин «Obsidian Git» автоматизирует pull/commit/push в фоне. Минус: меньше контроля, иногда коммитит мусор. Подходит тем, кто хочет вообще забыть о git.

**Начните с GitHub Desktop.** Через месяц захотите чего-то другого — попробуете.

## 3. Первичная настройка (один раз, 15 минут)

### 3.1. Что спросить у команды в первый день

Перед стартом спросите у разработчика или тех-лида:
- URL репозитория на GitHub
- Доступ к GitHub-организации (вас должны добавить как участника)
- Имя, под которым вас добавят в `TEAM.md`
- Email, который будете использовать для коммитов (обычно рабочий)
- 10-минутный тур по структуре: что где живёт, что можно редактировать, что трогать не стоит

Это стандартный онбординг — разработчик его уже делал для других. Не стесняйтесь.

### 3.2. Шаги настройки

**Шаг 1.** Установите GitHub Desktop с [desktop.github.com](https://desktop.github.com).

**Шаг 2.** Запустите. Войдите в свой GitHub-аккаунт. Если его нет — зарегистрируйтесь, потом попросите добавить вас в организацию.

**Шаг 3.** Установите markdown-редактор:
- **Obsidian** ([obsidian.md](https://obsidian.md)) — бесплатный, специально для vault'ов, граф ссылок, удобный поиск. Рекомендация по умолчанию.
- **VS Code** ([code.visualstudio.com](https://code.visualstudio.com)) — если интересно посмотреть командный код.
- Любой markdown-редактор из app store вашей ОС — если не хочется ставить новое.

**Шаг 4.** В GitHub Desktop: File → Clone Repository → URL. Вставьте URL репозитория. Выберите папку на компьютере, например `~/work/<repo-name>/`. Нажмите Clone.

За 30 секунд копия репозитория появится у вас. Откройте в Obsidian (или другом редакторе) как vault — это термин Obsidian для открытой папки, к фреймворку отношения не имеет.

**Готово.** У вас рабочая копия командного знания на компьютере.

## 4. Ежедневный рабочий цикл

Каждый раз, садясь работать с репозиторием, повторяйте одну и ту же последовательность из четырёх шагов.

**Шаг 1: Pull (получить свежее, 5 секунд).**

Откройте GitHub Desktop. Если видите кнопку «Pull origin» с числом коммитов — нажмите. Это скачает изменения, сделанные другими.

Делайте **безусловно перед началом работы**. Пропустите — рискуете создать merge-конфликт.

**Шаг 2: Работа в редакторе.**

Откройте нужные файлы в Obsidian, редактируйте, создавайте новые. Тут с git не взаимодействуете — он сам замечает все изменения.

**Шаг 3: Commit (сохранить изменения, 30 секунд).**

Когда сделали что-то осмысленное (закончили мысль, добавили решение, обновили статус) — переключитесь в GitHub Desktop. Слева увидите список изменённых файлов.

В нижнем левом углу — поля «Summary» (короткое описание) и «Description» (опционально, длиннее).

Хорошие примеры summary:
- `add ADR-0024: switch from REST to tRPC`
- `glossary: add 'product-led growth' term`
- `add new pattern: error handling on server`
- `feedback: 3 user interviews summary`

Плохие примеры:
- `update`
- `changes`
- `file update`

Заполнили — нажмите «Commit to main».

**Шаг 4: Push (отправить на GitHub, 5 секунд).**

После commit'а сверху появится кнопка «Push origin». Нажмите — ваши изменения улетят на GitHub, остальные смогут забрать через свой Pull.

**Цикл закончен.**

## 5. Типичные ситуации и что с ними делать

**Ситуация 1: «После commit ничего не происходит».**

Скорее всего, забыли push. Проверьте кнопку «Push origin» сверху.

**Ситуация 2: «Pull показывает число, но я ещё работаю над своим».**

Сначала закоммитьте своё (шаг 3 в цикле выше), потом pull. Pull может попросить разрешить «merge» — обычно работает кнопка по умолчанию.

**Ситуация 3: Конфликт (merge conflict).**

Случается, когда вы и кто-то ещё одновременно изменили одно и то же место в одном файле. GitHub Desktop покажет конфликтный файл и пометит конфликтные строки `<<<<<<<` и `>>>>>>>`.

**Что делать:**
- Откройте файл в редакторе
- Найдите блок:
  ```
  <<<<<<< HEAD
  ваша версия
  =======
  чужая версия
  >>>>>>> branch-name
  ```
- Решите, что должно быть (часто — обе версии вместе), уберите маркеры
- Сохраните файл
- Вернитесь в GitHub Desktop, сделайте commit с сообщением `resolve merge conflict`
- Push

Если запутались — позовите разработчика. 5 минут работы вдвоём.

**Ситуация 4: «Случайно удалил файл / сломал что-то важное».**

Git помнит всё. В GitHub Desktop: вкладка History → найдите коммит, в котором файл ещё был ок → правый клик → Revert. Это создаст новый коммит, отменяющий проблемные изменения.

Боитесь — позовите разработчика, ничего не трогайте. Файл точно вернётся.

**Ситуация 5: «GitHub Desktop хочет, чтобы я сделал branch».**

Branch — параллельная версия файлов. Для **большинства работы с репозиторием это не нужно** — пишите прямо в main.

Branch нужен, только если:
- Вы готовите большой черновик, который пока не должен быть виден
- Работаете с разработчиком над PR, требующим ревью

В обоих случаях — попросите разработчика помочь сделать branch. Один раз сделаете — поймёте.

## 6. Фичи живут в GitHub Issues, не в репозитории

Это главный сдвиг v2. Фичи — что строится, кто owner, какой scope, что считается успехом — живут в **GitHub Issues**, а не как markdown-файлы в этом репозитории.

Когда у вас идея фичи, вы не создаёте файл. Вы заходите в кодовый репо команды на GitHub.com, жмёте «New Issue», выбираете шаблон фичи и заполняете: Why, Scope, Success criteria. Этот Issue *и есть* спека, трекер и обсуждение. Никаких branch, commit, PR с вашей стороны. Хватает браузера; мобильное приложение GitHub тоже работает.

Конкретно это значит:
- Новая идея → открыть Issue.
- Обсуждение фичи → в комментариях Issue.
- Фича уехала → разработчик закроет Issue при мерже PR. Вы нигде ничего не помечаете.
- Ищете старую фичу → ищите Issues с фильтром closed; это и есть исторический след.

Репозиторий из этого документа — для **устойчивого знания**: ADR, глоссарий, архитектура, журнал, ресёрч. GitHub Issues — для **живого workflow**. Не пытайтесь зеркалить состояние Issue в markdown — копия начнёт расходиться с истиной за неделю.

## 7. Что можно и нельзя в репозитории

Эта секция — про репозиторий на вашем компьютере. По фичам см. секцию 6 — они живут не здесь.

**Можно (и нужно) свободно:**
- Создавать новые файлы в `journal/feedback/` — это append-only папка, туда пишут все
- Создавать новые файлы в `journal/retros/`
- Добавлять новые термины в `GLOSSARY.md`

**Спросите разработчика, прежде чем менять:**
- ADR (`decisions/`) — это технические решения, незваные правки тут странно смотрятся
- Технические паттерны (`patterns/`) — то же
- `AGENTS.md` — это инструкция для AI, требует осторожности
- Структура папок (создание новых top-level папок)
- `STACK.md`, `TEAM.md` — декларативные файлы

**Не трогать:**
- Файлы с расширениями `.gitignore`, `.gitattributes` — это техническая обвязка
- Папку `.git/` (она и так скрыта, но на всякий случай)

### 7.1. Если команда использует knowledge-модуль

Если ваша команда подключила knowledge-модуль (папка `knowledge/` в репозитории), у него отдельные правила. См. секцию 9.

## 8. Альтернатива: редактирование через GitHub web

Если GitHub Desktop пугает, есть путь проще: редактирование прямо на github.com.

**Как:**
1. Откройте репозиторий на GitHub.com
2. Кликните на нужный файл
3. Иконка карандаша (Edit) в правом верхнем углу
4. Редактируйте прямо в браузере
5. Внизу — поле «Commit changes», заполните и сохраните

Минусы:
- Нет фич Obsidian (поиск, граф, backlinks)
- Неудобно для длинных правок
- Нельзя массово создавать файлы

Но для одноразовых правок вроде добавления термина в глоссарий или фикса опечатки работает отлично. Особенно с мобилки. (Для фич это вообще не нужно — см. секцию 6.)

### 8.1. Когда репозиторий на телефоне

Иногда нужно зафиксировать мысль или фидбек, когда нет компьютера.

**Варианты:**
- **Obsidian Mobile** + плагин Obsidian Git — синхронизирует в фоне (требует немного настройки от разработчика)
- **GitHub Mobile app** — можно открыть репо, найти файл, отредактировать прямо в приложении
- **Просто пишите в заметках телефона**, утром перенесите в репозиторий

Не превращайте репозиторий в телефонный таск-лист. Серьёзная работа — за компьютером.

## 9. Как работать с knowledge-модулем

*Эта секция актуальна, только если ваша команда подключила knowledge-модуль (папка `knowledge/` в репозитории). Если нет — можно пропустить.*

Knowledge-модуль — место, где команда накапливает **внешние знания**: интервью пользователей, статьи о конкурентах, research-отчёты, транскрипты звонков. AI (Claude или Cowork) автоматически компилирует их в структурированные wiki-страницы со ссылками между собой.

**Почему это важно вам:**
- Вы скорее всего один из главных людей, кладущих материалы в `knowledge/raw/`
- Сможете задавать вопросы вроде «что мы знаем про enterprise-пользователей?» и получать синтез по всем накопленным источникам
- Но есть строгие правила, которые нельзя нарушать, иначе вся система разваливается

### 9.1. Как класть источники в raw/

Вы работаете в подпапке `knowledge/raw/`. Структура:

```
knowledge/raw/
├── user-interviews/
├── competitors/
├── industry/
└── calls/
```

Кладите новый файл в нужную папку. Имя файла: `YYYY-MM-DD-краткое-описание.md`. Например:
- `knowledge/raw/user-interviews/2026-04-18-acme-corp-discovery.md`
- `knowledge/raw/competitors/2026-04-19-linear-new-pricing.md`

Первые строки файла — frontmatter:

```markdown
---
type: user-interview
source: Acme Corp
date: 2026-04-18
author: @ваше-имя
---

(дальше содержимое — транскрипт, текст статьи, заметки)
```

Commit с сообщением `raw: <что это>` — например, `raw: Acme Corp discovery interview`.

Через какое-то время разработчик (или вы через Cowork) запустит ingest-сессию, и Claude добавит этот материал в wiki/.

### 9.2. Два железных правила

**Правило 1: `knowledge/raw/` иммутабельно.** Положив файл, **вы не редактируете его содержимое**. Нашли ошибку? Добавьте новый файл с правкой (`2026-04-18-acme-corp-discovery-correction.md`). Старый не трогайте.

Это важно, потому что Claude помнит, что и из какого источника он скомпилировал. Если вы тихо измените источник, wiki начинает расходиться с реальностью.

**Правило 2: `knowledge/wiki/` руками не трогать.** Это зона Claude. Видите неправильное в `wiki/personas/small-business-owner.md`? Не редактируйте напрямую. Вместо этого:
- Либо добавьте правку в raw/ и попросите разработчика сделать ingest
- Либо попросите Claude через Cowork: «на странице X утверждение Y неверно, перепроверь источники»

Прямое редактирование wiki — Claude перезапишет ваши правки в следующий раз, и вы будете воевать с системой.

### 9.3. Как задавать вопросы wiki

Через Cowork или Claude Code (попросите разработчика показать, как подключить Cowork к репозиторию — это 5 минут настройки).

Примеры рабочих вопросов:
- «Что мы знаем про enterprise-пользователей? Краткое описание и цитаты.»
- «Какие конкуренты упоминаются в наших интервью? Сравни их по pricing.»
- «Есть ли противоречия в том, что пользователи говорят про onboarding?»

Claude идёт в `wiki/`, возвращает ответ со ссылками на raw-источники. Если wiki-страницы ещё нет, но в raw есть релевантное — Claude обычно предложит создать страницу.

### 9.4. Что делать, когда Claude что-то выдумал

LLM иногда галлюцинируют — атрибутируют цитату не тому интервью, делают обобщения, которых нет в источниках.

Если читаете wiki-страницу и видите подозрительное утверждение:
1. Кликните по ссылке `[[raw/...]]` — откроется источник
2. Проверьте, есть ли утверждение там
3. Если нет — скажите Claude: «В wiki/personas/X говорится Y со ссылкой на raw/Z. Я проверил Z — там этого нет. Обнови страницу».

Claude поправит. Это нормальный цикл — не паникуйте.

### 9.5. Минимальный ритуал

Раз в неделю (или чаще, если много новых материалов) — ingest-сессия. Обычно её запускает разработчик или тех-лид. С вашей стороны достаточно:
- В течение недели класть новые raw/ файлы
- В день ingest'а проверить отчёт Claude («обновлено X, создано Y, найдено Z противоречий»)
- Если что-то в отчёте смущает — спросить

Полная документация по knowledge-модулю — в `knowledge.md`. Для вас как контрибьютора в raw/ описанного выше достаточно.

## 10. Анти-паттерны для нетехнических

**1. «Я постоянно забываю pull, ничего не работает».**
Превратите в привычку: открыли GitHub Desktop → первое действие — посмотреть на «Pull origin».

**2. «Накопил 30 изменений и закоммитил всё одним 'update repo'».**
Маленькие частые коммиты — хорошо. Один большой без описания — плохо. Через месяц никто (включая вас) не вспомнит, что там было.

**3. «Я что-то поменял в коде разработчика, потому что 'ну это же просто markdown'».**
Общий репозиторий — да, ваша зона. Код-репозитории — нет. Если в README код-репозитория надо что-то поправить — попросите разработчика.

**4. «Боюсь нажать что-то не то и сломать всё».**
В git практически невозможно сломать что-то необратимо. Любое действие можно отменить. Спрашивайте, не сидите в страхе.

**5. «Не коммичу, пока не 'закончу'».**
Лучше сохранить половину работы, чем потерять её при сбое. Промежуточные коммиты — норма.

**6. «Редактирую wiki/ напрямую, потому что там ошибка» (если используется knowledge-модуль).**
См. 9.2. Никогда не редактируйте `knowledge/wiki/` напрямую — исправляйте через источники или просите Claude.

## 11. Реалистичные ожидания

**Первая неделя:** будете спрашивать у разработчика 2–3 раза в день. Это нормально.

**Через месяц:** комфортно делаете цикл pull → edit → commit → push без помощи. Конфликты, если попадаются, решаете сами.

**Через квартал:** не думаете о git вообще, он становится прозрачным фоном.

**Что точно НЕ нужно учить:** branches/merging без необходимости, rebase, cherry-pick, submodules, hooks и прочую «настоящую» git-обвязку. Это для разработчиков. Вам — 4 операции и всё.

## Финал

Этих 4 операций (clone / pull / commit / push), GitHub Desktop и Obsidian достаточно, чтобы быть полноценным членом команды на Shubin Framework. Не нужно становиться разработчиком. Нужно научиться тихо обмениваться файлами с командой так, чтобы git не мешал.

Если через месяц поймёте, что хотите больше контроля — изучите branches и Pull Requests. Но 95% работы идёт без них. Не тратьте время заранее.

**Когда что-то идёт не так и непонятно, что произошло** — не молчите и не делайте ничего, что может ухудшить ситуацию. Спросите разработчика. В git практически невозможно сломать что-то необратимо, но можно создать запутанную ситуацию, которую проще распутать со свежим взглядом.

Удачи. Через квартал git станет прозрачным фоном, и вы перестанете о нём думать.
