# Shubin Framework: vault для нетехнических членов команды

**Полное название:** Гид по работе с vault и git без терминала
**Часть:** Shubin Framework (companion document)
**Версия:** 2.0

**Для кого:** PM, дизайнер, маркетолог, основатель, доменный эксперт — любой член команды, который должен работать с vault, но никогда не открывал терминал.

**Время чтения:** 30 минут.
**Время первичной настройки:** 45 минут.

**Что вам НЕ нужно знать перед чтением:**
- Программирование
- Командная строка
- Что такое git до сегодня

**Что вам нужно знать о контексте:**
Ваша команда использует «vault» — общую папку с markdown-файлами, где хранятся знания команды (решения, фичи, документация). Эта папка синхронизируется через git — систему, которая хранит историю всех изменений и позволяет работать вместе без конфликтов.

Вы должны вносить изменения в vault — добавлять описания фич, обновлять статусы, фиксировать обратную связь. Этот документ научит вас делать это без боли.

**Связанные документы:**
- `framework.md` — полный фреймворк для tech lead команды. Читать необязательно.
- `knowledge.md` — модуль LLM-wiki. Если ваша команда подключила его — вам понадобится раздел 8 здесь.

## 1. Что такое git и зачем он вам

Совсем коротко:

**Git** — это система, которая хранит историю изменений ваших файлов. Каждое изменение — это «коммит» (commit) с автором, датой и описанием. Можно вернуться к любой прошлой версии. Можно работать одновременно с другими людьми без конфликтов.

**GitHub / GitLab / Bitbucket** — это сайт, где хранится копия вашего git-репозитория, к которой имеют доступ все в команде.

**Репозиторий (repo)** — это папка с проектом, отслеживаемая git-ом. У вашей команды их несколько: vault, frontend, backend, etc.

Для работы с vault вам нужно понимать четыре операции:
- **clone** — скачать репозиторий с GitHub на свой компьютер (один раз в начале)
- **pull** — обновить локальную копию из GitHub (получить чужие изменения)
- **commit** — сохранить свои изменения локально с описанием
- **push** — отправить свои коммиты на GitHub (чтобы их увидели остальные)

Этих четырёх операций достаточно для 95% работы.

## 2. Инструмент: GitHub Desktop

Разработчики работают с git из командной строки. Вам это не нужно — используйте графический клиент.

**Рекомендованные инструменты:**

**GitHub Desktop** ([desktop.github.com](https://desktop.github.com)) — бесплатно, простой интерфейс, идеально для начала. Работает с GitHub из коробки. Все примеры в этом документе — для него.

**Obsidian с git plugin** — если вы и так редактируете vault в Obsidian, плагин «Obsidian Git» автоматизирует pull/commit/push в фоне. Минус: меньше контроля, иногда коммитит мусор. Подходит тем, кто хочет полностью забыть про git.

**Начните с GitHub Desktop.** Если через месяц захочется чего-то ещё — попробуете.

## 3. Первичная настройка (один раз, 15 минут)

### 3.1. Что попросить у команды в первый день

Прежде чем начинать, попросите разработчика или tech lead дать вам:
- URL vault-репозитория на GitHub
- Доступ к организации на GitHub (вас должны добавить как member)
- Имя, под которым вас добавят в `stable/team.md`
- Email, который вы будете использовать для коммитов (обычно рабочий)
- 10-минутный тур по структуре: где что лежит, что можно править, что не трогать

Это стандартный onboarding, разработчик его делал и другим. Не стесняйтесь просить.

### 3.2. Шаги настройки

**Шаг 1.** Установите GitHub Desktop с [desktop.github.com](https://desktop.github.com).

**Шаг 2.** Запустите. Войдите в GitHub-аккаунт. Если аккаунта нет — сначала регистрируйтесь, потом попросите добавить в организацию.

**Шаг 3.** Установите редактор для markdown:
- **Obsidian** ([obsidian.md](https://obsidian.md)) — бесплатно, заточен под vault, граф связей, удобный поиск. Рекомендуется по умолчанию.
- **VS Code** ([code.visualstudio.com](https://code.visualstudio.com)) — если вам любопытно заглядывать в код команды.
- Любой markdown-редактор из app store вашей ОС — если не хотите устанавливать ничего нового.

**Шаг 4.** В GitHub Desktop: File → Clone Repository → URL. Вставьте URL vault-репозитория. Выберите папку на компьютере, например `~/work/vault/`. Нажмите Clone.

Через 30 секунд на компьютере появится копия vault. Откройте её в Obsidian (или другом редакторе) как vault.

**Готово.** Теперь у вас на компьютере — рабочая копия знаний команды.

## 4. Ежедневный рабочий цикл

Каждый раз, когда садитесь работать с vault, повторяете ту же последовательность из 4 шагов.

**Шаг 1: Pull (получить свежее, 5 секунд).**

Откройте GitHub Desktop. Если в интерфейсе видна кнопка «Pull origin» с числом коммитов — нажмите её. Это скачает изменения, которые сделали другие.

Делайте это **обязательно перед началом работы**. Если пропустите — рискуете создать конфликт правок.

**Шаг 2: Работайте в редакторе.**

Открываете нужные файлы в Obsidian, редактируете, создаёте новые. Никакого взаимодействия с git здесь не нужно — git автоматически замечает все изменения.

**Шаг 3: Commit (сохранить изменения, 30 секунд).**

Когда сделали что-то осмысленное (закончили мысль, добавили решение, обновили статус) — переключитесь в GitHub Desktop. Слева увидите список изменённых файлов.

В нижнем левом углу — поле «Summary» (короткое описание) и «Description» (опционально, длиннее).

Хорошие примеры summary:
- `add ADR-0024: switch from REST to tRPC`
- `update onboarding feature: shipped`
- `add new pattern: error handling on server`
- `feedback: 3 user interviews summary`

Плохие примеры:
- `update`
- `changes`
- `обновление файла`

После заполнения — нажмите «Commit to main».

**Шаг 4: Push (отправить на GitHub, 5 секунд).**

После commit вверху появится кнопка «Push origin». Нажимаете — изменения улетают на GitHub, остальные могут их забрать через свой Pull.

**Цикл закончен.**

## 5. Типичные ситуации и как их разрулить

**Ситуация 1: «Ничего не происходит после commit».**

Скорее всего, забыли push. Проверьте кнопку «Push origin» вверху.

**Ситуация 2: «Кнопка Pull показывает число, но я ещё работаю над своим».**

Сначала закоммитьте своё (шаг 3 в цикле выше), потом pull. Pull может попросить разрешить «merge» — обычно достаточно нажать default-кнопку и продолжить.

**Ситуация 3: Conflict (конфликт правок).**

Это случается, когда вы и кто-то ещё одновременно поменяли одно и то же место в одном файле. GitHub Desktop покажет файл с конфликтом и отметит конфликтные строки в файле метками `<<<<<<<` и `>>>>>>>`.

**Что делать:**
- Откройте файл в редакторе
- Найдите блок:
  ```
  <<<<<<< HEAD
  ваша версия
  =======
  чужая версия
  >>>>>>> branch-name
  ```
- Решите, что должно быть (часто — обе версии одновременно), удалите метки
- Сохраните файл
- Вернитесь в GitHub Desktop, commit с сообщением `resolve merge conflict`
- Push

Если запутались — позовите разработчика. 5 минут работы на двоих.

**Ситуация 4: «Я случайно удалил файл / испортил что-то важное».**

Git помнит всё. В GitHub Desktop: вкладка History → найдите коммит, где файл ещё был в порядке → правый клик → Revert. Это создаст новый коммит, отменяющий проблемные изменения.

Если страшно — позовите разработчика, ничего не трогая. Файл точно восстановится.

**Ситуация 5: «GitHub Desktop хочет, чтобы я сделал branch».**

Branch (ветка) — это параллельная версия файлов. В работе с vault для **большинства задач это не нужно** — пишите прямо в main.

Branch нужен только если:
- Вы готовите большой draft, который пока не должен видеть никто
- Вы работаете с разработчиком над PR, который требует ревью

В обоих случаях — попросите разработчика помочь создать branch. Один раз сделаете — поймёте.

## 6. Что можно и что нельзя

**Можно (и нужно) свободно:**
- Создавать новые файлы в `active/features/`
- Создавать новые файлы в `log/feedback/` — это append-only папка, туда пишут все
- Создавать новые файлы в `log/retrospectives/`
- Обновлять существующие feature-страницы (статус, log, описание)
- Добавлять новые термины в `stable/glossary.md`

**Спросите разработчика перед изменением:**
- ADR (`active/decisions/`) — это технические решения, непрошенные правки здесь странно смотрятся
- Технические паттерны (`active/patterns/`) — то же самое
- `COMMON.md` — это инструкция для AI, требует осторожности
- Структура папок (создание новых top-level папок)
- `stable/stack.md`, `stable/team.md` — декларативные файлы

**Не трогайте:**
- Файлы с расширением `.gitignore`, `.gitattributes` — это служебные
- Папка `.git/` (она и так скрыта, но мало ли)

### 6.1. Если команда использует knowledge-модуль

Если ваша команда подключила knowledge-модуль (папка `knowledge/` в vault), у него отдельные правила. См. раздел 8 этого документа.

## 7. Альтернатива: правка через GitHub web

Если GitHub Desktop кажется страшным, есть совсем простой путь: редактирование прямо на сайте github.com.

**Как:**
1. Откройте репозиторий vault на GitHub.com
2. Нажмите на нужный файл
3. Иконка карандаша (Edit) в правом верхнем углу
4. Правьте прямо в браузере
5. Внизу — поле «Commit changes», заполните и сохраните

Минусы:
- Нет преимуществ Obsidian (поиск, граф, обратные ссылки)
- Неудобно для длинных правок
- Нельзя массово создавать файлы

Но для разовых правок «обновить статус фичи» работает отлично. Особенно с мобильного.

### 7.1. Когда vault на телефоне

Иногда нужно записать мысль или фидбек, когда нет компьютера.

**Варианты:**
- **Obsidian Mobile** + Obsidian Git plugin — синхронизирует в фоне (требует немного настройки от разработчика)
- **GitHub Mobile app** — можно открыть репо, найти файл, отредактировать прямо в приложении
- **Просто запишите в заметки телефона**, утром перенесёте в vault

Не превращайте vault в задачник на телефоне. Серьёзная работа — за компьютером.

## 8. Как работать с knowledge-модулем

*Этот раздел нужен только если ваша команда подключила knowledge-модуль (папка `knowledge/` в vault). Если нет — можете пропустить.*

Knowledge-модуль — это место, где команда накапливает **внешние знания**: юзер-интервью, статьи про конкурентов, research-отчёты, транскрипты звонков. AI (Claude или Cowork) автоматически компилирует эти материалы в структурированные wiki-страницы с cross-references.

**Для вас это важно, потому что:**
- Вы, вероятно, будете одним из основных людей, кладущих материалы в `knowledge/raw/`
- Вы сможете задавать вопросы вроде «что мы знаем про enterprise-пользователей?» и получать синтез из всех накопленных источников
- Но тут есть жёсткие правила, которые нельзя нарушать, иначе вся система ломается

### 8.1. Как класть источники в raw/

Вы работаете в подпапке `knowledge/raw/`. Структура:

```
knowledge/raw/
├── user-interviews/
├── competitors/
├── industry/
└── calls/
```

Положите новый файл в нужную папку. Имя файла — `YYYY-MM-DD-краткое-описание.md`. Например:
- `knowledge/raw/user-interviews/2026-04-18-acme-corp-discovery.md`
- `knowledge/raw/competitors/2026-04-19-linear-new-pricing.md`

Первые строки файла — frontmatter:

```markdown
---
type: user-interview
source: Acme Corp
date: 2026-04-18
author: @your-name
---

(дальше содержимое — транскрипт, текст статьи, заметки)
```

Commit с сообщением `raw: <что это>` — например, `raw: Acme Corp discovery interview`.

Через какое-то время разработчик (или вы сами через Cowork) запустит ingest-сессию, и Claude добавит этот материал в wiki/.

### 8.2. Два железных правила

**Правило 1: `knowledge/raw/` — immutable.** После того как вы положили файл, **вы не редактируете его содержимое**. Нашли ошибку? Добавьте новый файл с исправлением (`2026-04-18-acme-corp-discovery-correction.md`). Старый не трогайте.

Это важно потому, что Claude запоминает, что он скомпилировал из какого источника. Если вы тихо измените источник, wiki начнёт расходиться с реальностью.

**Правило 2: `knowledge/wiki/` — не трогайте руками.** Это зона, которую поддерживает Claude. Увидели в `wiki/personas/small-business-owner.md` что-то неверное? Не правьте напрямую. Вместо этого:
- Либо добавьте correction в raw/ и попросите разработчика запустить ingest
- Либо попросите Claude через Cowork: «на странице X утверждение Y неверно, перепроверь источники»

Если править wiki напрямую — Claude в следующий раз затрёт ваши правки, и вы поссоритесь с системой.

### 8.3. Как задавать вопросы wiki

Через Cowork или Claude Code (попросите разработчика показать как подключить Cowork к вашему vault — это 5 минут настройки).

Примеры вопросов, которые работают:
- «Что мы знаем про enterprise-пользователей? Дай краткое описание и цитаты.»
- «Какие конкуренты упоминаются в наших интервью? Сравни их по pricing.»
- «Есть ли противоречия в том, что пользователи говорят про onboarding?»

Claude идёт в `wiki/`, возвращает ответ со ссылками на raw-источники. Если wiki-страницы ещё нет, но в raw/ что-то есть — Claude обычно предложит создать страницу.

### 8.4. Что делать, если Claude что-то придумал

LLM иногда галлюцинирует — приписывает цитату не тому интервью, делает обобщения, которых нет в источниках.

Если вы читаете wiki-страницу и видите подозрительное утверждение:
1. Кликните на `[[raw/...]]` ссылку — откроется источник
2. Проверьте, есть ли утверждение в нём
3. Если нет — скажите Claude: «В wiki/personas/X написано Y со ссылкой на raw/Z. Я проверил Z — там этого нет. Обнови страницу.»

Claude исправит. Это нормальный цикл — не пугайтесь.

### 8.5. Минимальный ритуал

Раз в неделю (или чаще, если много нового material'а) — ingest-сессия. Обычно её запускает разработчик или tech lead. Вам достаточно:
- Положить новые raw/-файлы в течение недели
- В день ingest'а проверить отчёт Claude («обновил X, создал Y, нашёл Z противоречий»)
- Если есть что-то, что вас беспокоит в отчёте — спросить

Полная документация по knowledge-модулю — `knowledge.md`. Для вас как для contributor'а в raw/ того, что выше, достаточно.

## 9. Анти-паттерны нетехнических пользователей

**1. «Я каждый раз забываю pull, ничего не работает».**
Сделайте привычку: открыли GitHub Desktop → первое действие — посмотреть на «Pull origin».

**2. «Накопил 30 изменений и закоммитил одним коммитом „update vault“».**
Маленькие частые коммиты — хорошо. Один большой коммит без описания — плохо. Через месяц никто (включая вас) не вспомнит, что там было.

**3. «Изменил что-то в коде разработчика, потому что „ну там же markdown“».**
Vault — да, ваша зона. Кодовый репозиторий — нет. Если что-то нужно поменять в README кодового репозитория — попросите разработчика.

**4. «Боюсь нажать что-то не то и сломать всё».**
В git практически невозможно сломать что-то безвозвратно. Любое действие отменяется. Спрашивайте, не сидите в страхе.

**5. «Не делаю commit, пока не „допишу до конца“».**
Лучше зафиксировать половину работы, чем потерять её при сбое. Промежуточные коммиты — норма.

**6. «Правлю wiki/ напрямую, потому что там ошибка» (если используете knowledge-модуль).**
См. раздел 8.2. Никогда не правьте `knowledge/wiki/` напрямую — исправляйте через источники или через запрос к Claude.

## 10. Реалистичные ожидания

**Первая неделя:** будете спрашивать разработчика по 2–3 раза в день. Это нормально.

**Через месяц:** комфортно делаете цикл pull → редактирование → commit → push без подсказок. Случайные конфликты разруливаете самостоятельно.

**Через квартал:** не задумываетесь о git вообще, он становится прозрачным фоном.

**Что точно НЕ нужно учить:** branches/merging без необходимости, rebasing, cherry-picking, submodules, hooks, всю остальную «настоящую» git-кухню. Это для разработчиков. Вам нужны 4 операции, и это всё.

---

## Финальное

Этих 4 операций (clone / pull / commit / push), GitHub Desktop и Obsidian — достаточно, чтобы быть полноценным членом команды, использующей Shubin Framework. Вам не нужно становиться разработчиком. Вам нужно научиться тихо обмениваться файлами с командой так, чтобы git не мешал.

Если через месяц использования вы поймёте, что хотите больше контроля — изучите branches и Pull Requests. Но 95% работы делается без них. Не тратьте на них время заранее.

**Когда что-то пошло не так и вы не понимаете, что произошло** — не молчите и не делайте ничего, что может усугубить. Спросите разработчика. В git практически невозможно что-то безвозвратно сломать, но можно создать запутанную ситуацию, которую разрулить легче со свежим взглядом.

Удачи. Через квартал git будет прозрачным фоном, и вы перестанете о нём думать.
